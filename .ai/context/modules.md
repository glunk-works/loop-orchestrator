# loop-engine — module walkthrough & architecture

Referenced from the lean root `CLAUDE.md`. Load this when you need the
module-by-module map, the API-key contract, or the container setup. The
**enforced module boundaries** and the day-to-day **commands** stay in
`CLAUDE.md` itself (they are guardrails, not reference).

## API key setup

The Anthropic API key is **never** a CLI flag or env var — it's retrieved exclusively from the OS keyring:

```bash
hatch run python -c "import keyring; keyring.set_password('loop-engine', 'anthropic_api_key', 'sk-ant-...')"
```

In the devcontainer this is provisioned automatically from Infisical on container start (see `.devcontainer/infisical-start.sh` / `seed-secrets.sh`); the underlying `keyring.get_password(...)` contract is unchanged either way. A double-gated env var fallback (`LOOP_ENGINE_ALLOW_ENV_CREDENTIAL=1` + `LOOP_ENGINE_CI_API_KEY`) exists solely for CI/automation contexts that can't mount an encrypted keyring file — never use it elsewhere.

## Architecture

```
 human ⇄ GitHub Issue (filed by the engine; answers as ```answers comments)
              ⇅
             PM ──────────── owns project_spec; resolver of last automated resort
              ⇅
          Architect ───────── owns architecture_definition; resolves Coder questions
              ⇅
     Sprint Breakdown ─────── re-entered on "plan"-impact rework
              ⇅
            Coder ─────────── per-sprint inner loop; escalates, never guesses

 forward path per stage: persona.run() → gate → ACCEPT (snapshot, next stage)
                                              → REVISE (bounded, findings fed back)
                                              → ESCALATE (resolver ladder → issue)
```

- **`core/state.py`** — `State`, a Pydantic v2 model (`schema_version` 3: `run_id`, `status`, `questions`, `pending_issue`, `counters`, `stage_history`, `artifacts`, `artifact_refs`) with `extra="forbid"`. `RunStatus` makes termination explicit (`completed` / `failed_stage` / `budget_exceeded` / `awaiting_issue`) — never inferred from "no exception". `migrate_state_payload()` upgrades v1/v2 snapshots on load. `artifacts` (inline bodies) and `artifact_refs` (disk pointers: `ArtifactRef` = path + sha256 digest + size) coexist during the LangGraph migration — the inline bodies are dropped once the LangGraph engine is the sole reader.
- **`core/engine.py`** — `run_loop(loop: Loop, ...)`: per stage, a bounded propose → gate → accept/revise/escalate cycle. The per-stage cycle is factored into `execute_stage()` (returning a `StageOutcome`), the shared primitive both engines drive so they stay behaviorally identical.
- **`core/graph_engine.py`** — the LangGraph engine (`run_graph_loop`): a `StateGraph` of one node per stage plus a completion node, with conditional edges that advance / re-enter (blast radius) / terminate by calling `execute_stage`. Selected at runtime by `LOOP_ENGINE_ENGINE=langgraph`; the classic `run_loop` stays the default until this path proves out. Parity is guarded by `tests/core/test_graph_engine.py`, which reruns `run_loop`'s own scenarios against the graph. Gate findings feed revision attempts against the prior artifact — the persona sends it as an assistant turn, gets back only corrected sections, and merges via `personas/sections.py` (identical findings twice = stop paying, escalate); escalated questions walk the stage's `resolvers` ladder; questions nobody answers become a GitHub issue and the run pauses (`AWAITING_ISSUE`). Resolved questions carry an `impact` that routes re-entry via `Loop.impact_reentry`. Hard caps (`MAX_ESCALATIONS_PER_STAGE`, `MAX_REPLANS_PER_RUN`) keep every feedback edge finite. Every exit path persists a snapshot.
- **`core/gates.py`** — `ArtifactGate`: content validation (present, JSON-shaped if declared, not question-shaped) plus `## Open Questions` extraction into `Question` objects.
- **`personas/base.py`** — `BasePersona`: abstract `run(state, llm_client, findings=None)`, `consumes`/`produces` artifact declarations (engine pre-checks inputs), and overridable `resolve_questions()` for resolver duty. Personas receive the LLM client via injection — they cannot construct their own or touch credentials directly.
- **`personas/{pm,architecture,agile_sprint_breakdown,coder_iac}/`** — the four default personas; prompts are batch-mode (assumptions + `## Open Questions`, never "ask and wait"). `PMPersona` keeps its critic revision loop (`personas/pm/critic.py`) with no-progress detection, and additionally implements `resolve_questions` (from the spec) and `fold_answers` (folds human issue answers into the spec and classifies impact). `CoderIacPersona` runs an inner agentic loop — one `run_tool_loop` per sprint block with read/execute tools (`read_file`/`list_files`/`grep`/`run_tests` from `tools/coder_tools/`) — emits full-content or SEARCH/REPLACE edit blocks that the persona applies via `write_artifact` (failures are recorded on the report for the gate), and stops at a sprint that raises questions; on findings re-entry it re-runs only the sprint(s) targeted by resolved questions (`Question.origin_detail`) or by sprint-path-prefixed gate findings, falling back to a full re-run when findings carry no attribution. Its stage gate is `core/coder_gate.py`: content checks plus a deterministic pytest run — ACCEPT is evidence-based, and pytest exit 5 (no tests) is a REVISE. That gate pytest runs **in-process**, so it is a second untrusted-execution surface (besides the `run_tests` tool): under `container`/`sandbox` isolation it raises `IsolationUnavailableError` rather than run model code in the orchestrator (sandboxing the gate's pytest is deferred Phase 3b host-side work). Sprint plans and implementation reports are written to `sprints/` via `write_artifact`. **Ralph-loop alternative (Phase 4 · part 1, `LOOP_ENGINE_CODER=ralph`, default `classic`):** `personas/coder_iac/ralph.py`'s `RalphCoderPersona` does **one task per invocation** from a fresh context (mode-agnostic mechanics shared via `personas/coder_iac/shared.py`), selecting the next `task_manifest` task whose deps are all done, tracking progress as the `.agent/STATE.md` `completed_tasks` checklist and appending a lesson to `.agent/MEMORY.md` per increment. Its self-loop is `execute_stage`'s existing revise loop (the Coder stage's `max_revisions` is raised to the iteration cap `LOOP_ENGINE_RALPH_MAX_ITERS`, default 30); its gate is `core/coder_gate.py`'s `RalphCoderGate` — coverage-aware, so green is necessary but **not** sufficient (ACCEPT requires every manifest task checked off AND green pytest), inheriting the same `container`/`sandbox` in-process-pytest guard. It's a behavior change, so it's flag-gated and **not** parity-claimed against the classic Coder (only the `classic` default path and cross-engine equivalence are parity-tested). `AgileSprintBreakdownPersona` additionally emits a deterministic `task_manifest` artifact (`personas/agile_sprint_breakdown/manifest.py`, parsed from its own `**Task N:**` markdown — no extra LLM call; `sprint_plans` unchanged); under Ralph the Sprint-Breakdown stage gate is the `ManifestArtifactGate` (validates the manifest: non-empty, every sprint yields ≥1 task, deps resolvable + acyclic). The Coder stage wiring is chosen at run time by `loops/default/loop.build_default_loop()` reading `LOOP_ENGINE_CODER`.
- **`personas/declarative/`** — **Declarative persona alternative (Phase 4 · part 2, `LOOP_ENGINE_PERSONAS=declarative`, default `classic`).** `GeneratorNode` (`node.py`) is one config-driven single-shot node that replaces the per-class boilerplate of **PM, Architecture, and Sprint Breakdown** (the Coder stays its own agentic node — *personas generate, gates accept, the graph routes*). Each port is a per-persona YAML (`configs/*.yaml`) parsed by a Pydantic-validated `yaml.safe_load` loader (`config.py`, `GeneratorConfig`) selecting from a registry of **shared services** (`services.py`): an **input-wrapper** (`none`/`untrusted` — the latter reuses PM's exact `<untrusted_artifact>` framing), an **output-adapter** (`markdown` → raw string; `sprint_blocks` → parse `### FILEPATH:` blocks, write each file, emit `sprint_plans` **and** `task_manifest`; `json_object` → PM's fence-stripped checklist parse), a **revision-style** (`section_merge` assistant-replay+`sections.merge`; `key_merge` JSON re-extract+dict-merge; `full_reextract`), and an optional `resolve_via_document` resolver. Prompts are **externalized to `prompts/`** and byte-identical to the classic personas' embedded `PROMPT_TEMPLATE` (loaded once at construction so the cached `system_blocks` stay byte-stable; a parity test pins file==embedded). The three ports are thin identity subclasses (`ArchitectureGenerator`/`SprintBreakdownGenerator`/`PMGenerator`) so each stage keeps its own `type(persona).__name__` (distinct escalation counter / question origin). The accept path is **byte-parity** with the classic personas for identical LLM responses (Architecture/Sprint strictly; PM on the clean-extraction path); config format = PyYAML (pinned, `safe_load`-only). `mode.py` is the single reader of `LOOP_ENGINE_PERSONAS`.
- **`personas/pm/critic_gate.py`** — `CriticGate` (Phase 4 · part 2). Retires the `MAX_REVISION_CYCLES` loop that lived inside `PMPersona.run`: it composes the plain `ArtifactGate("project_spec", …)` then runs the pure `critic.review` checks on the parsed spec → REVISE with per-field findings (empty ⇒ ACCEPT). The engine's existing revise loop drives the declarative PM's `key_merge` re-extraction and its identical-findings→escalate is the no-progress detector `MAX_REVISION_CYCLES` used to implement by hand. A non-converging PM escalates **one combined** human-issue question (documented, flag-gated behavior change — not N per-field questions). `fold_answers`/`resolve_questions` stay resolver services on `PMPersona` (resume-time), unchanged. Lives under `personas/pm/` (not `core/`) because it reuses `critic.review` and the import-boundary test forbids `core` importing a persona module — the same core-safe-gate pattern as `agile_sprint_breakdown/manifest.ManifestArtifactGate`.
- **`loops/default/loop.py`** — `DEFAULT_LOOP`, a `Loop` of four `Stage`s wiring gates, the resolver ladder, and `impact_reentry`. No YAML/JSON loop-definition format — loops are just Python. `build_default_loop()` reads both `LOOP_ENGINE_CODER` (Ralph-vs-classic Coder + manifest gate) and `LOOP_ENGINE_PERSONAS` (declarative-vs-classic document personas + `CriticGate`-vs-`ArtifactGate` for the PM stage) at call time; the two flags compose, and both default off ⇒ byte-identical to the pre-Phase-4 loop.
- **`tools/llm/client.py`** — `LLMClient`: retrieves the API key from `keyring` exactly once per instance, prices every call from the per-model rate table in `tools/llm/pricing.py` and enforces the per-run USD budget *before* each call (`BudgetExceededError`; heuristic estimate, refined via `count_tokens` once it reaches 50% of the remaining budget), sends caller-supplied `system_blocks` as a cached system prefix (`cache_control: ephemeral` on the last block — personas keep these byte-identical across calls and put volatile content in the user prompt), drives the Coder's bounded tool loop via `run_tool_loop` (per-iteration budget debit, executor errors surfaced as `is_error` tool results, iteration cap fails the stage honestly), and raises `TruncatedResponseError` when `stop_reason == "max_tokens"` (a truncated artifact must never propagate). **This is the only module in the codebase permitted to import `keyring`** — enforced by `tests/tools/test_keyring_boundary.py`.
- **`tools/issue_io/`** — human-escalation GitHub verbs (shells out to `gh`; no token in-process). Files the escalation issue, parses the human's ```answers comment, maps numbered answers back to questions. Sibling to `tools/repo_io` (Phase 5, sprint 22b) — together they are the codebase's GitHub-owning modules; no other module talks to GitHub.
- **`tools/repo_io/`** — the repo/branch/PR **factory** verbs (Phase 5, sprint 22b): `create_repository`, `clone_repo`, `create_branch`, `open_pr`, each shelling to the already-authenticated `gh` (a second `gh` consumer alongside `issue_io`, no new subprocess surface — the local-git surface, e.g. `git push` inside a cloned tree, is deferred to Sprint 23). Returns typed `RepoRef`/`PullRef` (pydantic, `extra="forbid"`); `clone_repo`'s `dest` is validated (relative, no `..`, no post-resolution symlink escape — the `tools/coder_tools.resolve_tool_path` discipline, minus its artifact-root allowlist). No merge verb — auto-merge stays prohibited. Imports no `keyring`, writes no files. These verbs are orchestrator-invoked only (see `mcp_servers/github_server.py` below) — never in the model's coder tool loop.
- **`tools/state_io/writer.py`** — the sole writer of `State` snapshots (`state/`) and produced artifacts (`docs/`, `sprints/`, `src/`). Validates `run_id`/artifact paths against path traversal before any filesystem call. Also owns the `.agent/` writers (`write_agent_scratchpad`, `append_agent_memory` — the latter enforces the append-only ledger invariant), kept separate from the model-facing `write_artifact` allow-list.
- **`tools/agent_state/`** — the `.agent/` semantic-state layer: `.agent/STATE.md` (mutable scratchpad) and `.agent/MEMORY.md` (append-only decisions/lessons ledger), parsed/rendered via deterministic anchors; all writes delegate to `state_io`. **(This is the loop-engine *product's* runtime state, distinct from the `.ai/` dev-workflow layer.)**
- **`tools/artifact_store.py`** — mirrors inline artifact bodies to disk and maintains `State.artifact_refs`; `mirror_to_disk` (idempotent, called by the engine at snapshot time), plus `get_artifact`/`has_artifact` readers that new (LangGraph) code uses instead of touching `state.artifacts`.
- **`tools/worktree/`** — Phase 3a execution isolation. `worktree_run(run_id)` is a context manager the CLI wraps `run`/`resume` in: under `LOOP_ENGINE_ISOLATION=worktree` (default off) it creates/reuses a per-run `git worktree` on a `loop/<run_id>` branch and `chdir`s into it, so the model-facing artifact tree and both tool paths (which key off `Path.cwd()`) confine to the worktree — while State snapshots stay in the main checkout because it pins `state_io`'s `state_root()` to the original CWD first. Off ⇒ no-op passthrough. The flag now has four values (read by `tools/isolation.py`): `container`/`sandbox` also require a worktree (they mount it). Worktrees are retained (resume / PR source / inspection); removal is explicit via `cleanup`/`prune_all` (CLI `prune-worktrees`). **3a is blast-radius isolation, not a security sandbox** — on the default tools path model code still runs in-process; the real boundary is Phase 3b (`container`/`sandbox`), whose inert seam now exists (`tools/mcp` launch params + guards, argv-tested) but whose executing sandbox is deferred to a daemon-bearing host (see `docs/phase3_execution_isolation_plan.md`, `sprints/18_execution_isolation_container/sprint_plan.md`).
- **`mcp_servers/coder_tools_server.py`** — the Coder's read/execute tools (`read_file`/`list_files`/`grep`/`run_tests`) as a stdio MCP server, delegating to `tools/coder_tools` (logic re-fronted, not rewritten). Run as `python -m loop_engine.mcp_servers.coder_tools_server`.
- **`mcp_servers/github_server.py`** (Phase 5, sprint 22b) — the system's **second** MCP server and its **first credentialed, network-enabled** one: the four `tools/repo_io` factory verbs (`create_repository`, `clone_repo`, `create_branch`, `open_pr`) re-fronted over MCP, native (not the official ~60-tool GitHub MCP server — that includes a `merge_pull_request` tool, contradicting "auto-merge prohibited", and can't be asserted to expose *exactly* four tools). Holds no credentials itself (`repo_io` shells to `gh`'s own auth) and imports side-effect-free — no `gh` call at import, so discovery (`list_tools`) stays offline/hermetic. Run as `python -m loop_engine.mcp_servers.github_server`. Reached **only** via `build_github_provider()` — a consumer-scoped provider structurally separate from `build_coder_tool_provider`, so these verbs never enter the model's coder tool loop (`tests/tools/test_mcp_provider.py` asserts the two providers' tool sets are exactly `{read_file, list_files, grep, run_tests}` and `{create_repository, clone_repo, create_branch, open_pr}`, disjoint). This sprint ships the capability only — no production flow chains the verbs yet (Sprint 23: trigger surface, maintenance flow, bootstrap flow).
- **`tools/mcp/`** — the runtime MCP client: `MCPToolProvider` connects to one or more stdio servers on a background event-loop thread, discovers tools via `list_tools`, and exposes `(tools, execute)` to `run_tool_loop`, routing each call by tool name to its owning server session. Selected by `LOOP_ENGINE_TOOLS=mcp`; the Coder's `_CoderToolBackend` opens one provider per run (lazily, on first tool-loop use) and closes it in a `finally`. Default stays the in-process `CODER_TOOLS`/`_execute_tool` dispatch. **Discovery is config-driven and consumer-scoped (Phase 5, sprint 22a):** `tools/mcp/config.py`'s `load_mcp_config()` reads a repo-root **`loop_engine.mcp.json`** (`{"servers": {"<name>": {"command", "args", "cwd"}}}`, pydantic `extra="forbid"`) and returns logical-name → launch-spec, merged over a built-in `coder_tools` default so file-absence is byte-identical to pre-22a behavior. As of sprint 22b, the file is **committed at the repo root** with one real entry — a static `github` stanza (`{"command": "python", "args": ["-m", "loop_engine.mcp_servers.github_server"], "cwd": null}`; `command` resolves via PATH to the active-env interpreter on the factory host) — the first non-default server it declares. `build_provider_for(names, *, cwd)` builds a provider scoped to **only** the named servers — a consumer names the servers it wants, so a `loop_engine.mcp.json` declaring other servers never leaks their tools to a caller that didn't ask for them. `build_coder_tool_provider(cwd)` is a thin wrapper (`build_provider_for(["coder_tools"], cwd=cwd)`) and still picks the `coder_tools` server's *launch params* by isolation mode (Phase 3b): `container` → `container_server_params` (docker/podman, mounts only the worktree), `sandbox` → `sandbox_server_params` (`bwrap`, daemon-free — secondary/inert), else the config's local profile — this isolation-aware override is a recognized special case for `coder_tools` only; every other named server (including `github`) launches from its static config spec as-is. `build_github_provider()` (sprint 22b) is the `github` counterpart: `build_provider_for(["github"])`, no `cwd` override — it is the orchestrator's own helper, never named by the model's tool loop. Both sandbox builders raise `IsolationUnavailableError` when no runtime is present — the mode **never** silently degrades to in-process execution — and `container`/`sandbox` require `LOOP_ENGINE_TOOLS=mcp` (there is no server to sandbox on the in-process path). The container/sandbox is spawned by the same MCP `stdio_client` that launches the local server (only `command`/`args` change), so it adds **no new sanctioned subprocess surface**; runtime detection is `shutil.which`, not a subprocess. **Inert on this branch** (no runtime here; `unshare -Ur` is denied): argv-shape tested only — the real `docker run` e2e + the sandboxed *gate* pytest are deferred (`sprints/18_execution_isolation_container/sprint_plan.md`, `docs/phase3_execution_isolation_plan.md`). **`loop_engine.mcp.json` is not `.mcp.json`** — see the containers/devcontainer section below for the latter, which is Claude Code's own project MCP config and belongs to a different tool entirely.
- **`tools/isolation.py`** — the single reader of `LOOP_ENGINE_ISOLATION` (`none`|`worktree`|`container`|`sandbox`; unknown value raises): `isolation_mode()`, `worktree_needed()` (true for all three isolating modes — container/sandbox still run inside a worktree the sandbox mounts), `sandbox_runtime_mode()` (the `container`/`sandbox` tool-sandboxing modes), and `IsolationUnavailableError` (imported by `core/coder_gate.py` so `core/` can signal "can't honor this mode" without importing `tools/mcp`). Imports no other `loop_engine` module.
- **`tools/logging_config.py`** — structured per-stage cost logging.
- **`cli.py`** — the Typer entrypoint (`run`, `resume`, `cost-summary`). A thin wrapper; all logic lives in the library layer. `run`'s `--resume-from` path stays inline (its own `_resolve_loop`/`_select_engine`, patchable via `cli.run_loop`/`cli.run_graph_loop`/`cli.LLMClient`); its fresh-run path delegates to `runner.run_new`. Resume verifies snapshot stage names against the loop before slicing (never resumes a mismatched loop).
- **`runner.py`** (Phase 5, sprint 23) — the shared "start a fresh run from `human_input`" orchestration, factored out of `cli.run` so both the CLI and the trigger surface's dispatcher call one source of truth. `run_new(human_input, *, budget_usd, loop_name) -> State` mints a `run_id`, builds the initial `State`, constructs `LLMClient`, and runs `with worktree_run(run_id): _select_engine()(...)`. Carries its own `_resolve_loop`/`_select_engine`/`NAMED_LOOPS` (mirrored from `cli.py`, not shared — each module's engine indirection is independently patchable by its own tests) and `DEFAULT_BUDGET_USD` (the single source `cli.py` imports).
- **`trigger/`** (Phase 5 piece 2, sprint 23) — the trigger surface: a FastAPI webhook that turns a GitHub event into a real `runner.run_new` call. `trigger/parse.py` — `RunRequest` (pydantic, `extra="forbid"`: `human_input`, `budget_usd`, `loop_name`, plus `repo_full_name`/`issue_number` provenance) and `parse_event(event_name, payload) -> RunRequest | None`, implementing the locked grammar (an `issues`/`labeled` event with label `agent-action`, or an `issue_comment`/`created` event whose first non-empty line is `/agent-run`; `human_input` = issue `title + "\n\n" + body`; everything else — including `ping` — is `None`, defensively, never a raise). `trigger/dispatch.py` — the `RunDispatcher` `Protocol` (`async def dispatch(request) -> None`, fire-and-forget) and its sole 23 implementation `InProcessDispatcher`: in-memory dedupe on `(repo_full_name, issue_number)`, then `runner.run_new` scheduled on a worker thread (`asyncio.to_thread`) via `asyncio.create_task` so `dispatch` returns before the run finishes and the event loop never blocks. `trigger/app.py` — `create_app(dispatcher=None) -> FastAPI` reads `LOOP_ENGINE_WEBHOOK_SECRET` at construction and **fails closed** (raises) if unset; the single `POST /webhook` route reads the raw body, HMAC-SHA256-verifies `X-Hub-Signature-256` over those exact raw bytes (`hmac.compare_digest`) before any JSON parsing, then parses → `204` (no-op) or `dispatch()` → `202`; a module-level `app` is built eagerly only when the secret is already set in the environment (so importing the module never raises in a dev/test environment lacking it) for eventual `uvicorn` hosting (not shipped in 23). Boundary posture (`tests/trigger/test_boundaries.py`, AST-scanned like the `keyring`/`state_io` boundary tests): no `keyring` import, no direct file write, no subprocess surface (dispatch is in-process — the sanctioned-surface count stays three). All 23 coverage is hermetic (`TestClient` + injected fake dispatcher + patched `runner.run_new`); real webhook delivery is deferred (`sprints/DEFERRED_VERIFICATION.md` §6).

`tests/` mirrors `src/loop_engine/` layout, plus `tests/integration/` for cross-module tests (e.g. `test_budget_abort.py`, `test_no_credential_leakage.py`).

## Containers / devcontainer

- Root `Dockerfile` is multi-stage: `dev` (full contributor toolchain — hatch, ruff, git, gh) and `prod` (minimal runtime). Both are local-only, no registry publish.
- `.devcontainer/` targets the `dev` stage. It bind-mounts a forwarded host GPG agent socket (`GPG_HOST_DIR`) for commit signing, and provisions `ANTHROPIC_API_KEY`, `GITHUB_PERSONAL_ACCESS_TOKEN`, and `LOOP_ENGINE_KEYRING_PASSPHRASE` from Infisical (project identified by root `.infisical.json`, machine identity via `INFISICAL_CLIENT_ID`/`INFISICAL_CLIENT_SECRET` forwarded from the host shell) on every container start via `infisical-start.sh` → `seed-secrets.sh`. `infisical run` under machine-identity auth needs `--projectId` and `--path=/loop-engine` passed explicitly — it does not read `.infisical.json`'s `workspaceId` automatically the way interactive login does. The keyring passphrase file must be written *before* anything calls `keyring.set_password`, since the container's cryptfile keyring backend reads that file on every call.
- A bare Linux container has no OS-native keyring backend, so `containers/keyring_backend/cryptfile_backend.py` is wired in as a custom encrypted-file backend via `keyring`'s own backend-discovery config (not the PyPI `keyrings.cryptfile` package). It reads two *file paths* from env vars (`LOOP_ENGINE_KEYRING_FILE`, `LOOP_ENGINE_KEYRING_PASSPHRASE_FILE`) — never secret values directly.
- `.mcp.json` connects Claude Code/Cursor to GitHub's hosted remote MCP server using a bearer-token `GITHUB_PERSONAL_ACCESS_TOKEN`, sourced via `gh auth token` (itself seeded by the Infisical flow above) rather than OAuth, so the devcontainer can authenticate non-interactively at container start. **This is not the same file as `loop_engine.mcp.json`** (see `tools/mcp/` above) — `.mcp.json` is Claude Code's own project MCP config (`{"mcpServers": {...}}`, dev-tooling only, committed `65ed47c`), while `loop_engine.mcp.json` is loop-engine's own product-runtime stdio server-launch config, read only by `tools/mcp/config.py`. Neither module reads the other's file; the two schemas are incompatible on purpose (a stray `.mcp.json` in the repo root is why `load_mcp_config()` must never default to that filename — see the Phase 5 sprint 22a revision note in `docs/migration_roadmap.md`).

See `docs/architecture_definition.md` for the full architecture and threat-model writeup, and the README's "Running in a container" section for command-level detail on all of the above.
