# loop-engine — module walkthrough & architecture

Referenced from the lean root `CLAUDE.md`. Load this when you need the
module-by-module map, the API-key contract, or the container setup. The
**enforced module boundaries** and the day-to-day **commands** stay in
`CLAUDE.md` itself (they are guardrails, not reference).

## API key setup

The Anthropic API key is **never** a CLI flag or env var — it's retrieved exclusively from the OS keyring:

```bash
hatch run python -c "import keyring; keyring.set_password('loop-engine', 'anthropic_api_key', 'sk-ant-...')"
```

In the devcontainer this is provisioned automatically from Infisical on container start (see `.devcontainer/infisical-start.sh` / `seed-secrets.sh`); the underlying `keyring.get_password(...)` contract is unchanged either way. A double-gated env var fallback (`LOOP_ENGINE_ALLOW_ENV_CREDENTIAL=1` + `LOOP_ENGINE_CI_API_KEY`) exists solely for CI/automation contexts that can't mount an encrypted keyring file — never use it elsewhere.

## Architecture

```
 human ⇄ GitHub Issue (filed by the engine; answers as ```answers comments)
              ⇅
             PM ──────────── owns project_spec; resolver of last automated resort
              ⇅
          Architect ───────── owns architecture_definition; resolves Coder questions
              ⇅
     Sprint Breakdown ─────── re-entered on "plan"-impact rework
              ⇅
            Coder ─────────── per-sprint inner loop; escalates, never guesses

 forward path per stage: persona.run() → gate → ACCEPT (snapshot, next stage)
                                              → REVISE (bounded, findings fed back)
                                              → ESCALATE (resolver ladder → issue)
```

- **`core/state.py`** — `State`, a Pydantic v2 model (`schema_version` 3: `run_id`, `status`, `questions`, `pending_issue`, `counters`, `stage_history`, `artifacts`, `artifact_refs`) with `extra="forbid"`. `RunStatus` makes termination explicit (`completed` / `failed_stage` / `budget_exceeded` / `awaiting_issue`) — never inferred from "no exception". `migrate_state_payload()` upgrades v1/v2 snapshots on load. `artifacts` (inline bodies) and `artifact_refs` (disk pointers: `ArtifactRef` = path + sha256 digest + size) coexist during the LangGraph migration — the inline bodies are dropped once the LangGraph engine is the sole reader.
- **`core/engine.py`** — `execute_stage()` (returning a `StageOutcome`): one bounded propose → gate → accept/revise/escalate cycle, the single unit of stage progress. **Phase 6, sprint 27 deleted `run_loop`**, the classic `while`-loop driver; the LangGraph engine's stage nodes are thin wrappers over this primitive, so all per-stage behavior lives here and there is no second engine to keep in step. **Phase 6, Sprint 26** added an injectable `issue_filer` write seam (`IssueFiler = Callable[[State, list[Question], str], IssueRef]`), threaded `run_graph_loop` → `execute_stage` → `_pause_for_issue`. The parameter defaults to `None` and is resolved to the classic `file_question_issue` **by module-global lookup inside the function body**, not as a literal default-argument value — binding the default directly would snapshot the reference at import time and break the existing `monkeypatch.setattr("loop_engine.core.engine.file_question_issue", ...)` test pattern. Injecting `tools.issue_io.mcp_issue_filer(provider)` routes filing through the `issue` MCP server instead; the classic direct call stays the runtime default (no flag flips it).
- **`core/graph_engine.py`** — the LangGraph engine (`run_graph_loop`): a `StateGraph` of one node per stage plus a completion node, with conditional edges that advance / re-enter (blast radius) / terminate by calling `execute_stage`. **The only engine since Phase 6 (sprint 27)** — `LOOP_ENGINE_ENGINE` and the classic `run_loop` are deleted (recoverable at the `pre-phase6-classic` tag). Its behavioral contract is `execute_stage`'s, pinned by `tests/core/test_engine.py`; `tests/core/test_graph_engine.py` was the classic-vs-graph parity harness and is now reduced to what is genuinely graph-specific — a stage node re-entering *itself* along a conditional edge (the Ralph mechanism), including a long self-loop that would trip LangGraph's super-step recursion guard. Gate findings feed revision attempts against the prior artifact — the persona sends it as an assistant turn, gets back only corrected sections, and merges via `personas/sections.py` (identical findings twice = stop paying, escalate); escalated questions walk the stage's `resolvers` ladder; questions nobody answers become a GitHub issue and the run pauses (`AWAITING_ISSUE`). Resolved questions carry an `impact` that routes re-entry via `Loop.impact_reentry`. Hard caps (`MAX_ESCALATIONS_PER_STAGE`, `MAX_REPLANS_PER_RUN`) keep every feedback edge finite. Every exit path persists a snapshot.
- **`core/gates.py`** — `ArtifactGate`: content validation (present, JSON-shaped if declared, not question-shaped) plus `## Open Questions` extraction into `Question` objects.
- **`personas/base.py`** — `BasePersona`: abstract `run(state, llm_client, findings=None)`, `consumes`/`produces` artifact declarations (engine pre-checks inputs), and overridable `resolve_questions()` for resolver duty. Personas receive the LLM client via injection — they cannot construct their own or touch credentials directly.
- **`personas/{pm,architecture,agile_sprint_breakdown,coder_iac}/`** — what survives of the original persona classes after Phase 6. The **classic PM / Architecture / Sprint-Breakdown persona classes are deleted** (the declarative generators below replaced them; `prompts/` holds their prompt content), and so is the classic per-sprint **`CoderIacPersona`** and its `CoderGate`. What remains in these packages is the logic the surviving path still calls: `personas/pm/persona.py` keeps **`fold_answers`** (folds human issue answers into the spec and classifies each answer's blast radius — driven by `cli.resume` after an issue round-trip; `PMGenerator` delegates to it, and it has no declarative-config equivalent because it is a bespoke second LLM call, not a document-generation pass) plus `_wrap_untrusted_artifact` / `_parse_extraction_response`, which `declarative/services.py` reuses verbatim; `personas/pm/critic.py` + `critic_gate.py` are the PM's structural checks, now driving the PM stage **gate** rather than an in-persona revision loop; `personas/agile_sprint_breakdown/persona.py` keeps only `_parse_sprint_blocks` (the `### FILEPATH:` splitter `declarative/services.py` reuses) and `manifest.py` builds the deterministic **`task_manifest`** from each plan's `**Task N:**` markdown (no extra LLM call) and supplies the `ManifestArtifactGate`. **`personas/coder_iac/`** is the Coder: `ralph.py`'s **`RalphCoderPersona`** — the only Coder — does **one manifest task per invocation** from a fresh context (mechanics in `shared.py`), selecting the next task whose deps are done, tracking progress as the `.agent/STATE.md` `completed_tasks` checklist and appending a lesson to `.agent/MEMORY.md` per increment. Its self-loop is `execute_stage`'s existing revise loop (the Coder stage's `max_revisions` is the iteration cap `LOOP_ENGINE_RALPH_MAX_ITERS`, default 30 — the one flag `personas/coder_iac/mode.py` still reads, and genuine runtime config rather than scaffolding). It runs an inner agentic loop (`run_tool_loop` over the MCP-served read/execute tools) and emits full-content or SEARCH/REPLACE edit blocks the persona applies via `write_artifact` (failures are recorded on the report for the gate). Its gate is `core/coder_gate.py`'s **`RalphCoderGate`** — coverage-aware, so green is necessary but **not** sufficient: ACCEPT requires every manifest task checked off AND a green pytest run, and pytest exit 5 (no tests) is a REVISE. Ralph was never parity-claimed against the classic Coder (it is a behavior change, not a refactor); its sunset justification was the V2 host run reaching terminal COMPLETED under the full production config. Sprint plans and implementation reports are written to `sprints/` via `write_artifact`.
- **`personas/declarative/`** — **The document personas (Phase 4 · part 2; the *only* ones since Phase 6 — `LOOP_ENGINE_PERSONAS` and the classic PM/Architecture/SprintBreakdown persona classes are deleted).** `GeneratorNode` (`node.py`) is one config-driven single-shot node that replaces the per-class boilerplate of **PM, Architecture, and Sprint Breakdown** (the Coder stays its own agentic node — *personas generate, gates accept, the graph routes*). Each port is a per-persona YAML (`configs/*.yaml`) parsed by a Pydantic-validated `yaml.safe_load` loader (`config.py`, `GeneratorConfig`) selecting from a registry of **shared services** (`services.py`): an **input-wrapper** (`none`/`untrusted` — the latter reuses PM's exact `<untrusted_artifact>` framing), an **output-adapter** (`markdown` → raw string; `sprint_blocks` → parse `### FILEPATH:` blocks, write each file, emit `sprint_plans` **and** `task_manifest`; `json_object` → PM's fence-stripped checklist parse), a **revision-style** (`section_merge` assistant-replay+`sections.merge`; `key_merge` JSON re-extract+dict-merge; `full_reextract`), and an optional `resolve_via_document` resolver. Prompts are **externalized to `prompts/`**, which is now the **sole source of truth** (loaded once at construction so the cached `system_blocks` stay byte-stable). The old file==embedded parity test is gone with the embedded copies; `tests/personas/declarative/test_config_prompts.py` instead asserts every config resolves to a real prompt file carrying its adapter's mandated markers. The three ports are thin identity subclasses (`ArchitectureGenerator`/`SprintBreakdownGenerator`/`PMGenerator`) so each stage keeps its own `type(persona).__name__` (distinct escalation counter / question origin). The accept path is **byte-parity** with the classic personas for identical LLM responses (Architecture/Sprint strictly; PM on the clean-extraction path); config format = PyYAML (pinned, `safe_load`-only).
- **`personas/pm/critic_gate.py`** — `CriticGate` (Phase 4 · part 2). Retires the `MAX_REVISION_CYCLES` loop that lived inside `PMPersona.run`: it composes the plain `ArtifactGate("project_spec", …)` then runs the pure `critic.review` checks on the parsed spec → REVISE with per-field findings (empty ⇒ ACCEPT). The engine's existing revise loop drives the declarative PM's `key_merge` re-extraction and its identical-findings→escalate is the no-progress detector `MAX_REVISION_CYCLES` used to implement by hand. A non-converging PM escalates **one combined** human-issue question (documented, flag-gated behavior change — not N per-field questions). `fold_answers`/`resolve_questions` stay resolver services on `PMPersona` (resume-time), unchanged. Lives under `personas/pm/` (not `core/`) because it reuses `critic.review` and the import-boundary test forbids `core` importing a persona module — the same core-safe-gate pattern as `agile_sprint_breakdown/manifest.ManifestArtifactGate`.
- **`loops/default/loop.py`** — `DEFAULT_LOOP`, a `Loop` of four `Stage`s wiring gates, the resolver ladder, and `impact_reentry`. No YAML/JSON loop-definition format — loops are just Python. **Phase 6 collapsed it to one unbranched wiring**: the declarative generators + PM `CriticGate` + `ManifestArtifactGate` + `RalphCoderPersona`/`RalphCoderGate`. `build_default_loop()` reads no mode flag (there are none left); it is still rebuilt per run so `ralph_max_iterations()` is read at run time rather than baked in at import.
- **`tools/llm/client.py`** — `LLMClient`: retrieves the API key from `keyring` exactly once per instance, prices every call from the per-model rate table in `tools/llm/pricing.py` and enforces the per-run USD budget *before* each call (`BudgetExceededError`; heuristic estimate, refined via `count_tokens` once it reaches 50% of the remaining budget), sends caller-supplied `system_blocks` as a cached system prefix (`cache_control: ephemeral` on the last block — personas keep these byte-identical across calls and put volatile content in the user prompt), drives the Coder's bounded tool loop via `run_tool_loop` (per-iteration budget debit, executor errors surfaced as `is_error` tool results, iteration cap fails the stage honestly), and raises `TruncatedResponseError` when `stop_reason == "max_tokens"` (a truncated artifact must never propagate). **This is the only module in the codebase permitted to import `keyring`** — enforced by `tests/tools/test_keyring_boundary.py`.
- **`tools/issue_io/`** — human-escalation GitHub verbs (shells out to `gh`; no token in-process). Files the escalation issue, parses the human's ```answers comment, maps numbered answers back to questions. Sibling to `tools/repo_io` (Phase 5, sprint 22b) — together they are the codebase's GitHub-owning modules; no other module talks to GitHub. **Phase 6, Sprint 26** split the module into pure rendering/parsing (`render_question_issue` → `(title, body, label)`; `parse_issue_answers(issue_data)`; both no `gh`) and the two `gh`-touching primitives (`create_issue(title, body, label) -> IssueRef`; `read_issue(issue_number) -> dict`), so only strings/JSON need to cross the new MCP boundary below — `State`/`Question` never do. The classic `file_question_issue`/`read_issue_answers` are now thin, behavior-preserving wrappers over the split (`file_question_issue = create_issue(*render_question_issue(...))`). `tools/issue_io/mcp_client.py` adds the MCP-side adapters: `mcp_issue_filer(provider)` (renders locally, then dispatches `create_issue`) and `mcp_read_issue(provider, n)` (dispatches `read_issue`, parses JSON) — both signature-compatible with their classic counterparts, so either drops into the seams below. **The classic direct `gh` calls remain the runtime default** — the MCP route is a capability, not yet selected by anything at runtime (no new flag governs it, per Phase 6 planning-pass LD3).
- **`tools/repo_io/`** — the repo/branch/PR **factory** verbs (Phase 5, sprint 22b): `create_repository`, `clone_repo`, `create_branch`, `open_pr`, each shelling to the already-authenticated `gh` (a second `gh` consumer alongside `issue_io`, no new subprocess surface — the local-git surface, e.g. `git push` inside a cloned tree, is deferred to Sprint 23). Returns typed `RepoRef`/`PullRef` (pydantic, `extra="forbid"`); `clone_repo`'s `dest` is validated (relative, no `..`, no post-resolution symlink escape — the `tools/coder_tools.resolve_tool_path` discipline, minus its artifact-root allowlist). No merge verb — auto-merge stays prohibited. Imports no `keyring`, writes no files. These verbs are orchestrator-invoked only (see `mcp_servers/github_server.py` below) — never in the model's coder tool loop.
- **`tools/git_io/`** (Phase 5 piece 3, sprint 24) — the local-git working-tree write surface `flows/maintenance` needs (`checkout_branch`, `commit_all`, `push_branch`, plus a read-only `has_changes` probe over `git status --porcelain` so a no-op run never hits `commit_all`'s empty-index failure) in a repo **cloned by `tools/repo_io.clone_repo`**: a fourth sanctioned subprocess surface, kept in its own module rather than bolted onto `tools/repo_io` (`gh`, the remote API) or `tools/worktree` (the orchestrator's own per-run isolation, keyed off `run_id` — a maintenance clone is a foreign tree). Mirrors `tools/worktree/manager.py::_git`'s posture exactly (fixed argv `["git", "-C", <tree>, ...]`, `shell=False`, a hard timeout, `check=False` + explicit `GitIOError` raise on non-zero) and validates every `tree` argument by reusing `tools/repo_io/github.py::_validate_clone_dest` (relative, no `..`, no symlink escape). `git push` rides `gh`'s own credential helper established at clone time — no `keyring` import, no new credential path. `tests/tools/test_subprocess_surfaces.py` is the repo-wide static assertion that exactly four modules shell out (naming all four); `tests/tools/git_io/test_local.py` exercises every verb (incl. `has_changes`) end to end against a `tmp_path` git repo + a local bare remote.
- **`tools/scaffold/`** (Phase 5 piece 4, sprint 25) — the skeleton-writing module: the **second** sanctioned file-write surface (alongside `tools/state_io`), mirroring how sprint 24 carved out `tools/git_io` as the fourth subprocess surface. `write_skeleton(tree, *, kind="python", pkg_name, repo_name) -> list[str]` validates `tree` by reusing `tools/repo_io/github.py::_validate_clone_dest`, sanitizes `pkg_name` to a safe Python identifier (rejects/normalizes anything outside `[A-Za-z0-9_]`, prefixes a leading digit — raises `ValueError` if nothing salvageable remains, so a crafted repo name can never inject a `../` path segment or a non-identifier into the written tree), renders the bundled `kind`-parametrized template set (`templates/<kind>/`, package data reached via `importlib.resources.files("loop_engine.tools.scaffold") / "templates"`) plus the shared conventions `templates/CLAUDE.md`, substituting `__PKG_NAME__`/`__REPO_NAME__` tokens with plain `str.replace` (no template-engine dependency), and writes every rendered file via `write_text`. Only `kind="python"` ships in 25 (a minimal `src/`-layout skeleton: `pyproject.toml`, `src/<pkg>/__init__.py`, a trivial passing smoke test, `README.md`, `.gitignore`); a `kind="iac"` OpenTofu set drops in later behind the same seam. The bundled `templates/CLAUDE.md` is asserted **byte-identical** to `.ai/context/conventions.md` (`tests/tools/scaffold/test_conventions_sync.py` — drift fails CI); the non-`.py` template files ship in a built wheel via hatchling's **default** `packages` file selection (verified with `hatch build -t wheel` — no `force-include` needed; one was tried and rejected, since it duplicate-conflicts with the default inclusion). Imports no `subprocess`/`keyring` — a file-write surface only, so the four sanctioned subprocess surfaces are unchanged. `tests/tools/test_state_io_boundary.py`'s allow-set is `{state_io, scaffold}`.
- **`tools/state_io/writer.py`** — the sole writer of `State` snapshots (`state/`) and produced artifacts (`docs/`, `sprints/`, `src/`). Validates `run_id`/artifact paths against path traversal before any filesystem call. Also owns the `.agent/` writers (`write_agent_scratchpad`, `append_agent_memory` — the latter enforces the append-only ledger invariant), kept separate from the model-facing `write_artifact` allow-list.
- **`tools/agent_state/`** — the `.agent/` semantic-state layer: `.agent/STATE.md` (mutable scratchpad) and `.agent/MEMORY.md` (append-only decisions/lessons ledger), parsed/rendered via deterministic anchors; all writes delegate to `state_io`. **(This is the loop-engine *product's* runtime state, distinct from the `.ai/` dev-workflow layer.)**
- **`tools/artifact_store.py`** — mirrors inline artifact bodies to disk and maintains `State.artifact_refs`; `mirror_to_disk` (idempotent, called by the engine at snapshot time), plus `get_artifact`/`has_artifact` readers that new (LangGraph) code uses instead of touching `state.artifacts`.
- **`tools/worktree/`** — Phase 3a execution isolation. `worktree_run(run_id)` is a context manager the CLI wraps `run`/`resume` in: under `LOOP_ENGINE_ISOLATION=worktree` (default off) it creates/reuses a per-run `git worktree` on a `loop/<run_id>` branch and `chdir`s into it, so the model-facing artifact tree and both tool paths (which key off `Path.cwd()`) confine to the worktree — while State snapshots stay in the main checkout because it pins `state_io`'s `state_root()` to the original CWD first. Off ⇒ no-op passthrough. The flag now has four values (read by `tools/isolation.py`): `container`/`sandbox` also require a worktree (they mount it). Worktrees are retained (resume / PR source / inspection); removal is explicit via `cleanup`/`prune_all` (CLI `prune-worktrees`). **3a is blast-radius isolation, not a security sandbox** — on the default tools path model code still runs in-process; the real boundary is Phase 3b (`container`/`sandbox`), whose inert seam now exists (`tools/mcp` launch params + guards, argv-tested) but whose executing sandbox is deferred to a daemon-bearing host (see `docs/phase3_execution_isolation_plan.md`, `sprints/18_execution_isolation_container/sprint_plan.md`).
- **`mcp_servers/coder_tools_server.py`** — the Coder's read/execute tools (`read_file`/`list_files`/`grep`/`run_tests`) as a stdio MCP server, delegating to `tools/coder_tools` (logic re-fronted, not rewritten). Run as `python -m loop_engine.mcp_servers.coder_tools_server`.
- **`mcp_servers/github_server.py`** (Phase 5, sprint 22b) — the system's **second** MCP server and its **first credentialed, network-enabled** one: the four `tools/repo_io` factory verbs (`create_repository`, `clone_repo`, `create_branch`, `open_pr`) re-fronted over MCP, native (not the official ~60-tool GitHub MCP server — that includes a `merge_pull_request` tool, contradicting "auto-merge prohibited", and can't be asserted to expose *exactly* four tools). Holds no credentials itself (`repo_io` shells to `gh`'s own auth) and imports side-effect-free — no `gh` call at import, so discovery (`list_tools`) stays offline/hermetic. Run as `python -m loop_engine.mcp_servers.github_server`. Reached **only** via `build_github_provider()` — a consumer-scoped provider structurally separate from `build_coder_tool_provider`, so these verbs never enter the model's coder tool loop (`tests/tools/test_mcp_provider.py` asserts the two providers' tool sets are exactly `{read_file, list_files, grep, run_tests}` and `{create_repository, clone_repo, create_branch, open_pr}`, disjoint). This sprint ships the capability only — no production flow chains the verbs yet (Sprint 23: trigger surface, maintenance flow, bootstrap flow).
- **`mcp_servers/issue_io_server.py`** (Phase 6, sprint 26) — the system's **third** MCP server, mirroring `github_server.py`'s shape exactly: the two `tools/issue_io` `gh`-touching primitives (`create_issue`, `read_issue`) re-fronted over MCP. Holds no credentials (`issue_io` shells to `gh`'s own auth, the same already-sanctioned surface `github_server`/`repo_io` use) and imports side-effect-free — no `gh` call at import, so discovery stays offline/hermetic. Run as `python -m loop_engine.mcp_servers.issue_io_server`. Reached **only** via `build_issue_provider()` — structurally separate from `build_coder_tool_provider`, so these verbs never enter the model's coder tool loop; `tests/tools/test_mcp_provider.py`'s disjointness assertion is now **three-way** (`coder_tools`/`github`/`issue`, pairwise disjoint). Capability + seams only (see the `core/engine.py` and `cli.py` bullets) — the classic direct `issue_io` calls remain the runtime default.
- **`tools/mcp/`** — the runtime MCP client: `MCPToolProvider` connects to one or more stdio servers on a background event-loop thread, discovers tools via `list_tools`, and exposes `(tools, execute)` to `run_tool_loop`, routing each call by tool name to its owning server session. **The only coder-tool path since Phase 6** — `LOOP_ENGINE_TOOLS` and the in-process `CODER_TOOLS`/`_execute_tool` dispatch are deleted. The Coder's `_CoderToolBackend` opens one provider per run (lazily, on first tool-loop use) and closes it in a `finally`. **Discovery is config-driven and consumer-scoped (Phase 5, sprint 22a):** `tools/mcp/config.py`'s `load_mcp_config()` reads a repo-root **`loop_engine.mcp.json`** (`{"servers": {"<name>": {"command", "args", "cwd"}}}`, pydantic `extra="forbid"`) and returns logical-name → launch-spec, merged over a built-in `coder_tools` default so file-absence is byte-identical to pre-22a behavior. As of sprint 22b, the file is **committed at the repo root** with a static `github` stanza (`{"command": "python", "args": ["-m", "loop_engine.mcp_servers.github_server"], "cwd": null}`; `command` resolves via PATH to the active-env interpreter on the factory host) — the first non-default server it declares; **sprint 26 adds a second, identically-shaped `issue` stanza** (`args: ["-m", "loop_engine.mcp_servers.issue_io_server"]`). `build_provider_for(names, *, cwd)` builds a provider scoped to **only** the named servers — a consumer names the servers it wants, so a `loop_engine.mcp.json` declaring other servers never leaks their tools to a caller that didn't ask for them. `build_coder_tool_provider(cwd)` is a thin wrapper (`build_provider_for(["coder_tools"], cwd=cwd)`) and still picks the `coder_tools` server's *launch params* by isolation mode (Phase 3b): `container` → `container_server_params` (docker/podman, mounts only the worktree), `sandbox` → `sandbox_server_params` (`bwrap`, daemon-free — secondary/inert), else the config's local profile — this isolation-aware override is a recognized special case for `coder_tools` only; every other named server (including `github`/`issue`) launches from its static config spec as-is. `build_github_provider()` (sprint 22b) and `build_issue_provider()` (sprint 26) are the `github`/`issue` counterparts: `build_provider_for(["github"])`/`build_provider_for(["issue"])`, no `cwd` override — both are orchestrator-only helpers, never named by the model's tool loop. Both sandbox builders raise `IsolationUnavailableError` when no runtime is present — the mode **never** silently degrades to in-process execution. (The old "container/sandbox requires `LOOP_ENGINE_TOOLS=mcp`" refusal is retired: with the in-process path deleted, tool execution *only* ever happens in the server subprocess, so the guarantee is structural rather than conditional.) The container/sandbox is spawned by the same MCP `stdio_client` that launches the local server (only `command`/`args` change), so it adds **no new sanctioned subprocess surface**; runtime detection is `shutil.which`, not a subprocess. **Inert on this branch** (no runtime here; `unshare -Ur` is denied): argv-shape tested only — the real `docker run` e2e + the sandboxed *gate* pytest are deferred (`sprints/18_execution_isolation_container/sprint_plan.md`, `docs/phase3_execution_isolation_plan.md`). **`loop_engine.mcp.json` is not `.mcp.json`** — see the containers/devcontainer section below for the latter, which is Claude Code's own project MCP config and belongs to a different tool entirely.
- **`tools/isolation.py`** — the single reader of `LOOP_ENGINE_ISOLATION` (`none`|`worktree`|`container`|`sandbox`; unknown value raises): `isolation_mode()`, `worktree_needed()` (true for all three isolating modes — container/sandbox still run inside a worktree the sandbox mounts), `sandbox_runtime_mode()` (the `container`/`sandbox` tool-sandboxing modes), and `IsolationUnavailableError` (imported by `core/coder_gate.py` so `core/` can signal "can't honor this mode" without importing `tools/mcp`). Imports no other `loop_engine` module.
- **`tools/logging_config.py`** — structured per-stage cost logging.
- **`cli.py`** — the Typer entrypoint (`run`, `resume`, `cost-summary`). A thin wrapper; all logic lives in the library layer. `run`'s `--resume-from` path stays inline (its own `_resolve_loop`, calling `run_graph_loop` directly — still a module-global lookup, so `cli.run_graph_loop`/`cli.LLMClient` remain patchable); its fresh-run path delegates to `runner.run_new`. Resume verifies snapshot stage names against the loop before slicing (never resumes a mismatched loop). **Phase 6, Sprint 26** added an analogous read seam for `resume --from-issue`: `_resolve_issue_reader()` returns the module-global `_issue_reader` (default `None`) or falls back to the classic `issue_io.read_issue`, a module-global pattern so tests can monkeypatch `cli._issue_reader` to inject `tools.issue_io.mcp_read_issue` bound to a fake provider. Default stays classic — no flag governs this seam either.
- **`runner.py`** (Phase 5, sprint 23) — the shared "start a fresh run from `human_input`" orchestration, factored out of `cli.run` so both the CLI and the trigger surface's dispatcher call one source of truth. `run_new(human_input, *, budget_usd, loop_name) -> State` mints a `run_id`, builds the initial `State`, constructs `LLMClient`, and runs `with worktree_run(run_id): _select_engine()(...)`. Carries its own `_resolve_loop`/`_select_engine`/`NAMED_LOOPS` (mirrored from `cli.py`, not shared — each module's engine indirection is independently patchable by its own tests) and `DEFAULT_BUDGET_USD` (the single source `cli.py` imports). **`run_in_tree(human_input, tree_path, *, budget_usd, loop_name) -> State`** (Phase 5 piece 3, sprint 24) is a third entrypoint for `flows/maintenance`: same loop-build/`State`-init as `run_new`, but chdir's into `tree_path` (a foreign clone) instead of opening `worktree_run` — the clone *is* the isolation boundary, and `worktree_run` would chdir into the orchestrator's own `.worktrees/<run_id>` instead, the wrong tree. Cwd is restored in a `finally` on every exit path, including an engine exception; this holds even with `LOOP_ENGINE_ISOLATION=worktree` set, since `run_in_tree` never calls `worktree_run`. State snapshots are left to follow the engine's normal cwd-relative behavior (not pinned back to the original cwd) — the clone is the run's tree, so its snapshots belong alongside it.
- **`trigger/`** (Phase 5 piece 2, sprint 23) — the trigger surface: a FastAPI webhook that turns a GitHub event into a real `runner.run_new` call. `trigger/parse.py` — `RunRequest` (pydantic, `extra="forbid"`: `human_input`, `budget_usd`, `loop_name`, plus `repo_full_name`/`issue_number` provenance) and `parse_event(event_name, payload) -> RunRequest | None`, implementing the locked grammar (an `issues`/`labeled` event with label `agent-action`, or an `issue_comment`/`created` event whose first non-empty line is `/agent-run`; `human_input` = issue `title + "\n\n" + body`; everything else — including `ping` — is `None`, defensively, never a raise). `trigger/dispatch.py` — the `RunDispatcher` `Protocol` (`async def dispatch(request) -> None`, fire-and-forget) and its sole 23 implementation `InProcessDispatcher`: in-memory dedupe on `(repo_full_name, issue_number)`, then `runner.run_new` scheduled on a worker thread (`asyncio.to_thread`) via `asyncio.create_task` so `dispatch` returns before the run finishes and the event loop never blocks; the created task is retained in a strong-referenced `self._tasks` set (dropped via `add_done_callback` on completion) rather than fire-and-forget-unreferenced, closing a GC-cancellation window that could silently drop a run and permanently wedge its dedupe key (23a review-fix), and a run failure is caught and logged (`logger.exception`, key only, never the payload) rather than surfacing only as asyncio's default "never retrieved" noise. `trigger/app.py` — `create_app(dispatcher=None) -> FastAPI` reads `LOOP_ENGINE_WEBHOOK_SECRET` at construction and **fails closed** (raises) if unset; the single `POST /webhook` route reads the raw body, HMAC-SHA256-verifies `X-Hub-Signature-256` over those exact raw bytes (`hmac.compare_digest`) before any JSON parsing, then parses the already-read raw body via `json.loads` (an authenticated-but-unparseable body returns `400`, never `500` — 23a review-fix) → `204` (no-op) or `dispatch()` → `202`; a module-level `app` is built eagerly only when the secret is already set in the environment (so importing the module never raises in a dev/test environment lacking it) for eventual `uvicorn` hosting (not shipped in 23). Boundary posture (`tests/trigger/test_boundaries.py`, AST-scanned like the `keyring`/`state_io` boundary tests): no `keyring` import, no direct file write, no subprocess surface (dispatch is in-process — the sanctioned-surface count stays three). All 23 coverage is hermetic (`TestClient` + injected fake dispatcher + patched `runner.run_new`); real webhook delivery is deferred (`sprints/DEFERRED_VERIFICATION.md` §6).
- **`flows/maintenance/`** (Phase 5 piece 3, sprint 24) — the maintenance flow: the first caller to chain `tools/repo_io`'s factory verbs into an end-to-end run over a target repo. `MaintenanceRequest` (pydantic, `extra="forbid"`: `repo_full_name`, `human_input`, `branch`, `budget_usd`, `loop_name`, `base="develop"`, optional `clone_dest`) drives `run_maintenance(request, *, run_step=runner.run_in_tree, repo_io=..., git_io=..., run_tests=...) -> MaintenanceResult`, which chains: `repo_io.clone_repo` → `git_io.checkout_branch` → `run_step` (the injectable `RunStep` `Protocol`, concretely `runner.run_in_tree` — cwd pinned to the clone, **not** `worktree_run`) → a **completion** guard (the inner run's returned `State` must be `COMPLETED`; `FAILED_STAGE`/`BUDGET_EXCEEDED`/`AWAITING_ISSUE` → `status=run_incomplete`, so a partial or human-paused tree that happens to pass pytest is never shipped as a PR) → a **no-change** guard (`git_io.has_changes` — a completed run that produced no diff → `status=no_changes`, no gate, no commit) → a green gate (`coder_tools.run_pytest` against the clone's `src/`, the layout convention managed repos are scaffolded with) → **green-only** `git_io.commit_all` + `git_io.push_branch` + `repo_io.open_pr` (`MaintenanceResult(status=opened_pr, pull=...)`); a non-completed run, an empty diff, or a **red** gate short-circuits before any git write (no commit/push/PR). `open_pr`'s terminal-call, no-merge-verb property (inherited from `tools/repo_io`) keeps auto-merge impossible regardless of gate outcome. Every collaborator is injectable so tests fake every external effect. Does **not** wire a CLI subcommand or the trigger surface (23) to dispatch this flow, and does **not** build the bootstrap flow (piece 4) — Sprint 25+. Boundary posture (`tests/flows/test_boundaries.py`, same AST-scan style as `trigger/`'s): no `keyring` import, no direct file write, no subprocess surface of its own — the one new surface (`tools/git_io`) is *called*, not introduced here. Coverage is hermetic: `tests/flows/maintenance/test_flow.py` fakes every collaborator to prove call order/gating; `tests/flows/maintenance/test_integration.py` additionally exercises real `tools/git_io` against a `tmp_path` repo + local bare remote (only `repo_io`/the loop run are faked) to prove the green path actually pushes + opens a PR and the red path touches neither. Real `gh repo clone` → real default-loop run → real push → real PR is deferred (`sprints/DEFERRED_VERIFICATION.md`).

- **`flows/bootstrap/`** (Phase 5 piece 4, sprint 25) — the bootstrap flow: the factory verb that brings a new, conventions-conformant repo into *existence*, a sibling of `flows/maintenance`. `BootstrapRequest` (pydantic, `extra="forbid"`: `name`, `org="glunk-works"`, `kind="python"`, `private=True`, `default_branch="main"`, `integration_branch="develop"`, optional `clone_dest`; a derived `dest` property and a derived, sanitized `pkg_name` property reusing `tools/scaffold`'s own sanitizer) drives `run_bootstrap(request, *, repo_io=..., git_io=..., scaffold=...) -> BootstrapResult`, which chains: `repo_io.create_repository` → `repo_io.clone_repo` (empty tree) → `git_io.checkout_branch(default_branch)` (names the unborn default branch deterministically — a fresh `gh repo create`+clone lands on an unborn HEAD whose name follows the cloner's `init.defaultBranch`, which may not be `main`) → `tools/scaffold.write_skeleton` → `git_io.commit_all` → `git_io.push_branch(default_branch)` → `repo_io.create_branch(integration_branch, base=default_branch)` (ordering is load-bearing: `create_branch` reads the base ref's SHA over the API, so it must run *after* the push puts that ref on the remote) → `BootstrapResult(status=created, repo=..., default_branch=..., integration_branch=...)`. Deliberately **no** inner loop (`run_in_tree`), **no** LLM run/budget, **no** green gate, and **no** `open_pr`/CLI subcommand/trigger wiring — a brand-new repo has no commits and nothing to review into; the actual feature work is a later `flows/maintenance` run or trigger dispatch. Boundary posture identical to `flows/maintenance` (`tests/flows/test_boundaries.py`, which provably enumerates `flows/bootstrap`): no `keyring`, no direct file write, no subprocess surface of its own. Coverage is hermetic: `tests/flows/bootstrap/test_flow.py` fakes every collaborator to prove call order/ordering (incl. `create_branch` firing after the push); `tests/flows/bootstrap/test_integration.py` additionally exercises real `tools/scaffold` + real `tools/git_io` against a `tmp_path` repo (on a non-`main` initial branch name, proving the unborn-HEAD handling) + a local bare remote (only `repo_io` is faked) to prove the skeleton actually lands and `main` is really pushed before `develop` is created. Real `gh repo create`/clone/push/`create_branch` is deferred (`sprints/DEFERRED_VERIFICATION.md`).

`tests/` mirrors `src/loop_engine/` layout, plus `tests/integration/` for cross-module tests (e.g. `test_budget_abort.py`, `test_no_credential_leakage.py`).

## Containers / devcontainer

- Root `Dockerfile` is multi-stage: `dev` (full contributor toolchain — hatch, ruff, git, gh) and `prod` (minimal runtime). Both are local-only, no registry publish.
- `.devcontainer/` targets the `dev` stage. It bind-mounts a forwarded host GPG agent socket (`GPG_HOST_DIR`) for commit signing, and provisions `ANTHROPIC_API_KEY`, `GITHUB_PERSONAL_ACCESS_TOKEN`, and `LOOP_ENGINE_KEYRING_PASSPHRASE` from Infisical (project identified by root `.infisical.json`, machine identity via `INFISICAL_CLIENT_ID`/`INFISICAL_CLIENT_SECRET` forwarded from the host shell) on every container start via `infisical-start.sh` → `seed-secrets.sh`. `infisical run` under machine-identity auth needs `--projectId` and `--path=/loop-engine` passed explicitly — it does not read `.infisical.json`'s `workspaceId` automatically the way interactive login does. The keyring passphrase file must be written *before* anything calls `keyring.set_password`, since the container's cryptfile keyring backend reads that file on every call.
- A bare Linux container has no OS-native keyring backend, so `containers/keyring_backend/cryptfile_backend.py` is wired in as a custom encrypted-file backend via `keyring`'s own backend-discovery config (not the PyPI `keyrings.cryptfile` package). It reads two *file paths* from env vars (`LOOP_ENGINE_KEYRING_FILE`, `LOOP_ENGINE_KEYRING_PASSPHRASE_FILE`) — never secret values directly.
- `.mcp.json` connects Claude Code/Cursor to GitHub's hosted remote MCP server using a bearer-token `GITHUB_PERSONAL_ACCESS_TOKEN`, sourced via `gh auth token` (itself seeded by the Infisical flow above) rather than OAuth, so the devcontainer can authenticate non-interactively at container start. **This is not the same file as `loop_engine.mcp.json`** (see `tools/mcp/` above) — `.mcp.json` is Claude Code's own project MCP config (`{"mcpServers": {...}}`, dev-tooling only, committed `65ed47c`), while `loop_engine.mcp.json` is loop-engine's own product-runtime stdio server-launch config, read only by `tools/mcp/config.py`. Neither module reads the other's file; the two schemas are incompatible on purpose (a stray `.mcp.json` in the repo root is why `load_mcp_config()` must never default to that filename — see the Phase 5 sprint 22a revision note in `docs/migration_roadmap.md`).

See `docs/architecture_definition.md` for the full architecture and threat-model writeup, and the README's "Running in a container" section for command-level detail on all of the above.
