## What this is

loop-orchestrator runs a named sequence of decoupled AI "persona" stages against a single, explicit, versioned `State` object. The default loop is a **PM → Architecture → Agile Sprint Breakdown → Coder/IaC** pipeline, but it is not a one-way conveyor: every stage's output passes a content **gate** (accept / revise / escalate), questions escalate up a resolver ladder (Coder → Architect → PM → human via GitHub issue), and resolved questions route rework back down by blast radius ("task" re-runs the asker, "plan" re-enters Sprint Breakdown, "architecture" re-enters the Architect). A snapshot is persisted after every accepted stage AND on every exit path (completed / failed / budget-exceeded / awaiting-issue).

## Working here: personas & model routing

Development on this repo is split by model to keep each session lean and single-model
(the token/session-limit fix). The workflow is externalized into `.ai/` and driven by
skills — **`/resume`** (rehydrate from `.ai/` at the start of a session), **`/handoff`**
(serialize state before switching model/session), **`/critic-gate`** (the read-only QA-critic
pass over a coding diff, run after the green gate and before `/handoff` — defense-in-depth,
**not** the `architect-review` CI gate), and **`/archive-sprint`** (retire a completed,
HITL Gate-approved sprint). See `.ai/context/workflow.md` for the protocol and the **agent
catalog** (the `coder` / `architect` / `security-critic` / `guard-adversary` /
`mutation-triage` / `live-verify` / `docs-consistency` subagents and their trigger points).

- **Architect (Opus).** Architecture, design, sprint/phase **planning** (planning pass, one question at a time, HITL Gates), **Architect Review** of a coding session's diff, module-boundary decisions, non-trivial debugging, and roadmap/memory updates. This is the default model for planning and review sessions.
- **Coder (Sonnet).** Implementing an already-**defined** sprint task, writing/adjusting tests, mechanical refactors, running the green gate, fixing lint. Sonnet runs the implementation sessions (or is dispatched as the `coder` subagent for a small in-session task).

Rule of thumb: if the task requires deciding *what* to build or *whether* a diff is
correct → **Opus**; if the task is executing a spec that already exists → **Sonnet**.
Switch at sprint boundaries via `/handoff` → fresh session → `/resume`.

**Every sprint lands via a pull request — a merged PR is the human approval.** Work on a
`sprint/NN-slug` branch cut from `main`; commit and push freely there, then open a PR whose
**base is `main`**. Post the Opus Architect Review on it with `gh pr review --comment` (**never
`--approve`** — the human's merge is the approval, and `gh` authenticates as the PR author
anyway). **Never merge, and never force-push a pushed branch.** Full protocol in
`.ai/context/workflow.md` (including a historical note on the one-time `feat/mcp-langgraph-migration`
merge commit that landed the migration on `main` in sprint 35).

> **The Architect Review is a CI gate, not a courtesy — and it runs in a FRESH session.**
> Any PR touching `src/` fails the `architect-review` check until a review headed
> `**Opus/Architect HITL review (automated)**`, carrying the fresh-session attestation, is
> posted against its **current head commit** (`.github/workflows/hitl-review.yml`). That header
> is **frozen protocol** — it predates the Architect Review / HITL Gate naming and is matched by
> literal `contains()`, so paste it verbatim and never "fix" it to match this vocabulary. That check
> is **enforced**: the `protected-integration-branches` ruleset makes it a required check on
> `main` and `feat/**`, so an unreviewed `src/` PR cannot be merged (see **BL-11** — until
> sprint 33 the repo had *no* protection at all and every check here was advisory).
> **`/model opus` mid-session is not a review session** — switching model does not clear
> context, so the reviewer proofreads its own reasoning instead of re-deriving it. The
> sequence is `/handoff` → **new session** → `/resume` → `/code-review` → post.
> It is a gate because the convention failed: sprint 27's Task 8 shipped green with an R8 fix
> that covered `cli.py` and left every fresh-run path still filing issues on the wrong repo —
> the review that would have caught it was skipped and nothing noticed. Docs-only PRs are
> exempt. Full protocol in `.ai/context/workflow.md`.

## Commands

```bash
hatch run test                        # pytest (full suite)
hatch run test tests/core/test_engine.py            # single file
hatch run test tests/core/test_engine.py::test_name  # single test
hatch run test-parallel               # pytest-xdist, -n auto --dist=loadscope (CI uses this; ~1.7-1.8x faster on the full suite, not worth it for a single file/test)
hatch run lint                        # ruff check . (incl. S/bandit and B/bugbear rule sets)
hatch run format                      # ruff format .
hatch run audit                       # pip-audit CVE scan of pinned deps (CI gate)
hatch run sbom                        # regenerate sbom.json (CycloneDX) — required whenever pyproject.toml deps change
```

Run the loop itself:

```bash
hatch run loop-orchestrator run --input path/to/requirements.md --budget 5.00
hatch run loop-orchestrator run --resume-from state/<run_id>/01_ArchitectureGenerator.json
hatch run loop-orchestrator resume --from-issue <N>   # after answering a paused run's GitHub issue
hatch run loop-orchestrator cost-summary --run-id <run_id>
```

Exit codes from `run`/`resume`: 0 completed, 2 awaiting a GitHub issue answer, 3 budget exceeded, 4 aborted by the human (the pending issue was closed without an answers comment — distinct from 1, an unexpected failure), 5 awaiting a Slack answer (a run paused with `LOOP_ORCHESTRATOR_ESCALATION_TRANSPORT=slack` — resumed by the `slack-listen` daemon when a human replies in the escalation thread, **not** via `resume --from-issue`).
(The `loop-orchestrator resume` CLI subcommand is unrelated to the `/resume` dev-workflow skill.)

CI (`.github/workflows/ci.yml`) runs the unconditional chain `lint` → `format-check` → `test` → `secrets-scan` (gitleaks) / `dependency-audit` → `sbom`; no job carries an `if:`, so none can be **skipped by a condition**. `skipped` is still possible though: a job whose `needs:` dependency *fails* reports `skipped` too — safe only because the upstream **failure** blocks the merge on its own, not because `skipped` can't happen. Don't restate this as "no job can ever report `skipped`" (roadmap BL-10/BL-18). Three sibling workflows gate or guard independently, each gating nothing in `ci.yml` and gated by nothing: `pr-title.yml` (validates the PR title), `hitl-review.yml` (enforces the Architect review on any PR touching `src/`), and `ruleset-drift.yml` (daily cron + `workflow_dispatch`; fails loudly if the ruleset below is weakened or removed — deliberately **not** itself required, per BL-11/FD2). Every `uses:` is pinned to a commit SHA, not a floating tag, enforced by `tests/test_ci_config.py`. All eight checks (`lint`, `format-check`, `test`, `secrets-scan`, `dependency-audit`, `sbom`, `pr-title`, `architect-review`) are **required** by the `protected-integration-branches` ruleset on `main` + `feat/**`, which also blocks force-pushes/deletion and requires a PR to merge (full merge bar in `sprints/GLOBAL_DEFINITION_OF_DONE.md`). Required checks match by **check-run name** = job id, so **never add a `name:` override to those jobs** — it renames the check run and strands the requirement (`tests/test_ci_config.py` pins this; BL-11/FD5). The API key is **never** a CLI flag or env var — OS keyring only (setup + fallback in `.ai/context/modules.md`).

## Enforced module boundaries

These are checked by static tests, not just convention — don't casually violate them:

- `core/` imports no concrete persona module, only `personas/base.py`; it is otherwise unrestricted on `tools/*`. Live exception: `core/coder_gate.py` imports `tools/mcp` (scoped to that one `core/` file) so the `RalphCoderGate`'s evidence pytest can dispatch via `tools/mcp.run_gate_pytest` (in-process on `none`/`worktree`, sandboxed provider on `container`/`sandbox`). Footgun: `tools/issue_io/mcp_client.py`'s imports of `tools/mcp`/`repo_io`/`worktree` are **function-scoped on purpose** (finding F7) — hoisting them to module scope would transitively pull the MCP client stack into `core/engine`'s import graph through the `issue_filer` write seam.
- `tools/state_io` and `tools/scaffold` are the file-write-owning modules (`open`/`write_text`/`write_bytes`); everything else goes through `write_artifact`/`write_state_snapshot`. `tools/scaffold` (Phase 5 piece 4) is the **second** such surface — it writes a bundled skeleton (Python templates + the injected Global Conventions `CLAUDE.md`) into a foreign clone tree, validated via `repo_io._validate_clone_dest`; it imports no `subprocess`/`keyring`, so the five sanctioned subprocess surfaces below are unchanged.
- `tools/llm/client.py` is the only module that imports `keyring`. (`tools/llm/pricing.py` owns the `RATES` cost table and the canonical `DEFAULT_MODEL` fallback — the single source both non-declarative personas import; declarative personas route via their own `GeneratorConfig.model`.)
- `tools/issue_io` and `tools/repo_io` are the GitHub-owning modules — `issue_io` files/reads human-escalation issues, `repo_io` is the repo/branch/PR factory: **five** verbs (`create_repository`, `clone_repo`, `create_branch`, `open_pr`, `create_ruleset`; **no merge verb — auto-merge is prohibited**) plus repo introspection (`resolve_repo_slug`). `resolve_repo_slug` and `create_ruleset` are **orchestrator-only** `repo_io` verbs that never enter `mcp_servers/github_server` — the github server's four-verb set stays pinned and pairwise-disjoint. Both shell the already-authenticated `gh`; no other module talks to GitHub. (`resolve_repo_slug` gives the issue filer an **explicit** destination instead of `gh`'s implicit CWD resolution — finding R8; roadmap has the detail.)
- `tools/coder_tools/` is read/execute-only: paths are traversal- and symlink-validated (reusing `state_io`'s validator); its `run_tests` pytest subprocess (also used by the Coder gate) is a sanctioned subprocess surface. It runs model-generated code — the operating assumption is the sandboxed devcontainer. `tools/coder_tools/run_lint.py` (Sprint 29) is a **second** subprocess surface in this module — `ruff check` + `ruff format --check` against a validated path, same containment (fixed argv, `shell=False`, hard timeout, output-capped). Unlike `run_tests`, ruff statically parses the target and never executes model-generated code, so it is strictly lower-risk; it stays in the model's Coder tool loop only, never a gate verb (gate-enforced lint is out of scope for Sprint 29).
- `tools/git_io` (Phase 5 piece 3) owns local-git working-tree writes (`checkout_branch`, `commit_all`, `push_branch`) plus a read-only `has_changes` probe (`git status --porcelain`, so a no-op run never reaches `commit_all`'s empty-index failure) in a **foreign** cloned tree — distinct from `repo_io` (shells `gh`, the remote GitHub API) and `worktree` (the orchestrator's own per-run isolation, keyed off `run_id`). It mirrors `tools/worktree/manager.py::_git`'s posture exactly (fixed argv, `shell=False`, a hard timeout, `check`-handling) and validates every `tree` argument by reusing `tools/repo_io/github.py::_validate_clone_dest`. `git push` rides `gh`'s clone-established credential helper — no `keyring` import, no new credential path.
- Sanctioned subprocess surfaces are exactly **five**, each fixed-argv and `shell=False`: `coder_tools`' `pytest`, `coder_tools`' `ruff` (Sprint 29, `run_lint.py` — statically parses, never executes model-generated code), `issue_io`'s **and `repo_io`'s** `gh` (two consumers of the same surface — `repo_io` adds no sixth), `tools/worktree`'s `git worktree` (args derive only from a `validate_run_id`-checked run_id), and `tools/git_io`'s local `git` (args derive only from a `_validate_clone_dest`-checked tree). Nothing else shells out (`tests/tools/test_subprocess_surfaces.py`). The Phase 3b container/sandbox launch is **not** a sixth surface: it is spawned by the MCP `stdio_client` (the same mechanism that launches the local coder-tools server — only `command`/`args` differ), and runtime detection uses `shutil.which`, not a subprocess.
- `mcp_servers/` re-front native tools over MCP, each delegating with **no credentials of its own** (they ride `gh`'s auth or run read/execute-only): `coder_tools_server` (→ `tools/coder_tools`, same path validation), `github_server` (→ `tools/repo_io`, the four factory verbs), `issue_io_server` (→ `tools/issue_io`, the two escalation verbs `{create_issue, read_issue}`). Execution runs in the server subprocess, out of the orchestrator — the boundary is **moved, not relaxed**. This is the **only** coder-tool path; the in-process `CODER_TOOLS`/`_execute_tool` dispatch is deleted, so there is no unsandboxed fallback. `tools/mcp` is the client side (discovery + dispatch; no keyring, no file writes): `tools/mcp/config.py` reads the repo-root **`loop_orchestrator.mcp.json`** (distinct from Claude Code's own `.mcp.json`) and `build_provider_for(names, ...)` builds a provider for **only** the named servers. The three tool sets — coder `{read_file, list_files, grep, run_tests, run_lint}`, github `{create_repository, clone_repo, create_branch, open_pr}`, issue `{create_issue, read_issue}` — are asserted **pairwise disjoint** (`tests/tools/test_mcp_provider.py`). The github and issue verbs are **orchestrator-invoked only** (`build_github_provider()`/`build_issue_provider()`), never in the model's coder loop; the `issue_filer` write seam and `resume --from-issue` reader seam default to this MCP route. `create_issue`/`read_issue` take an **explicit** `repo` (owner/repo), resolved from `worktree.origin_cwd()` so **every** entrypoint targets the managed repo rather than loop-orchestrator itself — the bug that filed managed-repo escalations on the project repo (finding R8). `resume --from-issue` takes `--repo` and verifies the issue's URL against the snapshot's `pending_issue`. Deep history: `docs/migration_roadmap.md` Phase 6, R1–R10.
- Any change touching `State` must keep `schema_version` accurate (bump it and extend `migrate_state_payload` for breaking shape changes) and keep `extra="forbid"` intact.
- `trigger/` (Phase 5 piece 2) is a new top-level **orchestrator-level caller**, a sibling of `cli.py` — it may import `core`/the default loop/`State`/`LLMClient`/`worktree` (via the shared `runner.py`), but is not a `tools/` module and not an MCP server. Its boundary posture, asserted by `tests/trigger/test_boundaries.py`: imports no `keyring`, writes no files directly, and adds **no subprocess surface** — dispatch (`trigger/dispatch.py`'s `InProcessDispatcher`) runs the loop in-process on a worker thread. The webhook's HMAC secret comes from `LOOP_ORCHESTRATOR_WEBHOOK_SECRET`, an **env var**, not the keyring — it authenticates an inbound request, not an outbound LLM call, so it is a distinct credential class from the keyring-only Anthropic key.
- `flows/` (Phase 5 piece 3) is another new top-level **orchestrator-level caller**, a sibling of `cli.py`/`trigger/`. `flows/maintenance` chains `repo_io.clone_repo` → `git_io.checkout_branch` → `runner.run_in_tree` (the default loop, cwd pinned to the clone, **not** `worktree_run` — the clone is its own isolation boundary) → a **completion** guard (the inner run must end `COMPLETED`; a `FAILED_STAGE`/`BUDGET_EXCEEDED`/`AWAITING_ISSUE` run short-circuits to `run_incomplete` so a human-paused tree is never shipped) → a **no-change** guard (`git_io.has_changes`; a no-op run short-circuits to `no_changes`) → a green gate (`coder_tools.run_tests` on the clone) → **green-only** `git_io.commit_all`/`push_branch` + `repo_io.open_pr` (base defaults to `develop`); a non-completed run, an empty diff, or a red gate ⇒ no commit/push/PR. `open_pr` stays the terminal GitHub call — no merge verb exists, so auto-merge is impossible here too. Its boundary posture, asserted by `tests/flows/test_boundaries.py`: imports no `keyring`, writes no files directly, and adds **no subprocess surface of its own** — the one new surface, `tools/git_io`'s local `git`, is *called*, not introduced by `flows/`.
- `flows/bootstrap` (sibling of `flows/maintenance`) brings a new conventions-conformant repo into *existence*: `create_repository` (`private=False` by default — a ruleset is only installable on a public repo under the org's Free plan, so protection is the default and privacy knowingly forfeits it) → `clone_repo` → `git_io.checkout_branch(main)` → `tools/scaffold.write_skeleton` → `commit_all`/`push_branch(main)` → `create_branch(develop, base=main)` (**ordering is load-bearing** — `create_branch` reads the base ref's SHA over the API, so it runs only *after* the push) → `create_ruleset(main, develop)`, run **last** and skipped when `private=True` (a `pull_request` rule on `main` installed earlier would reject the scaffold's own initial push — FD7). A live `create_ruleset` failure returns `BootstrapStatus.RULESET_FAILED` with the `RepoRef` intact so a caller can tear down the unprotected repo. Deliberately **no** inner loop, **no** green gate, **no** `open_pr` — a brand-new repo has nothing to PR into; auto-merge stays impossible. Same boundary posture as `flows/maintenance` (no `keyring`, no direct file write, no subprocess surface of its own; `tests/flows/test_boundaries.py`).
- `tools/slack_io` (Sprint 39, BL-2 pass 1; **both directions** since Sprint 40, pass 2) is the Slack **transport**, and the only module that imports `slack_sdk` — every such import is **function-scoped** (`SlackNotifier.emit`, `build_listener_from_env`, `resolve_channel_id`, `send_ephemeral_reply`), never module scope, so neither the notifier's no-op default nor `cli.py`'s import of the daemon ever pulls it into the import graph — the same F7 posture as `tools/issue_io/mcp_client.py`'s deferred `tools/mcp` import. **Outbound:** `core/notify.py` is the pure, leaf `EventKind`/`LifecycleEvent`/`Notifier`/`NoOpNotifier` contract (no `slack_sdk` import); `core/graph_engine.py`'s `run_graph_loop` resolves a notifier via `build_notifier_from_env()` and emits lifecycle events through that seam, **fail-open at the call site** (`_safe_emit`) — a notification failure must never fail a run. **Inbound:** `build_listener_from_env()` **fails closed** instead (raises, never a no-op listener) — the deliberate inverse of the notifier's inert-by-default, because a trigger surface must never start half-configured (FD4); `resolve_channel_id()` (FD3) resolves name/`#name`/ID → ID once at startup, since Socket Mode payloads carry the ID only, and fails closed on any error; `send_ephemeral_reply()` is fail-open like the notifier. **Escalation round-trip (Sprint 41, BL-2 pass 3):** this same module is also the **outbound escalation filer** — `slack_escalation_filer`/`render_question_message` post a paused run's questions to `LOOP_ORCHESTRATOR_SLACK_CHANNEL` (mrkdwn-escaped, per-question length-capped) and `send_thread_message` posts the threaded ack/outcome (fail-open); the pure `parse_thread_answers(text, unresolved_count)` maps a numbered thread reply back to `{n: answer}` (bare prose ⇒ `{1: body}` only when exactly one question is open). Credentials are `LOOP_ORCHESTRATOR_SLACK_APP_TOKEN` (Socket Mode) / `LOOP_ORCHESTRATOR_SLACK_BOT_TOKEN` / `LOOP_ORCHESTRATOR_SLACK_CHANNEL`, **env vars, not keyring** (FD3) — a distinct credential class from the keyring-only Anthropic key, same posture as `trigger/`'s `LOOP_ORCHESTRATOR_WEBHOOK_SECRET`. Adds no subprocess surface, so the five sanctioned surfaces above are unchanged.
- `slack_control/` (Sprint 40, BL-2 pass 2) is the **Slack inbound trigger** and another top-level **orchestrator-level caller** — a sibling of `cli.py`/`trigger/`/`flows/`, not a `tools/` module and not an MCP server. `command.py` is the pure FD5 parser (`/agent-run --budget <n> <requirements>`; **never raises**, and never defaults the budget — fail-closed on the money cap); `SlackRunDispatcher` mirrors `trigger/`'s `InProcessDispatcher` (FD6) but dedupes on **`envelope_id`** (Socket Mode redelivers unacked envelopes) and serializes runs behind one `_run_lock` (BL-8's process-global `os.chdir`); `SlackDaemon` wires the listener → the **FD3 channel-scope guard** (anything from another channel is dropped silently) → parse → dispatch → an ephemeral reply, bridging the SDK's thread to the loop via `asyncio.run_coroutine_threadsafe` and never raising out of that callback. `build_daemon_from_env()` fails closed on all three env vars **and** resolves the channel before any socket opens. **Escalation round-trip (Sprint 41, BL-2 pass 3):** `daemon.py` now branches on `request.type` — the pass-2 `slash_commands` path is untouched, and a new `events_api` **message-event** path applies the same FD3 channel guard (against `payload["event"]["channel"]`, **before any state read**), ignores the bot's own posts (`bot_id`/`bot_message`/`message_changed`/`message_deleted`), correlates a thread reply to a paused snapshot via `tools/state_io.find_paused_snapshot_by_slack_thread` (latest-per-run-dir by **mtime**, since a blast-radius re-entry can write a lower stage-index file *later*), and resumes through `dispatch_resume` → `runner.resume_run`. `dispatch_resume` shares `dispatch`'s `_run_lock` + `envelope_id` dedupe and adds a `thread_ts` dedupe (`_active_threads`) closing the distinct-envelope-same-thread double-dispatch window. This **widens** the boundary posture: `slack_control/` now **reads the run-state tree (only via `tools/state_io`, never raw file I/O) and resumes runs via `runner`**, while still importing no `keyring`/`slack_sdk`, writing no files directly, and adding no subprocess surface (`tests/slack_control/test_boundaries.py`). The `core/` pause path is transport-agnostic via an `EscalationFiler` seam (`SlackRef` ⇒ finalize `AWAITING_SLACK` + store `State.pending_slack`; `IssueRef` ⇒ `AWAITING_ISSUE` + `pending_issue`), the schema-**v5** `pending_slack: SlackRef | None` field, and `build_escalation_filer_from_env()` (default `issue`; **fail-closed** when `LOOP_ORCHESTRATOR_ESCALATION_TRANSPORT=slack` but the Slack vars are unset). Deliberately shares no code with the parked `trigger/` (FD1: supersede, not unify). Entry point: `loop-orchestrator slack-listen`.
- `tools/inventory_db` (bounty loop Phase 0, sprint 44) is the Postgres persistence layer for the second (`loops/bounty/`) loop — `psycopg_impl.py` is the **sole** module permitted to import `psycopg` (`tests/tools/inventory_db/test_boundary.py`, the psycopg analog of the keyring boundary test above), and every `cur.execute()` call in it passes a static string literal as its SQL text, with all caller-supplied values flowing through the bound-`%s`-parameters tuple only (`tests/tools/inventory_db/test_sql_parameterization.py`) — the SQL-sink analog of fixed-argv/`shell=False` on the five sanctioned subprocess surfaces. `InMemoryInventory` (the hermetic fake) and `PsycopgInventory` (the real driver) implement the same `InventoryRepository` protocol with identical coalesce-on-`None` upsert semantics. The two JSONB columns (`assets.raw_scan_data`, `endpoints.tech_stack`) must be bound wrapped in `psycopg.types.json.Jsonb(...)` — psycopg 3 has no dict dumper, so a bare `dict` bind raises `ProgrammingError`; a bare `None` stays `None` (never `Jsonb(None)`, which would write JSON `null` instead of SQL `NULL`) (F1, PR #162 T2 review, fixed in the sprint-44 remainder).
