### FILEPATH: /sprints/39_bl2_slack_notify/sprint_plan.md

**Sprint Goal:** Land the **first surface** of BL-2's Slack "bot control plane" — **outbound notifications** — so a run's lifecycle events (started / completed / failed / budget-exceeded / **awaiting-issue**) surface in Slack. The **awaiting-issue** pause is the highest-value signal: today the human-escalation pause is visible only as a filed GitHub issue, and nobody is told it happened. The deliverable is a new `tools/slack_io` module (a thin wrapper over Slack's official `slack_sdk`), a **fail-open notifier seam** threaded through `run_graph_loop` exactly as the existing `issue_filer` seam is, and a message formatter for the **six event kinds** (the five lifecycle events above plus a `crashed` notice on an unexpected engine exception) — all inert by default (no-op unless configured), so every existing call site, test, and CI run behaves exactly as today. This is BL-2 **pass 1 of 3**; the inbound trigger (Sprint 2) and the escalation round-trip (Sprint 3) are deliberately out of scope.

**Out of scope:**
- **The inbound trigger surface (Slack command/message → kick off a run).** That is BL-2 **Sprint 2**, and it carries its own unresolved architecture question: Socket Mode (an outbound WebSocket that sidesteps BL-24's hosting blocker) **vs.** a Slack MCP gateway (the official cloud server is OAuth/read-search-oriented; a self-hosted one would be the repo's first third-party externally-credentialed MCP server). That decision is **explicitly deferred to Sprint 2 planning** — do not pre-commit the transport here. Sprint 1 adds **no inbound surface**, so it is entirely independent of BL-24.
- **The escalation round-trip (route a paused run's questions to a channel and fold the reply back).** BL-2 **Sprint 3** — bidirectional and stateful, and it *depends on* both notify (this sprint) and the inbound surface (Sprint 2) existing first.
- **A Slack MCP server / gateway of any kind (this sprint).** Owner-confirmed (FD2): outbound notify is a single, no-untrusted-code side-effect emitted from a headless daemon under a bot token. MCP's role in this repo is process isolation for *model-influenced execution*; notify has none, so a gateway buys nothing here and would disturb the first-party/credential-less MCP boundary. The "standard gateway, don't roll our own" instinct is satisfied at the right layer by the **official `slack_sdk`**. Revisit MCP-vs-direct at Sprint 2, where the interaction is genuinely tool-like.
- **Any change to `core/`'s decision logic, exit codes, or terminal-status semantics.** The seam only *observes*; it must never alter a run's outcome, exit code, or control flow. `cli.py`'s `_EXIT_CODES` map is untouched.
- **Discharging BL-24.** Notify does not touch the GitHub webhook; BL-24 stays owed and unaffected.

**Context (locked decisions — this planning pass, Opus/Architect, owner-confirmed; do not re-open):**

- **FD1 — Sequencing: outbound notify first; inbound + escalation are later sprints** *(locked 2026-07-16)*. Of BL-2's three candidate surfaces, notify is the only one that adds **no inbound surface**, is **independent of BL-24**, and delivers the **highest-value signal** (awaiting-issue) at the lowest risk. It also establishes the credential class and the module-boundary posture the later surfaces build on. The end-state remains the full bot control plane; this is pass 1 of 3.

- **FD2 — Transport is the official `slack_sdk` inside a new `tools/slack_io` module, NOT an MCP server** *(locked 2026-07-16, owner-confirmed after weighing the Slack MCP option)*. Slack's *official* MCP server is cloud-hosted, OAuth-based, and aimed at AI-client read/search — the wrong direction and wrong credential model for a headless bot-token notify daemon (it would reopen FD3). A self-hosted MCP server would be the repo's first third-party, externally-credentialed MCP server, for zero isolation benefit. `slack_sdk.WebClient.chat_postMessage` **is** the standard first-party gateway, at the library layer appropriate to a Python daemon — the exact parallel to `issue_io` using the pre-authed `gh` rather than standing up a GitHub gateway. `tools/slack_io` is a `tools/*` module: it imports **no `keyring`**, writes **no files** (nothing to route through `write_artifact`), and adds **no subprocess surface** (the five sanctioned surfaces are unchanged). It introduces the **second** HTTPS-egress surface after `tools/llm/client.py` — but note this is **descriptive**, not an enforced invariant: unlike the five subprocess surfaces there is **no** egress boundary test in the repo, so there is nothing for T1 to update on that axis (an egress allowlist test would be a reasonable future hardening, out of scope here). `slack_sdk` must be imported **function-scoped inside the Slack implementation** so the no-op default never pulls it into the import graph (mirrors the `issue_io/mcp_client` function-scoped-import posture).

- **FD3 — Credential: sourced from Infisical like every other secret; runtime store is an env var (`LOOP_ENGINE_SLACK_BOT_TOKEN` + `LOOP_ENGINE_SLACK_CHANNEL`), NOT the keyring** *(locked 2026-07-16, corrected after the "does it come from Infisical like the others?" check)*. **Source of truth = Infisical** (project `loop-engine`, env `dev`), the same store that already provisions `ANTHROPIC_API_KEY` / `GITHUB_PERSONAL_ACCESS_TOKEN` / `LOOP_ENGINE_KEYRING_PASSPHRASE`; in prod it arrives via the mounted `/run/secrets/...` path like the others (Infisical is the devcontainer bootstrap; the prod credential path is unchanged). **Runtime store = the env var itself.** The token authenticates an *outbound* call, but the deciding factor for the store is that notify **runs headless** — exactly like the inbound webhook server, whose `LOOP_ENGINE_WEBHOOK_SECRET` is an env var for the same reason — and this keeps the enforced **keyring single-importer boundary** (`tools/llm/client.py` is the *only* keyring importer) intact; the token is also far lower-value than the Anthropic key (posts to a channel; cannot spend money or run code). **Seeding mechanism differs from the Anthropic/GitHub secrets and matches the webhook secret:** those two are *bounced* from the Infisical-provided env var into a separate native store (keyring / `gh`) by `seed-secrets.sh` because their runtime readers use those stores; the Slack token has **no separate native store — the env var *is* its store**, so it is **not** bounced in `seed-secrets.sh`. Instead the notify-emitting process is **launched under `infisical run`** so `LOOP_ENGINE_SLACK_BOT_TOKEN`/`LOOP_ENGINE_SLACK_CHANNEL` are inherited directly (the same way `seed-secrets.sh` itself receives its secrets) — one secret store, nothing hand-set or left in plaintext on disk. The token is a distinct, third credential class and **must never be logged** (mirror `trigger/app.py`'s "no secret is ever logged" posture). `tools/slack_io` still reads only `os.environ`; when either var is unset the notifier is the **no-op default**.

- **FD4 — Emission seam: a `Notifier` resolved and emitted *inside* `run_graph_loop`, mirroring the `(issue_filer or default_issue_filer)` pattern; fail-open with a no-op default** *(locked 2026-07-16; sharpened against the engine after the critic pass)*. `run_graph_loop` ([`core/graph_engine.py:110`](../src/loop_engine/core/graph_engine.py#L110)) is the single chokepoint **all four** call sites flow through — `runner.run_new`, `runner.run_in_tree`, `cli run --resume-from`, and `resume --from-issue` — and it **already takes `issue_filer: IssueFiler | None = None` and threads it into `build_state_graph`**, so the notifier mirrors it exactly. **Concrete contract (pin this — T1/T2/T3 all depend on it):**
  - `EventKind(StrEnum)` = `started, completed, failed_stage, budget_exceeded, awaiting_issue, crashed` (**six** — the four terminal `RunStatus` values map 1:1, plus `started` and `crashed`, which are *not* `RunStatus` values). `RunStatus.RUNNING` never maps to an event. **`failed_stage` vs `crashed` are distinct:** `failed_stage` is a *handled* terminal status the engine sets on the returned state (a persona/gate failed — [`engine.py:286`](../src/loop_engine/core/engine.py#L286) etc.), so it arrives on the success path; `crashed` is an *unhandled* exception escaping `compiled.invoke` itself (e.g. LangGraph's `GraphRecursionError`, an engine bug).
  - `LifecycleEvent` = a frozen dataclass `{kind: EventKind, state: State, budget_usd: float | None = None, error: str | None = None}`. Set **`budget_usd` on every event** (it is trivially available from `llm_client.budget_usd` at both emit points, and the `budget_exceeded`/`completed` messages want it — "spent $X of $Y"); `error` = a short exception *type/message* string, set only for `crashed`, never a traceback or secret.
  - `class Notifier(Protocol): def emit(self, event: LifecycleEvent) -> None: ...` — a single method. `NoOpNotifier.emit` is `pass`.
  - **Emission points, both inside `run_graph_loop` (not `_finalize` — that fires per-stage many times), and each wrapped in its own defensive `try/except`:** resolve `active = notifier or build_notifier_from_env()` once at the top; emit `started` right after the prime ([`graph_engine.py:121`](../src/loop_engine/core/graph_engine.py#L121)) **only when the run is fresh** — gated on a new explicit `resuming: bool` param, **not** on `start_index` (see the fresh-vs-resume note below); wrap `compiled.invoke(...)` ([`graph_engine.py:132`](../src/loop_engine/core/graph_engine.py#L132)) in `try/except` — on success emit the terminal event mapped from `result["state"].status`; on an escaping exception emit `crashed` carrying the **pre-invoke primed `state`** (`result` is unbound in the `except` — only `run_id` is meaningful; `crashed` is not a finalized/snapshotted exit path) **then re-raise** (behavior unchanged either way).
  - **Fresh-vs-resume is a caller fact, not an inferrable one (E1).** `start_index` is *derived* on both resume paths (`_start_index_for` / `reentry_index`) and **can be `0`** (a run snapshotted at stage 0, or a blast-radius re-entry to stage 0), so `start_index == 0` does **not** mean "fresh" — it would fire a spurious `started` when a human answers an escalation. `initial_state.status`/`stage_history` are equally unreliable (a per-stage snapshot is `RUNNING`). The only correct signal is caller intent: add `resuming: bool = False` to `run_graph_loop`; the **two cli resume call sites pass `resuming=True`**; `runner.run_new`/`run_in_tree` (and thus `trigger`/`flows`) keep the default `False`.
  - **Minimal entrypoint surface.** Emission lives *inside* `run_graph_loop` (default resolved there, exactly as `default_issue_filer` is resolved inside `_pause_for_issue`), so each run emits **exactly once** with no multi-layer double-emit. The **only** entrypoint edits are the two cli resume sites passing `resuming=True`; `runner`/`trigger`/`flows` are untouched. Tests inject a fake by passing `notifier=` straight to `run_graph_loop`.
  - **Fail-open is guaranteed at the call site, not just in the impl (E2).** Each emit call in `run_graph_loop` is wrapped in its own `try/except (log, swallow)`, so **any** notifier — a buggy custom one, or the fail-open test's deliberately-raising fake — can never change a run's return value, exit code, or raise; the crashed-path emit is wrapped too, so a failing notification can't mask the original engine exception. `SlackNotifier.emit` is *also* internally fail-open (swallows Slack outage/bad-token/network errors under a `timeout=5` post) for clean logging — defense in depth, but the run's safety does not depend on it.
  - **Module layout (resolves the earlier neutral-module contradiction, faithful to `issue_filer`):** the pure contract — `Notifier` Protocol, `LifecycleEvent`, `EventKind`, and the trivial `NoOpNotifier` — lives in a new `core/notify.py` with **no `slack_sdk` import**; `SlackNotifier` + `build_notifier_from_env()` + `format_event()` live in `tools/slack_io`, importing the contract from `core/notify` and keeping `slack_sdk` **function-scoped**. `graph_engine` imports the contract from `core/notify` (for the annotation + to construct events) and `build_notifier_from_env` from `tools/slack_io` — precisely how `engine.py` imports `default_issue_filer` from `tools/issue_io` today ([`engine.py:26`](../src/loop_engine/core/engine.py#L26)). No import cycle (`core/notify` is a leaf), and `slack_sdk` never enters `core/`'s import graph.

**Security Considerations:** This sprint adds a **new outbound egress** (HTTPS to Slack) and a **new credential class** (`LOOP_ENGINE_SLACK_BOT_TOKEN`). Three diligence items, all keyed to the repo's threat model: (1) **the token is never logged** and never placed in a State field, an artifact, or a message body — same posture as the webhook secret; a boundary test asserts `tools/slack_io` does not import `keyring`. (2) **Message content is model-influenced** (the escalation question text, stage names) but the sink is a *structured* `slack_sdk` API call (`channel`/`text`/`blocks` params over HTTPS), **not** a shell/`gh` arg or a filesystem path — there is no injection sink, and no subprocess surface is added (`tests/tools/test_subprocess_surfaces.py` stays green). (3) **Fail-open is itself the safety property**: notify must never become a way to fail or stall a run, so all notifier exceptions are swallowed and the post carries a short timeout. `slack_sdk` is a new **runtime** dependency ⇒ `hatch run sbom` regenerated and `hatch run audit` must pass (CLAUDE.md rule). This sprint **touches `src/`** (`core/notify.py`, `tools/slack_io`, the two emit points in `core/graph_engine.py`, and the two `resuming=True` call sites in `cli.py`) ⇒ every such PR needs a fresh-session **`architect-review`**, and the `/critic-gate` pass applies (**security-critic** for the new credential + egress surface, **architect** for the boundary posture).

**Risks & Blockers:**
- **Fail-open is the whole ballgame (FD4).** The one thing this sprint must not do is let a Slack problem affect a run. Test it explicitly: a `Notifier` that raises on every call must leave `run_graph_loop`'s returned `final_state` and the CLI exit code *bit-identical* to a no-op run. If notify can change an outcome, the design is wrong.
- **Inert-by-default must be provably inert.** With `LOOP_ENGINE_SLACK_BOT_TOKEN`/`LOOP_ENGINE_SLACK_CHANNEL` unset, `build_notifier_from_env()` returns the no-op and `slack_sdk` is never imported. Every existing test and CI run must be unchanged — assert the default path imports no `slack_sdk` and makes no network call.
- **New runtime dep ⇒ supply-chain gates.** Adding `slack_sdk` to `[project].dependencies` changes `pyproject.toml` ⇒ regenerate `sbom.json` and keep `hatch run audit` green, or CI's `sbom`/`dependency-audit` checks fail. Pin an exact version (repo convention).
- **`src/`-touching ⇒ `architect-review` applies to each PR.** Keep tasks on separate, clearly-scoped PRs; none here is docs-only except T4, so most carry the fresh-session review gate. Do not switch model mid-session to "review" — `/handoff` → new session → `/resume` → `/code-review` → post.
- **Keep `slack_sdk` out of `core/`'s import graph — but do *not* over-isolate.** `core/graph_engine` legitimately imports `build_notifier_from_env` from `tools/slack_io` (exactly as `core/engine` imports `default_issue_filer` from `tools/issue_io`); what keeps Slack out of core is the **function-scoped `slack_sdk` import** inside `SlackNotifier`, not avoiding the `tools/slack_io` import. The pure contract (`Notifier`/`LifecycleEvent`/`EventKind`/`NoOpNotifier`) lives in `core/notify.py` and imports no `slack_sdk`; assert both facts with an import test.
- **PR structure — T1 and T2 land together (E3).** `SlackNotifier.emit` (T1) *calls* `format_event` (T2), so they are one coherent unit and ship as a **single PR** (both are `tools/slack_io`, both `src/`-touching); splitting them would land a `SlackNotifier` that calls a formatter that doesn't exist yet. T3 (the two emit points in `run_graph_loop`) builds on that contract and may be its own PR or fold in; T4 (docs/boundaries) records the posture last. Each `src/`-touching PR carries the fresh-session `architect-review`.
