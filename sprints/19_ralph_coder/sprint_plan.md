### FILEPATH: /sprints/19_ralph_coder/sprint_plan.md

**Sprint Goal:** Phase 4 · part 1 of the MCP/LangGraph migration — reshape the Coder stage into a **Ralph loop** that executes the project **one task at a time from a fresh context**, driven to completion by the engine's existing revise loop and an evidence-based, coverage-aware gate. The unit of work is a **task** (not a whole sprint): the Agile Sprint Breakdown already decomposes work into sprints → tasks → acceptance-criteria, which is exactly the discrete, dependency-ordered, independently-verifiable checklist a Ralph loop needs to terminate. This sprint (a) teaches the Sprint Breakdown to emit a **structured `task_manifest`** (deterministically parsed from its own sprint markdown — no new LLM call, `sprint_plans` unchanged), and (b) makes the Coder a Ralph loop that works that manifest down one task per invocation, tracking progress as a checklist in `.agent/STATE.md`, with per-task "done" meaning *its acceptance-criteria test passes*. Everything ships behind `LOOP_ENGINE_CODER=ralph` (default `classic`); the classic per-sprint Coder and the default Sprint-Breakdown acceptance behavior stay **byte-identical** and runnable. Load-bearing invariant: **the Ralph loop terminates deterministically — it advances while making progress (changing gate findings), escalates when stuck (identical findings), and is hard-bounded by an iteration cap and the per-run USD budget — a self-looping code-writer can never spin unbounded or burn unbounded cost.**

**Dependencies:** Phase 2 (`7368411`, MCP tool provider — Ralph's `run_tool_loop` reuses `_CoderToolBackend`), Phase 3a/3b (`951e377`/`cdc7c8f`, execution isolation — a self-looping untrusted-code-writer is precisely what the worktree/container boundary exists to contain; the gate's in-process-pytest guard under `container`/`sandbox` from sprint 18 still applies). Reuses the `.agent/` semantic-state layer (`tools/agent_state`: `read_scratchpad`/`write_scratchpad`/`append_memory`, all writes via `state_io`) built in Phase 1 as the Ralph loop's task checklist + cross-iteration memory. **Read `docs/migration_roadmap.md` (Phase 4 section + decisions log) first** for the locked decisions: two node archetypes (this delivers the `AgenticNode`/Ralph one; the `GeneratorNode` is part 2, sprint 20), Ralph-Coder-first sequencing, task-level increments, and the flag-gated-parity discipline. No new third-party dependency (the manifest is deterministic parsing; Ralph reuses the existing LLM client, tool provider, and `.agent/` writers), so `sbom.json` / `hatch run audit` are untouched.

**Design decisions baked in (from the planning pass):**
- **The sprint methodology is the right input for Ralph, not a legacy artifact.** Sprints → tasks → acceptance-criteria give Ralph its checklist, its dependency ordering, and its per-unit done-signal. Keep `AgileSprintBreakdownPersona` and its prompt unchanged; *add* a deterministic manifest view over its output.
- **Increment unit = one task** (dependency-respecting), not one sprint. This makes each fresh-context `run_tool_loop` a bounded unit and makes "done" observable per task (its acceptance-criteria test), closing the "report-presence is a proxy for done" gap.
- **Ralph reuses `execute_stage`'s revise loop (Strategy A) — no new `StateGraph` topology.** `execute_stage` already does `propose → gate → (revise: feed findings back, re-run) up to max_revisions`, with identical-findings→escalate and every-exit-path-snapshots. Ralph = that loop with (1) an incremental idempotent `run()`, (2) the Coder stage's `max_revisions` raised to the Ralph iteration cap, (3) a coverage-aware gate. Both `run_loop` and the LangGraph engine drive `execute_stage`, so both get Ralph identically.
- **Progress authority = `.agent/STATE.md` task checklist; `task_manifest` = the immutable backlog; `implementation_reports` stays sprint-keyed** (a human-readable prose trail, appended as a sprint's tasks complete — unchanged shape, compatible with `CoderGate`'s helpers).
- **Default-path safety:** the manifest is *emitted always* (additive artifact) but its *validation gate* and the Ralph persona are wired **only** under `LOOP_ENGINE_CODER=ralph`; the classic Coder and the classic Sprint-Breakdown gate are untouched.

**Security Considerations:**
- **Ralph is a self-looping writer of untrusted, model-generated code — the isolation boundary is load-bearing.** Every iteration executes model code twice (the Coder's `run_tests` tool + the gate's deterministic pytest). Under `LOOP_ENGINE_ISOLATION=container|sandbox` the sprint-18 guards must still fire: the Ralph gate raises `IsolationUnavailableError` rather than run pytest in-process, and `_CoderToolBackend.resolve` refuses the in-process path. This sprint must NOT weaken either guard, and `RalphCoderGate` must inherit the guard placement verbatim (assert it). A Ralph loop that silently ran model tests in-process under container mode would multiply the sprint-18 hole by the iteration count.
- **No new subprocess / write / credential surface.** Reuses the three sanctioned subprocess surfaces (coder-tools `pytest`, `gh`, `git worktree`); adds none. All `.agent/`, report, and manifest writes go through `tools/state_io` (`append_agent_memory`/`write_agent_scratchpad`/`write_artifact`); the manifest parsing, Ralph persona, and gate import no `keyring` and open no files directly. The `.agent/MEMORY.md` append-only invariant (`append_memory`) is preserved — Ralph appends lessons, never rewrites the ledger.
- **`task_manifest` is a new Pydantic-validated I/O boundary** → per the Global Definition of Done it needs a test proving invalid input (malformed/dangling-dep/empty manifest) is rejected. The Sprint Breakdown output is model-generated; a manifest that fails to parse tasks for a sprint is a manifest-gate REVISE, not a silent empty checklist.
- **The USD budget is the ultimate governor of a runaway loop** — `LLMClient` enforces the per-run budget before every call and debits per tool-loop iteration; a non-converging Ralph run exhausts budget and terminates via the existing `BUDGET_EXCEEDED` snapshot path. Do not add an LLM call path that bypasses the client's budget check.

**Risks & Blockers:**
- **A full Ralph run cannot be end-to-end verified in this devcontainer** (no live LLM key; no container runtime for a real security boundary). Verification is **unit-level against the existing fake-LLM engine test harness**: manifest emission/validation, task selection honoring deps, the coverage gate's REVISE/ACCEPT decisions, iteration-cap termination, no-progress escalation, and `.agent/` checklist read/append. The *observed convergence behavior and cost of a real Ralph run* is explicitly **deferred to a live host run** (`sprints/DEFERRED_VERIFICATION.md`). Do not fake an LLM into "converging" and claim the loop works end to end.
- **Behavior change, not a refactor — no parity claim against the classic Coder.** Parity is asserted only for the **default (`classic`) path** (byte-identical Coder + Sprint-Breakdown acceptance) and the **cross-engine** property (classic `run_loop` and the LangGraph engine drive the Ralph stage identically).
- **The green-but-incomplete trap** — the #1 correctness risk. After Ralph finishes task 1 (tests green), a naive gate ACCEPTs with the rest of the manifest unbuilt. Mitigation: the coverage gate ACCEPTs only when *every manifest task is checked off in `.agent/STATE.md`* AND pytest is green; its negative test (green pytest + an unchecked task ⇒ REVISE) is mandatory.
- **Deterministic manifest parsing is only as robust as template adherence.** The Sprint Breakdown prompt rigidly mandates the `**Task N:**` / `Acceptance Criteria` structure, and `_parse_sprint_blocks` already relies on the same rigidity for `### FILEPATH:`. A sprint block whose tasks don't parse yields zero tasks for that sprint → the manifest gate REVISEs ("sprint X produced no parseable tasks"), surfacing the defect rather than silently under-scoping the run. Do not paper over unparseable tasks with an LLM re-ask in this sprint (that is a possible part-2 refinement).
- **Per-task test attribution is coarse in v1.** pytest runs globally, so per-task "done" is the model's claim that its acceptance-criteria test passes, verified by *global* green at the gate — not by selecting only that task's tests. Selecting per-task tests is a noted refinement (deferred), not a v1 requirement.
- **Do not regress the classic Coder or Sprint Breakdown.** Shared Coder helpers are *extracted* (not rewritten); `CoderIacPersona`, `CoderGate`, and the classic Sprint-Breakdown gate keep their observable behavior and their existing tests verbatim.

- **Task 5: The incremental task-level Ralph Coder persona (one task per invocation)**
  - **Description:** Add `RalphCoderPersona(BasePersona)` in `src/loop_engine/personas/coder_iac/ralph.py`, `consumes=("architecture_definition","sprint_plans","task_manifest")`, `produces=("implementation_reports",)`. Its `run(state, llm_client, findings)` does **exactly one task** and returns:
    1. Load `task_manifest`; read the task checklist from `.agent/STATE.md` (`agent_state.read_scratchpad`); read the latest gate status finding.
    2. **Select the next task:** the first manifest task not marked done **whose `deps` are all done** (dependency-respecting, topological). If none remain → return state unchanged (the gate will ACCEPT).
    3. One `run_tool_loop` with a **fresh context**: system_blocks = `[shared.PROMPT_TEMPLATE, architecture]` (cache-stable); user turn = the selected task (title/description/acceptance_criteria/target_files) + the parent sprint's context slice + the latest finding + a short `.agent/STATE.md` digest. The model reads files/tests via its tools and implements just that task, running its acceptance-criteria test.
    4. Apply edits via `shared.apply_file_blocks` (record edit-application failures on the report — unchanged mechanism); append the task's outcome to the parent sprint's `implementation_reports[sprint_path]` and write it via `write_artifact`; extract `## Open Questions` (unchanged escalation; a blocked task can still escalate and break the loop).
    5. Update `.agent/`: mark the task done in the `STATE.md` checklist (`write_scratchpad`) and **append** one lesson line to `MEMORY.md` (`append_memory`).
    6. Return after this single task.
    Reuse `shared.*` verbatim; keep the classic per-sprint targeted-re-entry logic OUT (subsumed — on re-entry Ralph just reads the checklist and continues).
  - **Target Files:** `src/loop_engine/personas/coder_iac/ralph.py` (new), `tests/personas/coder_iac/test_ralph.py` (new)
  - **Acceptance Criteria:** With a fake LLM client and an N-task manifest, one `run()` completes exactly **one** task (the first with satisfied deps), marks it done in `STATE.md`, and leaves the rest unchecked. A task whose deps are unmet is **not** selected before its deps. With all tasks done and no failing finding, `run()` is a no-op. A `## Open Questions` in the increment surfaces as a `State.questions` entry (`origin_stage="RalphCoderPersona"`). `read_scratchpad` is read at start and `append_memory` called exactly once per non-noop increment (assert via the `agent_state`/`state_io` seam); the append-only `MEMORY.md` invariant holds.

- **Task 6: Coverage-aware Ralph gate (green is necessary, not sufficient)**

Every REVISE returns a **single self-contained status finding**: `"Ralph status — next task: <id/title|none>; unmet-dep tasks: <...>; failing tests: <digest|none>."` so the accumulated-findings list stays authoritative-by-latest.
  - **Target Files:** `src/loop_engine/core/coder_gate.py`, `tests/core/test_coder_gate.py`
