**Sprint Goal:** Execute the **host-gated flip block** that ends the migration with **one path, not N** — the tracked teardown of the migration scaffolding described in `docs/migration_roadmap.md` "Phase 6 — Collapse the flags". Concretely: **verify** each new path end-to-end on a daemon-bearing host, then **flip the default and delete the proven-redundant classic path** for the four sunsettable flags (`LOOP_ENGINE_ENGINE=langgraph`, `LOOP_ENGINE_TOOLS=mcp`, `LOOP_ENGINE_PERSONAS=declarative`, `LOOP_ENGINE_CODER=ralph`), strip the dual-field `artifacts`/`artifact_refs` from `State` (downstream of `run_loop` deletion; bump `schema_version` 3→4 + extend `migrate_state_payload`), collapse the `loops/default/loop.py` flag branches to the single surviving wiring, and flip the issue path onto MCP while landing Sprint 26's HITL findings **R1–R7**. `LOOP_ENGINE_ISOLATION` **stays** (genuine runtime config, not scaffolding). **⚠ HOST-GATED — this sprint does NOT start on a laptop.** Every deletion is gated on its replacement being **live-verified on a daemon-bearing host** (the same host the deferred 3b/Ralph/§5–§9 checks need: real `gh` auth, a container runtime, a real Anthropic key, network). If a host session cannot complete all of V1–V3 + all deletions in one sitting, split along the task boundaries below — each task is its own independently-committable, green, reviewable change (per the phase's "Discipline" rule).

- **Task 10: Close the issue-path flip's remaining correctness defects (HITL review of PR #34)** *(gates Task 9; lands on the `sprint/27-task8-followup` branch, i.e. into PR #34 itself)*
  - **Context:** The 2026-07-12 Opus/Architect review of #34 confirmed the PR's *altitude* is right — moving the escalation destination decision into `default_issue_filer`, rather than threading `cwd` from each entrypoint, is the correct fix for R8 — but found that two of the mechanisms implementing it do not hold, plus a run-losing crash path. Findings are numbered **F1–F7** in the review comment on PR #34. **Do not delete `DEFERRED_VERIFICATION.md` (Task 9) until F1/F3/F4 land** — the issue path is not proven while they are open.

  - **F1 + F2 — the wrong-repo resume guard is a tautology; its test conceals that.** *(the important one)*
    - **The defect:** `cli.resume` compares `issue_data["url"]` against `state.pending_issue.url`, but `snapshot_path` is parsed *out of the very issue that was just read*. Both sides of the comparison originate from the same issue and agree by construction. Resuming `--from-issue 17` from the loop-engine checkout when the run's issue is really `acme/managed#17` reads loop-engine's own #17, loads *its* snapshot, matches its own URL, and **silently resumes the wrong run**. The guard only fires when `--snapshot` is passed explicitly.
    - **The reframe that determines the fix:** there is **no inconsistent state to detect**. The run resumed genuinely *is* the run paused on the issue that was genuinely read. It is simply not the run the human meant. A better derived-value check cannot read the human's mind — so the fix must remove the ambiguity from the **input** and make the snapshot authoritative whenever one is known. Do not attempt to "strengthen" the comparison.
    - **F1a — make `--snapshot` the unambiguous path.** When a snapshot is named, load it *first*, then take **both** the repo and the issue number from `state.pending_issue.url` — recorded at pause time by the process that actually filed the issue, and the only first-hand record of the destination. CWD never enters. Add `repo_from_issue_url(url) -> str` ("owner/repo" from a canonical issue URL) to `tools/issue_io/github.py`. `IssueRef` already carries `number` + `url`, so **no `State` schema bump is needed** — parse, don't store.
    - **F1b — `--from-issue N` alone must resolve its repo explicitly and say so.** Never leave the destination to `gh`'s implicit CWD resolution: `read_repo = repo or resolve_repo_slug(Path.cwd())`, then `typer.echo(f"Reading issue #{from_issue} from {read_repo}")` before any expensive work. **The echo is the actual defense** — a human resuming a `flows/maintenance` run from the wrong checkout sees `Reading issue #17 from glunk-works/loop-engine` and stops.
    - **F1c — keep the URL comparison but relabel it.** It is a snapshot↔issue **integrity** check (did the link survive?), not a wrong-repo defense. Correct the comment, which currently overclaims.
    - **F2 — rewrite `tests/test_cli.py::test_cli_resume_rejects_a_same_numbered_issue_from_the_wrong_repo`.** It currently passes only because its stub pairs a wrong-repo issue URL with the **right** repo's snapshot path — a combination that cannot occur (loop-engine's #17 carries loop-engine's own `Snapshot:` line). Reconstruct the scenario faithfully, assert what actually happens, and cover both new paths (snapshot-authoritative; explicit-repo echo). The test must document the residual ambiguity rather than pretend it is closed.

  - **F3 + F6 — `_ORIGIN_CWD` is a process-global and the trigger surface runs loops concurrently.**
    - **The defect:** `InProcessDispatcher.dispatch` dedupes only on `(repo_full_name, issue_number)`, so two different issues dispatch concurrent `asyncio.to_thread(runner.run_new)` calls into one process. Run A sets `_ORIGIN_CWD` and chdirs; run B then captures `origin = Path.cwd()` — *A's worktree*; A's `finally` resets the global to `None` while B is still live. B's next escalation resolves to loop-engine — **the exact R8 leak returns, in the one surface built for many runs**. `runner.run_in_tree` (`flows/maintenance`) is hit harder still: it deliberately never opens `worktree_run` and *depends* on the global being unset so the fallback yields the clone.
    - **F3 — replace the module global with a `contextvars.ContextVar`.** `asyncio.to_thread` copies context per worker thread, so writes stop crossing between runs. Use `token = _ORIGIN_CWD.set(origin)` / `finally: _ORIGIN_CWD.reset(token)` — `.reset()` restores the prior value instead of clobbering to `None`, which **closes F6 (non-re-entrancy) in the same change**.
    - **Apply the identical change to `state_io`'s `_STATE_ROOT`**, which has the same bug for the same reason (six lines, same shape).
    - **This does NOT make concurrent runs safe, and the code must not imply it does.** `os.chdir` remains process-global — concurrent `worktree_run`s still corrupt each other's CWD, breaking snapshot paths and `run_in_tree`'s fallback. **So also serialize:** give `InProcessDispatcher` a lock so one loop runs at a time. It costs throughput the factory is not currently using and makes the `chdir` race *unreachable* rather than merely unlikely. Ship an honest bottleneck over a race that files issues on the wrong repo.
    - **Backlog (not this task):** the real fix is to stop using process CWD as an isolation mechanism and thread the tree path explicitly. That is its own sprint — file it as **BL-8**.

  - **F4 — a failed escalation currently destroys the run.** `core/engine.py::_pause_for_issue` calls `issue_filer` **before** `_finalize` persists the snapshot, so any raise inside the filer discards the run's entire work with **no `AWAITING_ISSUE` snapshot written**. #34 makes this reachable: `default_issue_filer` now unconditionally calls `resolve_repo_slug`, which shells `gh repo view` with `check=True` — an orchestrator launched from a non-repo CWD (a systemd unit with `WorkingDirectory=/`, a container entrypoint) raises `CalledProcessError` from inside the pause path.
    - **Persist before filing.** Reorder `_pause_for_issue` so the snapshot is written first and updated with the `IssueRef` after, so a failed escalation leaves a **resumable** run. This is the load-bearing fix.
    - **Make the failure legible.** Catch `CalledProcessError` in `default_issue_filer` and raise a typed error naming the cause (the origin CWD is not a GitHub repo, so the destination cannot be named). **Do NOT fall back to `repo=None`** — that silently restores the R8 leak this whole line of work exists to prevent.

  - **F5 — the URL guard soft-fails.** `if issue_url and ...` means a *missing* `url` skips the check. `read_issue` now always requests `url`, so an absent one signals something already went wrong — precisely when skipping is worst. Raise instead.

  - **F7 — import-time coupling.** `tools/issue_io/mcp_client.py`'s module-scope imports of `tools/mcp`, `tools/repo_io`, `tools/worktree` mean importing `core/engine` transitively drags in the whole MCP client stack. Move them inside `default_issue_filer`/`default_issue_reader` (only needed at call time). This restores CLAUDE.md's claim that exactly **one** `core/` module reaches `tools/mcp`, so **revert that CLAUDE.md bullet** to its pre-#34 wording once done.
