### FILEPATH: /sprints/34_ci_supply_chain_hardening/sprint_plan.md

**Sprint Goal:** Close the three items **BL-11** left open deliberately. They are one theme, not three chores: sprint 33 discovered that every "required" check in this repo was enforcing nothing, and switched enforcement on by creating the `protected-integration-branches` ruleset. This sprint makes that enforcement **hold** — against a hijacked third-party action (SHA-pin the four actions `ci.yml` consumes), against a merge strategy that silently discards the enforced PR title (restrict to squash-only), and against the ruleset itself being weakened or deleted with nobody noticing (drift detection). The through-line: BL-11 went unseen for the entire life of the CI config because **nothing in the repo asserted the control existed.** Every task here is a control plus the assertion that it is still there.

**Out of scope:** **Any change under `src/`.** Deliberate and load-bearing, exactly as in sprint 33: `hitl-review.yml` filters on `^src/`, so a CI-and-docs-only PR is exempt from the architect-review gate and can land in one pass. The two unfixed findings from PR #39's review (`publish_artifacts` reads every artifact off disk on every stage while both docstrings claim it *"does no I/O"*; that read-back uses `Path.read_text()` with no explicit encoding) are **real and still open** — they stay in the backlog. Also out of scope: the five standing checks in `sprints/DEFERRED_VERIFICATION.md`; any dependency change (so **no `sbom` regeneration**); any change to the eight checks themselves or to `pr-title.yml` / `hitl-review.yml`; and any *widening* of the PAT's scopes (FD1 makes it unnecessary).

**Context (locked decisions — do not re-open):**

- **FD1 — The drift check needs NO new PAT scope. Verified live, not assumed** *(locked 2026-07-13, Opus/Architect, sprint 34 planning pass; user-confirmed)*. BL-11's resolution note proposed granting `Administration: read` so a test could see the ruleset. **That is unnecessary.** With the temporary `Administration: write` grant already revoked, `GET /repos/glunk-works/loop-engine/rules/branches/main` — the **effective rules** endpoint — returns all four rule types (`deletion`, `non_fast_forward`, `pull_request`, `required_status_checks`) **and the eight required check contexts by name**. This is the same endpoint whose empty `[]` was BL-11's proof of *absence*; it reads back just as well as proof of *presence*. The admin-only `/rulesets` endpoint adds only an id and enforcement string. So the check is **read-only by construction** and the principle survives without anyone having to promise it: *a gate the governed party can remove is not a gate* — write access stays human-only, and no scope is widened to build the watchdog.
- **FD2 — A drift check that is a REQUIRED PR CHECK is circular and must not be built that way** *(locked 2026-07-13, Opus/Architect; user-confirmed)*. The obvious design — a 9th job in `ci.yml`, added to the required set — defeats itself on the exact failure it exists to catch. **The ruleset is what makes a check required.** Delete the ruleset and the drift check is no longer required either: it vanishes *with* the thing it was watching. It would still go red, but a red non-required check blocks nothing, so it cannot stop a merge precisely when it matters most. It would block *partial* weakening (one check dropped while the ruleset survives) — but total deletion, the actual BL-11 scenario, is the case it structurally cannot hold. Therefore: **a scheduled workflow** (no dependency on the ruleset, runs whether or not a PR exists) **plus a `/resume` preflight** (catches it at the natural moment — session start). **Do not add a drift job to `ci.yml` and do not propose making it a 9th required check.**
- **FD3 — A scheduled workflow only ever fires from the DEFAULT branch, so the drift workflow ships in its own PR based on `main`** *(locked 2026-07-13, Opus/Architect; user-confirmed)*. GitHub runs `on: schedule:` **only** from the repository's default branch, which here is **`main`**. Every other PR in this repo is based on `feat/mcp-langgraph-migration`. A cron workflow merged into `feat/**` would sit in the tree — correct, tested, reviewed — and **never execute**, possibly for weeks, until `feat/**` eventually reaches `main`. That is BL-11's failure mode (*a control that looks present and is inert*) reproduced by the very fix meant to detect it, which is why the convention is broken **once, on purpose**: Task 4 lands via a **separate, narrow PR whose base is `main`**, carrying only the drift workflow. Nothing is bypassed by this — `main` is covered by the same ruleset, so that PR still faces all eight required checks. **This is the one and only exception; every other task in this sprint uses the normal `feat/mcp-langgraph-migration` base.**
- **FD4 — SHA pinning trades tag-hijack risk for SILENT STALENESS, so Dependabot is part of the pin, not a nicety** *(locked 2026-07-13, Opus/Architect; user-confirmed)*. A pinned SHA cannot be moved under you — and it also cannot receive a security patch. Pin `actions/checkout` at a vulnerable commit and `@v4` consumers get the fix automatically while this repo stays frozen on the flaw, forever, with nothing in the tree saying so. Both failure modes are silent; Dependabot makes the second one loud (a PR per action, with the version comment updated). **`.github/dependabot.yml` already exists** (pip, daily) and is pinned by `tests/test_ci_config.py::test_dependabot_config_defines_daily_pip_updates` — this is an **extend**, not a create. Shipping the pins *without* the Dependabot entry is the same species of defect as BL-11 and is not an acceptable partial landing.
- **FD5 — The two settings toggles are human-only and must be applied AFTER the code lands. The ordering is load-bearing** *(locked 2026-07-13, Opus/Architect)*. `sha_pinning_required: true` makes GitHub **reject any workflow using a floating tag**. Enabling it before Task 1's SHAs are merged fails every workflow on every PR — including the PR carrying the fix, which would then be unmergeable. Sequence is: merge the pins → *then* flip `sha_pinning_required`. Squash-only (`allow_merge_commit: false`, `allow_rebase_merge: false`) has no such ordering constraint but is equally human-only. **Claude cannot set either** and must not pretend to have verified them; they go in the sprint's human-actions list and are confirmed in the UI.
- **FD6 — `GITHUB_TOKEN` is a DIFFERENT PRINCIPAL from the PAT; its read access must be verified live, not inferred** *(locked 2026-07-13, Opus/Architect)*. FD1 was established with the local PAT. The drift workflow runs as Actions' `GITHUB_TOKEN`, which has its own permission model (this repo sets `default_workflow_permissions: read`). It is *likely* sufficient for the effective-rules endpoint, but **assuming it is, is exactly the reasoning error BL-11 punished** — the whole sprint exists because a control was believed to be in force and never checked. Task 4 therefore has an explicit **live-observation** acceptance criterion: the workflow must be seen actually running and actually reading the eight check names on a real run. If `GITHUB_TOKEN` turns out to be 403, the fallback is `permissions: { contents: read }` plus, only if still refused, a repo secret holding a read-only PAT — **escalate to the human rather than widening any scope unilaterally.**

**Security Considerations:** This sprint is *entirely* supply-chain and enforcement hardening; that is the point rather than a side effect. Today `ci.yml` consumes four actions at floating tags — `actions/checkout@v4`, `actions/setup-python@v5`, `actions/upload-artifact@v4`, and `gitleaks/gitleaks-action@v2`. A tag is a **mutable pointer**: whoever controls the upstream repo (or anyone who compromises it) can repoint `v4` at arbitrary code, which then runs in this repo's CI with whatever the workflow's token grants. `gitleaks/gitleaks-action` deserves specific attention as **the only third-party (non-`actions/`) action here** — and the irony is sharp: it is the job that reads the entire repository looking for secrets. It is the highest-value hijack target in the file and the one most worth pinning. Note that `default_workflow_permissions: read` and `can_approve_pull_request_reviews: false` (both already verified correct in BL-11) meaningfully cap the blast radius of such a hijack — this sprint narrows the entry point rather than being the only thing standing in the way. No change to `src/`, to any runtime dependency, to any credential path, or to any of the five sanctioned subprocess surfaces. **No scope is widened anywhere** (FD1).

**Risks & Blockers:**
- **Do NOT flip `sha_pinning_required` before the pins merge** (FD5). It rejects floating tags repo-wide and would make the fixing PR itself unmergeable. Merge first, toggle second.
- **`uses:` requires a COMMIT SHA, not a tag-object SHA.** For an annotated tag, `gh api repos/OWNER/REPO/git/ref/tags/vX --jq .object.sha` returns the **tag object**, which GitHub will not accept in `uses:`. Resolve the commit directly: `gh api repos/actions/checkout/commits/v4.2.2 --jq .sha`. Pinning the wrong SHA fails every workflow at once; this will otherwise cost a confusing debug cycle.
- **Pin to the SHA behind a specific released version, and record that version in a trailing comment** (`uses: actions/checkout@<sha>  # v4.2.2`). The comment is what makes the pin auditable and is what Dependabot rewrites. A bare SHA with no comment is unreviewable.
- **The drift workflow is inert until it reaches `main`** (FD3). Do not "verify" it by merging it to `feat/**` and observing nothing — observing nothing is the failure. It must be seen running, on `main`, on a real schedule or a manual `workflow_dispatch`.
- **Give the drift workflow a `workflow_dispatch` trigger.** A cron-only workflow cannot be tested without waiting for the cron. `workflow_dispatch` is what makes FD6's live-observation criterion achievable in the same session it ships.
- **The drift workflow must NOT live in `ci.yml`** — it would join the cancelling concurrency group and, worse, invite the circular-required-check design FD2 rejects. New file, own triggers, gates nothing, gated by nothing (the same posture `pr-title.yml` and `hitl-review.yml` already hold, and which `tests/test_ci_config.py:164` already asserts for `hitl-review`).
- **The existing dependabot test is substring-based** (`'package-ecosystem: "pip"' in text`), which is the same weak-assertion anti-pattern sprint 33's FD4 called out. Task 2 upgrades it to a structural `yaml.safe_load` check while adding the `github-actions` assertion. Do not simply append a second `in text` line.
- **The `on:` → `True` YAML trap still applies.** `on` is a YAML 1.1 truthy, so `yaml.safe_load` yields `{True: {...}}` and `cfg["on"]` raises `KeyError`. `tests/test_ci_config.py` already has `_load_workflow`/`_trigger_types` helpers that handle this — **reuse them**, do not re-derive.
- **Dependabot will open PRs against the default branch (`main`) by default**, not `feat/**`. That is acceptable and arguably correct for actions updates, but it is a behavior change worth expecting rather than being surprised by. Do not add a `target-branch:` override without raising it — it interacts with the two-base convention.
