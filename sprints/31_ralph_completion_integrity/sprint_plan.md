### FILEPATH: /sprints/31_ralph_completion_integrity/sprint_plan.md

**Sprint Goal:** Close **F-RALPH-FALSE-COMPLETION** (recorded in `sprints/DEFERRED_VERIFICATION.md` from the V2 re-attempt #7 host run, 2026-07-11) so a fresh V2 re-attempt can reach terminal `COMPLETED`. `RalphCoderPersona` currently marks a manifest task **completed** in `.agent/STATE.md` based solely on whether the model raised an `## Open Questions` section — it never checks whether `apply_file_blocks` actually applied that task's edit. Because a task already in `completed_tasks` is never re-selected (`select_next_task`), and its report section is never revisited (`_upsert_task_section` only replaces the section for the task currently being run), a single malformed or never-attempted edit block on a "completed" task leaves a permanently unresolvable `## Edit Application Failures` marker baked into the sprint's concatenated report — which `CoderGate`/`RalphCoderGate`'s `edit_findings` check short-circuits on **before ever running pytest**, wedging the run until the revise cap exhausts and it escalates to a human with a misleadingly narrow message. This is a **correctness bug in the engine's own bookkeeping**, not a model-prompting problem (unlike sprint 30's F-RALPH-OVERSPEC-TEST) — the model has no way to know or control that a parse failure on its output becomes unrecoverable.

**Out of scope:** any change to `core/coder_gate.py` (see FD1 below — the existing `edit_findings`/coverage/regression machinery is sound once task completion is honest; no gate-level rescoping is needed). No change to the classic `CoderIacPersona` (see FD2 — it does not share this bug and is slated for deletion in sprint 27 Task 4). No engine, `State`, or manifest-schema change. No new subprocess/credential/network surface. No dependency change. The actual host `COMPLETED` observation is **not** a task here — it remains sprint 27's V2 obligation (this sprint is "resolved-in-code"; the host proof is owed to a future V2 re-attempt, exactly as sprints 28/29/30 were structured).

**Context (locked decisions — do not re-open):**

- **FD1 — Fix lives entirely in `personas/coder_iac/ralph.py`'s completion bookkeeping; `core/coder_gate.py` is untouched** *(locked 2026-07-11, Opus/Architect, F-RALPH-FALSE-COMPLETION planning)*. The permanent-taint symptom (a stale `## Edit Application Failures` marker blocking pytest forever) is a **consequence** of the completion bug, not an independent gate defect: once a task with a failed edit is correctly kept off `completed_tasks`, `select_next_task` re-selects it on the next increment, the existing REVISE-finding plumbing (`_split_findings`/`latest_status`) already carries the "Edit Application Failures" finding back into that task's next fresh-context prompt, and — on success — `_upsert_task_section` replaces *that task's own section*, naturally clearing the marker. If a task's edit never succeeds, the engine's existing identical-findings exhaustion guard (`_exhaustion_escalation`, `core/engine.py`) still bounds it to an honest escalation, now correctly describing an unresolved task rather than falsely claiming completion. A more invasive fix (narrowing `edit_findings`'s scan from the whole concatenated report to just the active/latest task's section) was considered and is **not needed**: it only matters if a *completed* task can still carry a stale marker, which this sprint's fix prevents at the source. Do not add that narrowing speculatively — if a future V2 re-attempt shows a residual staleness gap, that is new evidence for a follow-on, not something to pre-empt here.
- **FD2 — Classic `CoderIacPersona` is out of scope.** It replaces the whole sprint report every revise cycle (`reports[sprint_path] = report`, `persona.py:143`) rather than appending per task, so a corrected report on the next attempt already drops any prior `## Edit Application Failures` marker — it does not share this bug. It is also slated for deletion in sprint 27 Task 4 once `CODER=ralph` flips default. No change to `persona.py` in this sprint.
- **FD3 — "Edit failed, will retry" is a distinct bookkeeping state from "open question raised," and must not be conflated.** A task whose edit failed to apply (no model-raised question) must be kept off `completed_tasks` so it is retried, but it must **not** be added to `scratch.blocked_items` and must **not** synthesize a fake open question — `blocked_items`/`state.questions` are reserved for genuine model-raised escalations that route through the resolver ladder. An edit-application failure is a mechanical retry, not an escalation; conflating the two would incorrectly surface "Edit Application Failures" as a human-escalation-worthy question in its own right (defeating the entire fix — the whole point is that this should self-resolve via retry, not go to a human).

**Security Considerations:** This is a bookkeeping-logic change to when a task is recorded "done" — no new file-write path (still exclusively through `write_artifact`/`apply_file_blocks`, unchanged), no new subprocess/network/credential surface, no dependency change. Sanctioned subprocess surfaces stay **exactly five**; `tools/llm/client.py` stays the only `keyring` importer; `State`/`schema_version`/`extra="forbid"` untouched. The fix makes the Ralph loop's evidence (`.agent/STATE.md` checklist) a **more faithful** signal of real progress, not less — it closes a soundness gap in the `RalphCoderGate` ACCEPT precondition ("every manifest task checked off AND green pytest"), it does not weaken it.

**Risks & Blockers:**
- **`_finalize_report`'s signature change is a shared-shape edit.** It is called from both `_task_increment` and `_repair` (`ralph.py`). Both call sites must be updated for the new `(report, failures)` return; `_repair` never marks a task done today, so its behavior is otherwise unchanged — just unpack the tuple there.
- **Don't conflate "edit failed" with "escalated" in the scratchpad memory log** (FD3) — the memory entry text and any downstream tooling that greps `MemoryEntry` bodies for "escalated"/"Blocked task" should not be misled into treating a retry-pending task as a genuine human-escalation event.
- **Real proof is the host V2 re-attempt, same as sprint 30.** The unit tests pin the completion-bookkeeping contract in isolation (mocked LLM responses); do not claim F-RALPH-FALSE-COMPLETION *verified* (vs. *resolved-in-code*) until a fresh host run reaches terminal `COMPLETED`.
- **Don't over-fit to the textkit case.** The fix must be general (any task, any file, any malformed/missing edit block), not keyed to `slugify`/`word_count`/`textkit` — those are the illustrative instance from re-attempt #7, not the spec.
