### FILEPATH: /sprints/22b_native_github_server/sprint_plan.md

**Sprint Goal:** Build the system's **second MCP server** — the native `github_server` — the foundation slice of **Phase 5 proper** (pieces 2–4, the trigger surface + maintenance/bootstrap flows, all stack on it). Ship it as a **native re-front** mirroring `coder_tools_server`: a thin `FastMCP` server exposing exactly the four factory verbs `{create_repository, clone_repo, create_branch, open_pr}`, each delegating to a new GitHub-owning `tools/repo_io` module (a sibling to `issue_io`). Wire it into the existing `loop_engine.mcp.json` discovery (22a) via a **committed** repo-root config stanza, and expose it to the orchestrator through a **consumer-scoped** `build_github_provider()` helper — never to the model's coder tool loop. This is a **capability slice**: it lands the server + delegate + config + provider helper + hermetic tests + docs. It does **not** wire a production flow that chains the verbs (that is Sprint 23's maintenance/bootstrap flows), and it adds **no new subprocess surface** (all four verbs ride the existing `gh` executable — see the locked decision below).

**Dependencies:** Sprint 22a (`457f675`→`d0e118d`) — the `loop_engine.mcp.json` multi-server discovery + consumer-scoping substrate this sprint builds on: `load_mcp_config` (`tools/mcp/config.py`), `build_provider_for(names, *, cwd)` and the `MCPToolProvider` routing/teardown (`tools/mcp/provider.py`), and the consumer-scope guard pattern (`test_mcp_provider.py::test_extra_config_server_never_reaches_coder_provider`). This sprint adds a second entry to that registry and a second consumer. Precedents to mirror exactly: `mcp_servers/coder_tools_server.py` (the re-front shape), `tools/issue_io/github.py` (the `gh`-shelling delegate + its `_run_gh` fixed-argv wrapper and `unittest.mock.patch("...._run_gh")` test style). **Read the "Context (locked decisions)" section before starting** — the Architect (Opus) settled the two gating decisions in the 22b planning pass; do not re-open them.

**Context (locked decisions — do not re-open):**
- **gh-only; the local-git subprocess surface is DEFERRED to Sprint 23** *(locked 2026-07-09, Opus/Architect, 22b planning pass)*. All four factory verbs are implemented through the **existing `gh` executable** — `create_repository` → `gh repo create`, `clone_repo` → `gh repo clone`, `create_branch` → `gh api …/git/refs` (a remote ref), `open_pr` → `gh pr create`. This means `tools/repo_io` becomes a **second consumer of the already-sanctioned `gh` subprocess surface** (alongside `issue_io`) and adds **NO fourth subprocess surface**. The "exactly three sanctioned subprocess surfaces" invariant holds; only its `gh` clause widens from "`issue_io`'s `gh`" to "`issue_io`'s and `repo_io`'s `gh`". The genuine local-git surface (commit/`git push` inside a cloned tree) is **not** needed by these four verbs and is deferred to Sprint 23's maintenance flow, where it is flow-forced and ideally live-verifiable on a daemon-bearing host. **Do NOT** add a `git clone`/`git`-subprocess call to `repo_io`, and do **NOT** fold anything into `tools/worktree` (whose surface stays narrowly "args derive only from a `validate_run_id`-checked run_id" against loop-engine's own repo).
- **Deliverable boundary = capability slice + committed config** *(locked 2026-07-09, Opus/Architect)*. 22b ships the server, the `tools/repo_io` delegate (all four verbs), a **committed** repo-root `loop_engine.mcp.json` github stanza (the first real instance of that file), a consumer-scoped `build_github_provider()` helper, hermetic tests, and docs. It does **NOT** ship a production flow caller — no CLI subcommand, no loop wiring — because no flow chains these verbs until Sprint 23. The "orchestrator-invoked, no LLM loop" property is proven at the provider API level (a helper distinct from `build_coder_tool_provider`, guarded by a scope test), not by an end-to-end production run.
- **Native re-front, not the official GitHub MCP server** *(Phase 5 planning, locked)*. `github_server` mirrors `coder_tools_server`: it delegates to `tools/repo_io` (never calls `gh` from the server module itself), so the "only the GitHub-owning `tools/` module talks to GitHub" boundary holds — now widened to `{issue_io, repo_io}`. Native lets us assert *exactly* `{create_repository, clone_repo, create_branch, open_pr}` by static test; the official ~60-tool server (incl. `merge_pull_request`, contradicting "auto-merge prohibited") is uncontrolled and un-fakeable offline.
- **Credential = `gh`'s own auth** *(Phase 5 planning, locked)*. No token enters this process; `repo_io` shells out to an already-authenticated `gh` exactly as `issue_io` does. `tools/repo_io` imports no `keyring` — the `tools/llm/client.py`-only-imports-keyring invariant is untouched. Note: `github_server` is nonetheless the system's **first credentialed MCP server** (unlike credential-free `coder_tools_server`) — its launch profile is un-sandboxed, network-enabled (the static-spec side of 22a's "heterogeneous launch profiles" decision).
- **Consumer split is a hard invariant (cross-cutting #2).** The github verbs are **orchestrator-invoked and must never enter the model's coder tool loop.** `build_github_provider()` is a **separate** consumer-scoped helper; the coder loop keeps naming `coder_tools` alone. 22a's scope guard already prevents leakage pre-emptively; 22b adds the mirror-direction guard (Task 4).
- **`issue_io` stays on its current direct path, untouched.** Its escalation verbs (create/read/comment issues) do **not** move onto MCP in this sprint — that unification is Phase 6. Do not touch `issue_io` or `resume --from-issue`.

**Security Considerations:** **No new subprocess surface** — the four verbs ride the existing `gh` executable (fixed argv, `shell=False`), so `repo_io` is a second `gh` consumer, not a new surface; the sanctioned-surfaces count stays **three**. `repo_io`'s `_run_gh` wrapper MUST mirror `issue_io`'s: `["gh", *args]`, `shell=False`, `capture_output=True`, `timeout`, with the `S603`/`S607` noqas and their justifications. `clone_repo`'s destination path is the one caller-influenced filesystem target — validate it (reject traversal / symlink escape, reusing the `state_io`/`coder_tools` traversal-validation approach) so a malicious `dest` cannot land a clone outside the intended tree; the repo slug/URL argument is a fixed-position `gh` arg, never interpolated into a shell string. `github_server` holds no credentials and must import side-effect-free (no `gh` call at import time — discovery/`list_tools` must be hermetic). `tools/repo_io` imports no `keyring` and writes no files (clone I/O is `gh`'s, not a `state_io`-guarded `open`/`write_text`). The net-new security property to encode is the **mirror-direction consumer-scope guard** (Task 4): github's verbs must be provably unreachable from the model's coder provider, and the coder tools provably unreachable from the github provider.

**Risks & Blockers:**
- **First committed `loop_engine.mcp.json` must not perturb the coder path.** Committing a repo-root config with a `github` stanza makes `load_mcp_config()` (no-arg) return `{coder_tools (default), github}` for every caller. This is safe **only because** consumption is consumer-scoped: `build_coder_tool_provider` names `coder_tools` alone. Verified pre-write: every `test_mcp_config.py` case uses a `tmp_path` override (none reads the real repo root), and the sole no-arg `load_mcp_config()` is inside consumer-scoped `build_provider_for`. Do not add a test that asserts the *real* repo-root config equals coder-tools-only — that would now (correctly) fail.
- **github discovery tests must stay hermetic.** With the file committed, any test that builds a provider naming `github` will launch `python -m loop_engine.mcp_servers.github_server`. `list_tools` does not execute a tool, so discovery is offline — but only if the server module imports without calling `gh`. Keep the module import side-effect-free; assert the exact four-tool set via discovery, never by executing a verb.
- **No live network/`gh`-auth verification on this branch.** This is the system's first credentialed, network-enabled server launch, and no host here has `gh` auth or network. Hermetic coverage only: delegate argv/parse via mocked `_run_gh` (in-process); server tool-set via real-server discovery (offline `list_tools`); orchestrator path + scoping via `build_github_provider()` + the scope guards. Append the live end-to-end check (real `github_server` launch → each verb against a scratch repo, with `gh` auth + network) to `sprints/DEFERRED_VERIFICATION.md`.
- **Do not let github verbs leak into the model loop.** The bidirectional scope guard (Task 4) is load-bearing: adding the `github` stanza must not change `build_coder_tool_provider`'s tool set, and `build_github_provider()` must expose the four github verbs and nothing coder.
- **No new dependency.** `gh` is external; `FastMCP` is already pinned. No `pyproject.toml`/SBOM change — state this explicitly in the PR.

**Tasks:**

- **Task 1: `tools/repo_io` GitHub-owning delegate (four factory verbs, gh-only)**
  - **Description:** Add `tools/repo_io/` (sibling to `issue_io`): a `github.py` with a private `_run_gh(args: list[str]) -> str` wrapper identical in shape to `issue_io`'s (fixed `["gh", *args]`, `shell=False`, `capture_output=True`, `text=True`, `timeout=60`, with the `S603`/`S607` noqas + justifications), plus the four verbs, each shelling to `gh` and parsing its output:
    - `create_repository(name, *, org=None, private=True) -> RepoRef` → `gh repo create <name-or-org/name> [--private|--public] [--json ...]`; parse the created repo's URL/slug.
    - `clone_repo(slug, dest, *, depth=None) -> str` → validate `dest` (reject traversal/symlink escape via the `state_io`/`coder_tools` validation approach), then `gh repo clone <slug> <dest> [-- --depth N]`; return the clone path.
    - `create_branch(owner, repo, branch, *, base=None) -> str` → resolve the base SHA (`gh api repos/{owner}/{repo}/git/ref/heads/{base_or_default}`), then create the remote ref (`gh api --method POST repos/{owner}/{repo}/git/refs -f ref=refs/heads/<branch> -f sha=<sha>`); return the ref.
    - `open_pr(owner, repo, *, head, base, title, body) -> PullRef` → `gh pr create --repo <owner>/<repo> --head <head> --base <base> --title ... --body ...`; parse the PR URL/number. (Keep the "auto-merge prohibited" invariant — no merge verb.)
    Define small typed return values (Pydantic or dataclass, matching how `issue_io` returns `IssueRef`) as needed. `repo_io` imports no `keyring` and writes no files.
  - **Target Files:** `src/loop_engine/tools/repo_io/__init__.py` (new, exports the four verbs + return types), `src/loop_engine/tools/repo_io/github.py` (new), `tests/tools/test_repo_io.py` (new)
  - **Acceptance Criteria:** Each verb, with `_run_gh` patched (`unittest.mock.patch`), builds the expected `gh` argv (fixed-position args, no shell string) and parses the mocked stdout into the right return value. `clone_repo` rejects a traversal/symlink-escaping `dest` before any `gh` call. `repo_io` imports no `keyring` and performs no file writes (boundary tests green). Lint clean (noqas justified as in `issue_io`).

- **Task 2: `mcp_servers/github_server` native re-front (exactly four verbs)**
  - **Description:** Add `mcp_servers/github_server.py` mirroring `coder_tools_server.py`: `FastMCP("loop-engine-github")` with four `@mcp.tool()` functions (`create_repository`, `clone_repo`, `create_branch`, `open_pr`) that delegate to `tools/repo_io`, plus `main()`/`__main__` running `mcp.run()`. Each tool's signature/docstring describes a factory verb an **orchestrator** invokes (not a model). The module MUST import side-effect-free — no `gh` call at import (discovery must be offline). Run as `python -m loop_engine.mcp_servers.github_server`.
  - **Target Files:** `src/loop_engine/mcp_servers/github_server.py` (new), `tests/tools/test_github_server.py` (new)
  - **Acceptance Criteria:** Launching the server through a real `MCPToolProvider` and calling `list_tools` (offline — no `gh`) discovers **exactly** `{create_repository, clone_repo, create_branch, open_pr}` and nothing else (static assertion mirroring the coder-tools exact-set test). The module imports with no network/`gh` access. Lint clean.

- **Task 3: Committed `loop_engine.mcp.json` github stanza + `build_github_provider()` orchestrator helper**
  - **Description:** Commit a repo-root `loop_engine.mcp.json` (the first real instance) declaring the `github` server as a **static launch spec**: `{"servers": {"github": {"command": "python", "args": ["-m", "loop_engine.mcp_servers.github_server"], "cwd": null}}}` (un-sandboxed, network-enabled — the trusted-first-party profile from 22a's heterogeneous-launch decision; document that `command` resolves via PATH to the active-env interpreter on the factory host). Add `build_github_provider() -> MCPToolProvider` to `tools/mcp/provider.py` as a thin, consumer-scoped wrapper: `build_provider_for(["github"])` (no `cwd` sandbox override — github is not the coder-tools special case). Export it from `tools/mcp/__init__.py`.
  - **Target Files:** `loop_engine.mcp.json` (new, committed at repo root), `src/loop_engine/tools/mcp/provider.py`, `src/loop_engine/tools/mcp/__init__.py`, `tests/tools/test_mcp_provider.py` (or `test_mcp_multiserver.py`)
  - **Acceptance Criteria:** `load_mcp_config()` (no-arg, reading the committed file) returns both `coder_tools` (built-in default) and `github` (from file). `build_github_provider()` launches the github server and its `tools` are exactly the four github verbs. The committed file does not change `build_coder_tool_provider`'s behavior (its tool set is unchanged — asserted). Full existing `test_mcp_config.py`/`test_mcp_provider.py` suite stays green. Lint clean.

- **Task 4: Bidirectional consumer-scope guard (github ⟂ model coder loop) — security invariant**
