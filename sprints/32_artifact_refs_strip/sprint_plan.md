### FILEPATH: /sprints/32_artifact_refs_strip/sprint_plan.md

**Sprint Goal:** Resolve the deferred artifact-storage cleanup (roadmap decision **FD3**) by **inverting it**. FD3 deferred "strip `State.artifacts`, read through `artifact_refs`" on the correct grounds that it was a behavior-changing refactor rather than a deletion. The planning pass for this sprint establishes something further: the refactor is not merely risky, it is **backwards**. The field with no readers is not `artifacts` — it is **`artifact_refs`**, a write-only mirror whose read path (`get_artifact`) has **zero production callers**, and nothing in `src/` ever reads `docs/artifacts/` back. Meanwhile `State.artifacts` has ~14 read sites across 8 persona/gate/service modules, and those bodies are fed **directly into the prompt-cache prefix**. This sprint therefore deletes `artifact_refs` / `ArtifactRef` / `artifact_digest` / `get_artifact`, keeps `State.artifacts` as the single source of truth, and renames `mirror_to_disk` to what it has actually always been: **publication** of artifact bodies into the working tree. `schema_version` goes 3 → 4 and `migrate_state_payload` drops the retired key.

**Out of scope:** **Any change to the 8 artifact-reading modules** (`personas/{pm/persona,pm/critic_gate,architecture/persona,agile_sprint_breakdown/manifest,coder_iac/ralph,declarative/node,declarative/services}.py`, `core/{gates,coder_gate}.py`). Leaving them untouched is the *point* of this direction, not an omission — they are the hot path V1/V2 verified green, and they feed the cache prefix. Also out of scope: **BL-3** (the prompt-caching review). Token efficiency is a real goal but a **different** one; see FD2. No prompt change, no persona-behavior change, no new subprocess/credential/network surface, no dependency change.

**Context (locked decisions — do not re-open):**

- **FD1 — The direction is INVERTED: `artifact_refs` is stripped, `State.artifacts` survives. This supersedes roadmap FD3's assumed direction** *(locked 2026-07-12, Opus/Architect, sprint 32 planning pass; user-confirmed)*. Four independent findings force it. **(a) The read path was never adopted:** `get_artifact` — the only function that loads a body from disk — has **zero** production callers, and no module in `src/` reads `docs/artifacts/` back; `artifact_refs` is written by `mirror_to_disk` and read only by `has_artifact`, an existence check that the inline dict answers just as well. **(b) The snapshot's self-containment is load-bearing.** Snapshots are anchored to a pinned `state_root()` (the orchestrator's main checkout), but artifact bodies deliberately follow the chdir into the worktree/clone (`state_io/writer.py:14`). Strip the inline bodies and a snapshot stops being a complete run record and becomes a set of pointers into a tree that `prune-worktrees` can delete, that a `flows/maintenance` clone discards, or that a later `resume --from-issue` (days later, different CWD, possibly a different isolation mode — finding **R10**) cannot resolve. Today the inline fallback masks this entirely. **(c) It buys no tokens.** `State` is orchestrator-side and is never itself serialized into a prompt; the bytes sent to the API are the artifact body either way. Externalization shrinks a Python object and a JSON file, neither of which is billed. **(d) It endangers the cache** — see FD2. **Consequence:** the strip becomes purely subtractive after all, just in the opposite direction from the one FD3 assumed.
- **FD2 — The artifact bodies ARE the prompt-cache prefix; routing them through disk is a cache hazard with no upside** *(locked 2026-07-12, Opus/Architect)*. Under `prompt_style == "cached"`, `personas/declarative/node.py:72-78` wraps the artifact body and places it directly into the **system blocks**, and `tools/llm/client.py:88-89` stamps the `cache_control: {"type": "ephemeral"}` breakpoint on the last system block. Cache hits therefore require those bytes to be **byte-identical across calls**. Keeping them in memory (this sprint's direction) makes that true by construction. Reading them back through `write_text` → `read_text` (FD3's direction) does **not** guarantee it: `Path.read_text()` applies universal-newline translation, so a body carrying `\r\n` (LLM output, or a Windows-authored `requirements.md` arriving as `human_input`) returns changed — silently invalidating a multi-KB cached prefix (full input rate instead of the 0.1× cache-read rate, plus 1.25× to re-create it) **and** silently breaking the digest equality check `mirror_to_disk` relies on for idempotency. Actual token savings live in **prompt composition**, i.e. backlog **BL-3**, which is orthogonal to where the bytes are stored. Do not conflate the two, and do not justify this sprint on token grounds — **it saves zero tokens, by design, and that is fine.**
- **FD3 — The disk write is PUBLICATION and must be KEPT, not deleted with the refs** *(locked 2026-07-12, Opus/Architect)*. `mirror_to_disk` conflates two unrelated jobs: *externalization* (populate refs so `State` can be slimmed — never adopted, deleted here) and *publication* (write the design docs into the working tree). Publication is **load-bearing**: `docs/artifacts/` is **not** in the scaffold's `.gitignore` template, and `flows/maintenance` runs `git_io.commit_all` over the whole tree, so the bodies are committed into the **managed repo's PR** — the loop leaves its architecture doc, sprint plans and manifest behind as the PR's documentation. This works precisely *because* `write_artifact` is CWD-relative and follows the chdir into the clone. Deleting the write would silently strip the design docs out of every generated PR. So `mirror_to_disk` is **renamed** `publish_artifacts` and demoted to a pure side effect (returns `None`, mutates no state) — the function survives; only its state-mutating half dies.
- **FD4 — Removing a field under `extra="forbid"` is a BREAKING schema change, so `schema_version` goes 3 → 4 and `migrate_state_payload` must POP the retired key** *(locked 2026-07-12, Opus/Architect)*. Every prior bump was additive-with-a-default, so migration was a version bump and nothing more (`state.py:112-117`). This one is not: `State.model_config = ConfigDict(extra="forbid")`, so a persisted **v3** payload — which *does* carry `artifact_refs` — will raise a `ValidationError` on load unless the migration explicitly removes the key. Failing to do this bricks `--resume-from` and `resume --from-issue` for **every existing v3 snapshot on disk**. This is the single highest-risk line in the sprint; it gets its own dedicated test (see Task 3).

**Security Considerations:** Purely subtractive plus one rename. No new file-write path — `tools/state_io`'s `write_artifact` remains the single write seam and is unchanged, so `state_io`/`scaffold` stay the only file-write-owning modules. Sanctioned subprocess surfaces stay **exactly five**; `tools/llm/client.py` stays the only `keyring` importer; no network or credential surface is touched. `State` keeps `extra="forbid"` **intact** and `schema_version` **accurate** (3 → 4, with a migration that actually handles the removal — FD4). Removing `artifact_refs` removes a second source of truth for artifact content, which is a modest integrity *improvement*: there is no longer a path by which a stale on-disk body could diverge from the state the personas actually read.

**Risks & Blockers:**
- **FD4 is the sharp edge.** Get the `artifact_refs` pop wrong and every existing v3 snapshot fails to load under `extra="forbid"`. The migration must be exercised against a **realistic v3 payload that actually contains populated `artifact_refs`**, not an empty dict — an empty `artifact_refs: {}` would pass a naive pop-less migration and hide the bug.
- **Do not delete `default_artifact_path`.** It looks like refs machinery (it lives beside `artifact_digest`, which *is* deleted) but publication still needs it to compute `docs/artifacts/<run_id>/<key>`. `artifact_digest` and `ArtifactRef` go; `default_artifact_path` stays.
- **Do not "finish the job" by deleting the disk write.** FD3. The three deletions (`ArtifactRef`, `artifact_digest`, `get_artifact`) plus the field are the whole scope; `write_artifact` and the publication call are **kept**.
- **Preserve publication idempotency without digests.** Today `mirror_to_disk` skips a body whose ref digest already matches, so re-mirroring an unchanged state does **no I/O**. With refs gone that cache is gone, so `publish_artifacts` must compare against the **on-disk content** and skip the write when equal. This is not cosmetic: `flows/maintenance` gates its commit on `git_io.has_changes`, and needless rewrites churn mtimes for no reason. (Content-identical rewrites would not fool `git status --porcelain`, which is content-based — but do the content compare anyway; it preserves the current no-I/O property and is cheaper than the alternative.)
- **The 8 reader modules must come out of this sprint byte-identical.** If a diff touches them, the sprint has drifted into FD3's direction. That is the tell.
- **`engine.py` call sites change shape.** `state = mirror_to_disk(state)` (lines 130, 411) becomes `publish_artifacts(state)` — a statement, not an assignment. Dropping the reassignment is the correct change (nothing in state is updated any more); silently leaving `state = publish_artifacts(state)` would set `state` to `None` and detonate on the next line.
