**Design decisions baked in (from the planning pass + the code as it stands):**
- **Two node archetypes, not one-generic-for-everything.** `GeneratorNode` = single-shot generate â†’ optional revision-merge â†’ optional `resolve_via_document`. It drives the three *document* personas. The Coder stays its own `AgenticNode` (tool loop). A persona that fits neither is a signal that control flow belongs in a gate/edge â€” which is exactly the PM critic-loop finding this sprint acts on.
- **The genuinely-varying persona logic is a small registry of shared services, selected by config â€” not per-class code.** Three axes, each a named strategy:
  - **output-adapter** (`markdown` â†’ a raw-string artifact; `sprint_blocks` â†’ parse `### FILEPATH:` blocks to a JSON list, write each sprint file via `write_artifact`, **and additively emit `task_manifest`** exactly as today; `json_object` â†’ parse a single JSON object, dropping code fences).
  - **revision-style** (`section_merge` â†’ replay the prior artifact as an assistant turn, ask for only corrected `##`/`### FILEPATH:` sections, merge via `sections.merge`; `key_merge` â†’ the JSON analog: re-extract only the flagged keys and merge into the prior object dict, the PM-followup behavior generalized; `full_reextract` â†’ regenerate with findings inlined, no prior-artifact replay â€” the fallback when the prior artifact has no mergeable structure).
  - **input-wrapper** (`none`; `untrusted` â†’ wrap the consumed artifact in `<untrusted_artifact>` tags with the ignore-embedded-instructions preamble, the PM behavior).
- **The prompt file becomes the source of truth, not a pinned copy of it.** Today each persona embeds its template verbatim and `tests/personas/test_prompt_parity.py` guards the copy against `prompts/0N_*.md`. `GeneratorNode` **loads the prompt from disk** (once, at construction, held immutably so cached `system_blocks` stay byte-stable). The parity test inverts into a "config points at an existing, non-empty prompt file whose mandated section headers are present" check. **PM has no prompt file today** (ported from pm-agent-loop, pinned by phrase) â†’ this sprint *creates* `prompts/01_pm_extraction_prompt.md` from the embedded `PROMPT_TEMPLATE`, byte-for-byte, so the declarative PM's system prompt is identical to the classic one.
- **PM's critic *loop* â†’ a structural `CriticGate`; the *checks* (`critic.review`) are unchanged.** Classic `PMPersona.run` does extract â†’ `critic.review` â†’ up to `MAX_REVISION_CYCLES` followup re-prompts (with no-progress detection) â†’ persist (possibly partial) spec â†’ add unresolved findings as `PMPersona` questions that the stage's plain `ArtifactGate` then escalates. Under declarative, `PMPersona` becomes a `GeneratorNode` (`json_object` adapter, `untrusted` input, `key_merge` revision) that does **one** extraction and returns; the critic logic moves wholesale into `CriticGate` (composes `ArtifactGate("project_spec", parse_json="object", require_nonempty_parse=True)` then runs `critic.review` on the parsed spec â†’ REVISE with per-field findings). `execute_stage`'s revise loop replays those findings into the node's `key_merge` revision; its identical-findingsâ†’escalate **is** the no-progress detector the loop used to implement by hand; `MAX_REVISION_CYCLES` becomes the stage's `max_revisions`.
- **`fold_answers` and `resolve_questions` stay resolver *services*, not forward-path node logic.** `resolve_questions` (Architecture + PM) is exactly the shared `resolve_via_document` service (RESOLUTION_PROMPT + the owned document) â†’ `GeneratorNode` exposes it from config. `PMPersona.fold_answers` is resume-time-only (folds human issue answers into the spec + classifies impact); it stays a PM-keyed service the resume path calls, **outside** `GeneratorNode.run`. Neither is on the forward accept path.
- **Default-path safety + parity scope.** All three classic persona classes stay wired under `classic`. Under `declarative`: **Architecture + Sprint Breakdown are strict byte-parity** (identical `system_blocks`, identical prompts, identical adapters/merge â†’ identical artifacts for identical LLM responses, incl. `task_manifest`); **PM is byte-parity on the clean-extraction happy path** (no critic findings â‡’ one extraction â‡’ identical `project_spec`, `revision_history` empty in both) and a **documented behavior change on the multi-cycle/escalation path** (per-field questions become gate findings; a non-converging PM escalates one combined `execute_stage` question to the human issue instead of N per-field questions; the declarative multi-cycle path does not accumulate a `revision_history` trail). `fold_answers` on resume is unaffected â€” it derives `spec_updates` from the human answer + current spec, not from per-question granularity.

**Security Considerations:**
- **PyYAML is the sole new attack surface; constrain it.** The config loader uses `yaml.safe_load` exclusively (asserted by test); configs are in-repo trusted assets loaded by path, never from model output or user input. No `!!python/*` tags can instantiate. The dep is pinned to a CVE-free version and the SBOM regenerated â€” an unpinned or vulnerable YAML parser fails `hatch run audit` (CI gate).
- **The `untrusted` input-wrapper must be preserved verbatim for PM.** PM consumes `human_input` â€” untrusted document text. The declarative `untrusted` wrapper must reproduce the exact `<untrusted_artifact>` framing + ignore-embedded-instructions preamble the classic PM uses; a parity test asserts the wrapped prompt bytes match, so the prompt-injection boundary is not weakened by the refactor.
- **No new subprocess / write / credential surface.** `GeneratorNode` reuses the injected `LLMClient` (budget-enforced, keyring-owning) and writes only through `tools/state_io` (`write_artifact` for sprint files, unchanged path validation). The config loader imports no `keyring`, opens only its own `prompts/`/config files for read, and honors the module-boundary tests (`core/` imports no concrete persona; only `state_io` writes files).
- **`CriticGate` is a `core`-safe gate.** Like `RalphCoderGate`, it must be importable without pulling a concrete persona into `core/`. The `critic.review` checks are pure functions over a `dict[str,str]`; move/expose them so `core/` (or a `core`-safe home for the gate) uses them without importing `personas/pm/persona.py`. Import-layering tests must stay green.

**Risks & Blockers:**
- **The PM escalation-shape change is real â€” surface it, don't hide it.** Retiring the internal loop means a non-converging PM no longer files N per-field human questions; `execute_stage` files one combined "could not satisfy its output gate" question carrying every blank/vague field. This is acceptable (all fields are named in the one question; PM has no automated resolvers, so it always went to a human anyway; `fold_answers` still folds the free-text answer). It is nonetheless a behavior change â†’ **flag-gated, explicitly documented, and NOT parity-claimed for that path.** Do not attempt to reconstruct per-field human questions inside a stateless gate (the gate cannot know it is on the last attempt); that would smuggle loop state back into the gate and defeat the whole move.
- **`revision_history` is dropped on the declarative multi-cycle path.** Classic PM accumulates a `revision_history` list in the spec across followup cycles. The declarative node has no such per-run accumulator (the "history" now lives in `state.stage_history` + gate findings). On the happy path both are empty (byte-parity holds); on the multi-cycle path declarative omits the trail. Document it; do not bolt a bespoke accumulator onto the generic node to fake parity.
- **Prompt-file externalization changes the source of truth â€” the parity test must invert, not just relax.** Once the node loads `prompts/0N_*.md`, the file *is* the prompt; the old "embedded copy matches file headers" test is retired and replaced by "each declarative config names an existing prompt file containing its mandated section headers" (Architecture's `## Output Requirements`, Sprint Breakdown's `## ROLE`/`## OBJECTIVE`/`## TASK INSTRUCTIONS`, PM's extraction preamble phrase). The classic classes keep their embedded templates + existing parity assertions **unchanged** until Phase 6 deletes them, so the two suites coexist behind the flag.
- **Cache-stability regression risk.** The classic personas are hand-tuned so `system_blocks` are byte-identical across revision attempts (prompt caching). `GeneratorNode` must build `system_blocks` from immutable config + state fields only (prompt loaded once at construction), with all volatile content in user turns â€” a parity test on the exact `system_blocks` bytes for a revision attempt guards this.
- **This is a refactor of three load-bearing personas â€” no observable regression to the classic path.** The classic `ArchitecturePersona`/`AgileSprintBreakdownPersona`/`PMPersona` keep their code, behavior, and existing tests **verbatim**; the shared services are *extracted* (Architecture/Sprint Breakdown already share the section-merge revision shape) or *newly written* (the `json_object`/`key_merge`/`untrusted` trio for PM), never rewritten in place.
- **No live end-to-end verification on this branch** (no LLM key). Verification is unit-level against the fake-LLM harness: byte-parity of Architecture/Sprint-Breakdown artifacts classic-vs-declarative for identical fake responses; PM clean-path spec equality; `CriticGate` REVISE/ACCEPT/escalate decisions; cache-stable `system_blocks`; the `untrusted` wrapper bytes; cross-engine equivalence. Observed real-run behavior/cost is deferred to the live host run (`sprints/DEFERRED_VERIFICATION.md`).
