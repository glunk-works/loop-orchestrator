### FILEPATH: /sprints/35_migration_merge/sprint_plan.md

**Sprint Goal:** Land `feat/mcp-langgraph-migration` on `main` as its one-time merge commit, and close the two-branch topology that produced BL-12, BL-13, and BL-14 in a single week. The merge itself is the *smallest* part of this sprint: it is already conflict-free and adds no code to the integration branch. The work is everything that must be true **before** the merge button is pressed and **immediately after** — a settings sequence that is load-bearing in both directions (BL-13), a branching protocol that becomes false the instant the merge lands, two known-false claims in `src/` that should not become `main`'s first impression, and four Dependabot PRs that would turn a clean merge into a silently-uncontested one if they land first.

**Out of scope:** The five standing checks in `sprints/DEFERRED_VERIFICATION.md` (FD4 — none blocks this merge; they get their own sprint immediately after). Any dependency change (so **no `sbom` regeneration**). The Dependabot major-version bumps #50–53 themselves (FD5 — they are *sequenced* by this sprint, not performed by it). Any new feature, and any change to the eight checks, the ruleset, or the four workflows beyond what the merge itself carries.

**Context (locked decisions — do not re-open):**

- **FD1 — The merge is already clean, and it adds exactly ONE file to the integration branch. Measured, not assumed** *(locked 2026-07-13, Opus/Architect, sprint 35 planning pass; user-confirmed)*. `git merge-tree origin/main origin/feat/mcp-langgraph-migration` produces tree `ae7d088` with **zero conflicts**. Diffed against both sides: merged-tree vs. `feat` is **one file added — `.github/workflows/ruleset-drift.yml` (70 lines)** — and nothing else; merged-tree vs. `main` is the whole migration (190 files, ~17.8k insertions). So `main`'s three "ahead" commits (PRs #46/#47/#49 — the BL-12 and BL-14 backports) are **byte-identical to `feat`'s content** apart from that one workflow. The merge is a catch-up in one direction plus a single file in the other. It is **not** a code change: the merged tree *is* `feat`'s tree, plus a workflow. Every argument in this plan that treats the merge as low-risk rests on this fact — **re-run the `merge-tree` probe if anything lands on either branch, because it is only true until it isn't** (see FD5).
- **FD2 — The merged workflow set is green, verified by running it** *(locked 2026-07-13, Opus/Architect)*. `ruleset-drift.yml` has never existed in a tree that `tests/test_ci_config.py` has run against — the branch that carries the test and the branch that carries the workflow have been disjoint. `test_all_workflow_actions_are_pinned_to_commit_shas` **globs** `.github/workflows/*.yml`, so the merge silently widens its sweep to a fourth file. Checked rather than assumed: dropping `ruleset-drift.yml` into the tree and running `hatch run test tests/test_ci_config.py` gives **15 passed** — it declares no `uses:` at all (it is a pure `gh api` script), so the SHA-pin sweep is satisfied trivially. No other test reads it. **The migration PR's `test` job is therefore expected green, and if it is not, the cause is something this plan did not foresee — stop and diagnose, do not paper over it.**
- **FD3 — The two `publish_artifacts` findings land on `feat` BEFORE the migration PR is cut, and can NEVER ride inside it** *(locked 2026-07-13, Opus/Architect; user-confirmed)*. `hitl-review.yml` binds an accepted review to an **exact head commit**. The migration PR's head *is* `feat`, so pushing the fix to `feat` after that PR's review is posted **invalidates the review of the entire 109-commit merge** and forces a re-review. The ordering is therefore not a preference: fix first, on its own PR, with its own cheap review of a two-line diff; then cut the migration PR from a **settled** `feat` head and freeze it. The review cost is identical either way (the migration PR needs an `architect-review` regardless, because it touches `src/`) — so there is no saving to be had by deferring, only a known-false docstring shipped to the default branch.
- **FD4 — None of the five DEFERRED_VERIFICATION checks blocks this merge, because the merge verifies nothing and un-verifies nothing** *(locked 2026-07-13, Opus/Architect; user-confirmed)*. The merged tree is `feat`'s tree (FD1). Every line of it has been the branch all work happens on for ~109 commits, and no check in §1/§5/§6/§7/§8 becomes more or less true by changing which branch name points at it. Making them gate `main` would be **new policy invented at the merge**, not a standing obligation being honored. **But they do not get to stay "deferred" either:** "after the merge" has been the answer for five sprints, and an obligation with no scheduled home is an obligation nobody owns. Task 6 gives them one.
- **FD5 — Dependabot PRs #50–53 must NOT merge before the migration PR. They rewrite the exact lines the clean merge depends on** *(locked 2026-07-13, Opus/Architect)*. All four are open against `main` and each edits `ci.yml`'s `uses:` SHA lines — the very lines that are currently *identical* on both branches and therefore invisible to the merge (FD1). Merge any one of them first and `main`'s `ci.yml` diverges from `feat`'s on those lines, converting today's conflict-free merge into a **conflicted** one. That is not merely inconvenient: per `.ai/context/workflow.md`, **a conflicted PR runs ZERO CI, silently** — GitHub cannot build `refs/pull/N/merge`, so the eight required checks never report and the PR sits looking merely slow. The sprint would then be debugging a self-inflicted wound. Sequence is absolute: **migration merge first, Dependabot bumps second.** They are also all four **major-version** jumps (`checkout` 4→7, `upload-artifact` 4→7, `setup-python` 5→6, `gitleaks-action` 2→3) and need real review, not a rubber-stamp — which is Task 6's business, not this task's.
- **FD6 — `delete_branch_on_merge: true` will try to delete `feat/mcp-langgraph-migration`, and the ruleset forbids exactly that. Expect it; do not read it as a failed merge** *(locked 2026-07-13, Opus/Architect)*. The repo has `delete_branch_on_merge: true` (verified correct under BL-11), so merging any PR auto-deletes its head branch. The `protected-integration-branches` ruleset targets `refs/heads/feat/**` with a **`deletion`** rule. These two collide precisely once, on this PR. The outcome is benign — **the branch survives** — but GitHub may surface an error or silently skip the cleanup, and a session that has not been told to expect it will misread it as the merge having gone wrong. It did not. (This also means the post-merge retirement of `feat` in Task 5 is a *deliberate* act, not something the merge does for you.)
- **FD7 — The `architect-review` on the migration PR is an INTEGRATION review, not a re-review of 109 commits** *(locked 2026-07-13, Opus/Architect)*. The PR touches `src/`, so `hitl-review.yml` requires a fresh-session Opus review against its head commit — non-negotiable, and it is a required check on `main`. But re-reviewing 190 files and ~17.8k insertions wholesale is theatre: **every one of those commits already carried its own architect-review on its own sprint PR**, which is the entire point of the per-sprint gate. The review posted here must therefore *say what it actually reviewed* — the merged tree's coherence (does the merge produce the workflow set FD1/FD2 predict; is `ci.yml` the post-sprint-33 version; do the four workflows co-exist correctly; is `main`'s topology now correct) — and must **explicitly decline** to relitigate the history. A review that claims to have re-read the whole migration would be a lie of exactly the kind this repo's gates exist to prevent. It still carries the standard header and the fresh-session attestation.

**Security Considerations:** No new code, no new dependency, no new credential path, no new subprocess surface — the merged tree is `feat`'s tree (FD1). The security-relevant content of this sprint is entirely in **sequencing controls that already exist**, and every one of them fails *open* if mis-sequenced. `allow_merge_commit` is the sharp one: it must be **enabled** across the merge (BL-13) and **disabled immediately after**, and the window in between is the only period in this repo's life when a non-squash merge is possible on any PR. Keep that window short and do not open other PRs inside it. Note also that the merge is the moment `main` first receives the migration's full `pyproject.toml` dependency set — Dependabot's `pip` ecosystem (daily, and it reads config **only** from the default branch, per BL-14) will scan dependencies it has never seen, so a burst of alerts or PRs afterwards is **expected and correct**, not a sign of a new problem. Triage them on their merits.

**Risks & Blockers:**
- **Re-run the merge probe before opening the PR.** FD1's conflict-free finding is a fact about two specific commits, not a property of the branches. Anything that lands on `main` (a Dependabot PR, a hotfix) or on `feat` invalidates it. `git merge-tree --write-tree origin/main origin/feat/mcp-langgraph-migration` must still exit clean.
- **Do not disable `allow_merge_commit` before the merge** (BL-13). It is a repo-wide setting that does not distinguish PR by PR; flipping it early removes the merge-commit button from the one PR that needs it, and nobody re-reads a months-old plan at the moment it finally executes.
- **Do not SQUASH the migration PR.** Squashing collapses 109 commits of migration history into a single commit — irreversibly, on the default branch. The merge-commit button is the whole reason `allow_merge_commit` was held enabled. The repo's own `squash_merge_commit_title: PR_TITLE` setting and every other PR's convention point the other way; this is the documented one-time exception (`.ai/context/workflow.md`).
- **Freeze `feat` once the migration PR's review is posted** (FD3). Any push to it invalidates the `architect-review` binding and restarts the gate on a 190-file diff.
- **The PR title must be ≤72 characters** and conventional-commit shaped. `pr-title` is a required check on `main`. A long title on *this* PR is not the cheap mistake it was pre-sprint-33 — it now simply fails a required check (the BL-10 starvation path is closed), but it still costs a cycle.
- **A conflicted PR runs zero CI, silently.** If the eight checks appear to hang rather than fail, check `mergeable_state` **before** blaming slow CI (this is the exact trap FD5 exists to keep the sprint out of).
- **Two architect-reviews are needed in this sprint, each in its own fresh session** — one on the Task 1–5 PR into `feat` (it touches `src/`), one on the migration PR into `main` (FD7). `/model opus` mid-session is not a review session.

**Tasks:**

- **Task 1: Fix the two open `publish_artifacts` findings from PR #39's review**
  - **Description:** `tools/artifact_store.py`'s `publish_artifacts` docstring (and the module docstring above it) claim that *"publishing an unchanged state does no I/O"* — this is **false**. The function calls `path.exists()` and `path.read_text()` for **every artifact on every stage**; what it actually avoids is redundant *write* I/O. Correct both claims to say what the code does (skips the **write** when the on-disk body already matches — a read-compare, not a no-op). Separately, that `path.read_text()` carries **no explicit encoding**, so it decodes with the platform default; artifact bodies are model-generated text and can contain any UTF-8. Pass `encoding="utf-8"` explicitly, matching whatever `write_artifact` writes with (**check `tools/state_io` and match it — do not assume**). These are the two non-blocking findings the Opus review of PR #39 (sprint 32) left open; they are small, and FD3 explains why they land here rather than after the merge.
  - **Target Files:** `src/loop_engine/tools/artifact_store.py`, `tests/tools/test_artifact_store.py`
  - **Acceptance Criteria:**
    - Neither the module docstring nor `publish_artifacts`' docstring claims the function does no I/O; both accurately describe a read-compare that skips a redundant **write**.
    - `publish_artifacts`' read-back passes an explicit `encoding="utf-8"`, and it matches the encoding `write_artifact` uses (verified by reading `state_io`, not assumed).
    - A regression test covers the round-trip that motivated the finding: an artifact body containing non-ASCII characters is published, re-published unchanged, and the second call performs **no write** — with the body still byte-identical on disk.
    - `hatch run lint` / `hatch run format` clean; full suite green.
