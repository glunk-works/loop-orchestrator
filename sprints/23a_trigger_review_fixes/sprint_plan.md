**Context from the review (locked decisions â€” do not re-open):**
- **Finding #1 (CONFIRMED / high): the dispatched task is not retained.** `InProcessDispatcher.dispatch` does `asyncio.create_task(self._run(request, key))` (`trigger/dispatch.py:36`) and discards the returned `Task`; nothing else holds a strong reference (`self._active` stores only the `(repo, issue)` key tuple, not the task). Per the CPython docs, the event loop keeps only a **weak** reference to a task â€” "a task that isn't referenced elsewhere may get garbage collected at any time, even before it's done." Two-headed failure: under GC pressure while the run is parked in `await asyncio.to_thread(...)`, the task is collected and cancelled, so **(a)** the run silently dies, and **(b)** because `_run`'s `finally: self._active.discard(key)` never executes, that `(repo, issue)` is **permanently wedged** in `_active` and every future `/agent-run` for that issue is dropped as a phantom duplicate. The existing tests never surface this â€” they `await asyncio.sleep(0.2)` right after dispatch, which keeps the task transitively reachable and never forces a collection.
  - **Decision (Architect): retain in-flight tasks in a strong-referenced set with a done-callback that discards.** Hold each task in an instance set (e.g. `self._tasks: set[asyncio.Task]`), `add` on create, and `task.add_done_callback(self._tasks.discard)` so the reference is dropped exactly when the task finishes. Keep the existing `(repo, issue)` dedupe on `self._active` as-is (it is the semantic dedupe; the task set is the liveness fix). Do **not** switch to `threading.Thread`/an executor you manage by hand â€” stay on `asyncio.to_thread` + `create_task`, just retained.
- **Finding #2 (CONFIRMED / medium): unguarded `await request.json()`.** After HMAC passes (`trigger/app.py:55`), `payload = await request.json()` (`trigger/app.py:58`) decodes the body with no guard. A `json.JSONDecodeError` is not an `HTTPException`, so it propagates to the ASGI server â†’ **HTTP 500**, contradicting both the module docstring ("HMAC-verified *before* any JSON parsing") and `parse.py`'s "a webhook receiver must never 500 on an unrelated or misshapen delivery." Realistic trigger: a GitHub webhook configured with content-type **`application/x-www-form-urlencoded`** (a common misconfiguration) sends `payload=%7B...`; the `X-Hub-Signature-256` is computed over the raw body regardless of content-type, so HMAC **passes**, then `json.loads` fails â†’ 500 and the delivery is silently broken.
  - **Decision (Architect, HITL 2026-07-09): return `400 Bad Request` on an unparseable body (do NOT 204, do NOT 500).** The sender is authenticated (HMAC passed) but sent a body we cannot parse â€” a real misconfiguration the operator **should** see. `400` surfaces the delivery red in GitHub's webhook UI so the content-type gets fixed, while honoring "never 500." This differs deliberately from `parse_event`'s valid-JSON-but-non-dict / unrelated-event path, which stays a `204` no-op: an *unparseable* body is a broken delivery, not an unrelated one. Parse the JSON from the already-read `raw_body` bytes (do not re-read the request), guarding only the decode.
- **Finding #3 (CONFIRMED / low-medium): run failures are never logged by the dispatcher.** `_run` (`trigger/dispatch.py:38-49`) has `try/finally` but no `except`. When `runner.run_new` raises inside the worker thread, `asyncio.to_thread` re-raises into `_run`, the `finally` clears the key, and the exception propagates out of the (un-awaited) task â€” its only trace is asyncio's default `"Task exception was never retrieved"` at GC time, on the asyncio logger, not this module's `logger`. An operator sees `run starting for acme/widgets#5` and then nothing correlated.
  - **Decision (Architect): catch-log-swallow in `_run`.** Add `except Exception: logger.exception("run failed for %s#%s", *key)` before the `finally`. Swallow (do not re-raise): the run is fire-and-forget, the caller (the webhook) already returned `202`, and re-raising only re-creates the "never retrieved" noise. The `finally` still discards the dedupe key so the issue can be retried. This pairs with #1 â€” once the task is retained, its exception would otherwise be reliably reported by asyncio at task-done; explicit logging through the module logger is the intended, greppable signal.

**Risks & Blockers:**
- **Finding #1's regression test must actually pin the retention, not just re-prove the happy path.** A test that dispatches and sleeps proves nothing (that is exactly what the existing tests do and they pass today). Assert the *structural* invariant instead: (a) while a run is in-flight, the dispatcher holds a strong reference (`len(dispatcher._tasks) == 1`); (b) after the run finishes the set is empty (the done-callback fired); (c) â€” the real failure mode â€” when the run **raises**, the `(repo, issue)` key is still released from `_active` so a re-dispatch for the same issue succeeds. Do not attempt to force a real GC-collection race in CI (flaky); the reference-set assertion is the durable proof.
- **Do not change the dedupe semantics.** `self._active` on `(repo_full_name, issue_number)` is correct and load-bearing (redelivery / label+comment double-fire). The task-set is *additional* state for liveness only; keep the two separate and keep every existing `test_dispatch.py` assertion green unchanged.
- **`400` vs `204` is a deliberate split â€” keep it.** Only an *unparseable* body returns `400`; a well-formed-JSON-but-unrelated/ping/non-dict delivery must still return `204` via the existing `parse_event â†’ None` path. Do not collapse the two. Verify the existing `test_ping_and_unrelated_events_are_no_ops` (204) still passes unchanged alongside the new `400`-on-bad-body test.
