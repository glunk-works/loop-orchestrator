# Survivor triage — `core/` mutation audit (Sprint 38, Task 2, BL-23)

Task 2's deliverable: a `keep` / `fix` / `delete` verdict, with a reason, for every one
of the 147 survivors T1 produced in
[`mutation_baseline.md`](mutation_baseline.md). Per FD3, every survivor was also
cross-checked against the **full** test suite (not just `tests/core/`) by applying
that one mutant directly to `src/loop_engine/core/` and re-running everything —
distinguishing **mislocated coverage** (the full suite kills it; the real fix is a
local `tests/core/` unit test) from **genuinely uncovered** (the full suite misses it
too; the real fix is a new test or, in a couple of cases, a source fix) from
**equivalent** (no test — anywhere, ever — could observably distinguish the mutant
from the original, so no test is owed).

## Methodology note — the static-guard scope boundary (FD1, restated)

**This report says nothing about the repo's static structural guards** —
`tests/tools/test_subprocess_surfaces.py`, `test_encoding_boundary.py` /
`test_ast_open.py`, the `core/`↔`personas/` import-boundary tests, and
`test_mcp_provider.py`'s verb-disjointness assertions. That is by design, not an
oversight: mutmut's mutation operators (arithmetic/boundary/keyword swaps, string
literal wrapping, argument substitution) mutate *values inside a function body*, not
*the shape of the source tree*. They cannot generate the constructs those guards
exist to catch — a sixth `subprocess.run` call, an `import gzip as gz` that dodges a
name-based scan (BL-15), a back-channel `core/`↔`personas/` import, an overlapping MCP
verb set. A green mutation run says nothing about whether those guards would actually
fire on a real violation. **Do not read the absence of those guards from this report
as a clean bill** — their own adversarial-injection audit is a separate, deferred
backlog item (see `docs/backlog.md`), per the sprint plan's FD1.

## How to read a verdict

- **`keep`** — no code change. Either the mutant is **provably equivalent** (the
  mutated code is behaviorally identical to the original for every input the program
  can actually produce — proven here by tracing the actual call graph, not just by
  "no test happened to catch it"), or the mutated text is **cosmetic** (an
  LLM/human-facing message string that no code parses or asserts on character-for-character).
  Per FD5, an equivalent mutant is a legitimate `keep`, not a prompt to contort a test.
- **`fix`** — a real coverage gap. The report names the missing test (or, in a small
  number of cases, a genuine implementation smell worth a source-level fix) rather
  than just restating "add a test." Landing these is Task 3's job, not this one's.
- **`delete`** — a test that constrains nothing and should be removed. (None found in
  this pass — see the summary below.)

Two patterns recur often enough to name once instead of repeating in every row:

1. **Terminal `StageOutcome` arguments are structurally dead.** `execute_stage` returns
   `StageOutcome(state, next_index, carried_findings, carried_until, terminal=True)`
   from several early-exit branches (budget exhaustion, tool-loop exhaustion, an
   issue pause). Once `terminal=True`, `graph_engine._next_target` checks `gs["done"]`
   **first** and routes straight to `END` without ever reading `stage_index`;
   `run_graph_loop` then returns only `result["state"]`. So `next_index` /
   `carried_findings` / `carried_until` are provably unread on any terminal return —
   mutating them is invisible not because a test is missing, but because nothing
   downstream of a terminal outcome ever looks at them. This is verified by reading
   `graph_engine.py`, not assumed.
2. **A single-element list makes `"; ".join(...)` separator-invariant.** Several
   survivors mutate a join separator (e.g. `"; "` → `"XX; XX"`); every test that
   reaches that line only ever has exactly one item in the list being joined, so no
   separator could ever show up in the output. Real gap, same one-line fix each time:
   exercise the multi-item case.

non-equivalent survivors additionally carry their FD3 full-suite cross-check result.
