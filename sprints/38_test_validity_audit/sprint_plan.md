### FILEPATH: /sprints/38_test_validity_audit/sprint_plan.md

**Sprint Goal:** Turn the BL-15/16/18/20 failure mode — *a check that verifies the wrong property while reporting success* — on the test suite itself, starting where the backlog says to start (`core/`). The instrument is **mutation testing**: mutate `core/` source and ask, per mutant, *"would a test fail if the code were wrong?"* rather than *"does a test pass?"*. Every **surviving mutant** on `core/` is a line no test constrains. The deliverable is **a keep/fix/delete verdict per survivor, with a reason** (the backlog's explicit output shape — "not a smaller number"), the concrete fixes landed, and a reusable `hatch run mutate` instrument for later BL-23 passes. This is BL-23's **first** pass, deliberately scoped to `core/` (930 lines, 5 files) — small enough to run to completion and triage by hand this sprint.

**Out of scope:**
- **The static structural guards** (`test_subprocess_surfaces.py`, `test_encoding_boundary.py`/`_ast_open.py`, the `core/`↔`personas/` import-boundary tests, `test_mcp_provider.py`'s verb-disjointness). Off-the-shelf mutation operators (mutmut's catalog: operator/constant/boundary/keyword swaps) **cannot generate the violating construct** these guards exist to catch — a 6th `subprocess.run`, an `import gzip as gz` that dodges a name-scan (BL-15), a back-channel import, an overlapping MCP verb set. Running only mutmut against them would report them well-covered while leaving their real weakness unprobed — i.e. reproduce the BL-23 defect *inside* the BL-23 audit. Their audit is **adversarial invariant-injection**, a different instrument; filed this sprint as a new backlog item (see FD1 / the deliverable list) and taken as its own beat. **This is the single most important scoping decision in the plan — do not let mutmut's green on these guards be mistaken for coverage.**
- **The rest of the 555-test suite** beyond `core/` and its behavioral tests. BL-23 is a multi-sprint program; this is pass one. `personas/`, `tools/`, `flows/`, `trigger/` come in later beats.
- **A mutation-score CI gate.** Whether the instrument becomes a gate or stays an on-demand script is **deferred to after this run** (owner decision, this planning pass — evidence first): stand up the tooling, get real survivor counts and per-mutant timings, *then* decide in a follow-up beat. BL-16's lesson (don't add a gate you won't keep functional) plus mutation testing's minutes-to-hours cost make a blind commitment to a gate the wrong call now.
- **Deleting tests to make the suite faster.** That is **BL-22** (done, sprint 37), and conflating the two would let "this test is slow" masquerade as "this test is invalid." A slow test that kills mutants stays. The output is a *verdict*, not a smaller number.
- **Chasing a 100% mutation score.** Equivalent mutants (a mutation that produces behaviorally identical code) are expected and are a legitimate `keep` verdict (FD5). The goal is a reasoned triage of every survivor, not zero survivors.

**Context (locked decisions — this planning pass, Opus/Architect, owner-confirmed; do not re-open):**

- **FD1 — Scope is `core/` *behavioral* mutation only; the static structural guards are a separate instrument, deferred** *(locked 2026-07-15)*. Mutation testing answers "does a *behavioral* test constrain this line?" It is the right instrument for `core/` (engine, gates, coder_gate, graph_engine, state — all behavioral) and its behavioral tests in `tests/core/`. It is the **wrong** instrument for the repo's static structural guards, which assert on the shape of the source tree via AST scans and set algebra, not on runtime behavior — no mutation operator in mutmut's catalog emits the construct those guards catch (BL-15 is the proof: a name-based `open()` scan is defeated by `import gzip as gz`, a mutation no source-mutator will ever generate). Sprint 38 therefore audits `core/` with mutmut and **files a new backlog item — the adversarial invariant-injection audit of the static guards** (BL-15 generalized: deliberately inject each violating construct and assert the guard goes red). Do not attempt the static guards with mutmut this sprint.

- **FD2 — Tool is `mutmut`, added as a dev dependency to the hatch `default` env** *(locked 2026-07-15)*. `core/` is a narrow target that plays to mutmut's strengths (low-friction, `pyproject.toml` config, custom test runner, good `results`/`show`/`html` survivor reporting for the keep/fix/delete workflow) — cosmic-ray's session/distribution machinery is overkill for one small package now (revisit if BL-23 later scales to the whole suite). Adding the dep **changes `pyproject.toml` deps ⇒ `hatch run sbom` must be regenerated and `hatch run audit` must pass** (CLAUDE.md rule). Pin an exact version, matching the repo's fully-pinned convention.

- **FD3 — Per-mutant runner is scoped to `tests/core/`, but every survivor is cross-checked against the FULL suite before a fix/delete verdict** *(locked 2026-07-15)*. Running the full ~54s suite per mutant × hundreds of mutants is hours; running only `tests/core/` per mutant is minutes and correctly attributes survival to **`core/`'s own unit tests**. But a mutant that `tests/core/` misses might still be killed by an integration test elsewhere (`tests/personas/`, `tests/flows/`). That is **not a false survivor to discard — it is itself a finding**: coverage of a `core/` behavior living only in a far-flung integration test is mislocated and fragile. So the triage rule (T2) is: for each `tests/core/`-scoped survivor, re-run the **full** suite against that one mutant; if the full suite kills it → verdict is `fix` with reason *"core/ behavior covered only outside tests/core/ — add a local unit test"*; if the full suite also misses it → verdict is `fix` (genuine hole) or `keep` (equivalent mutant). Either way the survivor is triaged, never silently dropped.

- **FD4 — No numeric mutation-score threshold; the Definition of Done is a reasoned verdict per survivor** *(locked 2026-07-15)*. Consistent with the backlog ("a verdict per test — keep/fix/delete — with a reason, not a smaller number") and with BL-16 (an inert numeric gate passes every alarm because it is inert). Acceptance is: mutmut runs to completion on `core/`, **every** surviving mutant appears in a checked-in audit report with a `keep`/`fix`/`delete` verdict and a one-line reason, and every `fix`/`delete` verdict is landed. The score is recorded as *evidence*, never asserted as a gate.

- **FD5 — Equivalent mutants are expected and are a legitimate `keep`** *(locked 2026-07-15)*. Some `core/` lines are un-killable in principle (a mutation that yields behaviorally identical code — e.g. a reordering with no observable effect, a constant that no path distinguishes). Marking these `keep — equivalent mutant` with the reason is the correct triage, not a test to add. Do not contort a test to kill an equivalent mutant; that manufactures the exact implementation-pinning tax BL-3 and BL-23 warn against.

**Security Considerations:** `mutmut` is a **dev/test-only** dependency — the runtime app (`src/loop_engine/`) never imports it; it lives in the hatch `default` env alongside `pytest`/`ruff`/`pip-audit`, so `hatch run audit` (pip-audit) will scan it and `hatch run sbom` will record it. It introduces **no new credential path, no new subprocess surface** in `src/` (mutmut spawns test-runner subprocesses, but only from the dev env, never from shipped code — the five sanctioned `src/` subprocess surfaces are unchanged, and `tests/tools/test_subprocess_surfaces.py` stays green). The one diligence item: mutation testing **writes mutated source to disk and runs it**; it must run only against the working tree in a dev env, never in CI against `main` — which is exactly why the gate decision is deferred (FD4/out-of-scope). Any `fix`-verdict change to `src/core/` is normal reviewed source and goes through `architect-review` on its PR.

**Risks & Blockers:**
- **The static-guard scope error is the whole ballgame (FD1).** The failure this sprint must not commit is letting mutmut's green on the static structural guards read as "those guards are sound." They are out of scope *because the instrument can't attack them*, not because they're trusted. State this explicitly in the audit report so a reader never draws the wrong conclusion.
- **mutmut runner must invoke the tests through hatch/pytest correctly.** `core/` tests rely on repo fixtures/conftest; the mutmut runner must be `hatch run test tests/core/` (or the equivalent that loads `conftest.py`), not a bare `pytest` that misses config. Verify the *baseline* (unmutated) run is all-green under the mutmut runner before trusting any survivor — a misconfigured runner reports every mutant as "survived" and every survivor is spurious.
- **Runtime.** Even scoped to `core/` + `tests/core/`, mutmut runs `tests/core/` once per mutant. Record the wall-clock (it informs the deferred gate decision, FD-out-of-scope). If it is unexpectedly long, mutmut's own parallelism / coverage-guided test selection is the lever — not widening or narrowing the audit scope.
- **A `fix` verdict may touch `src/core/` (architect-review applies).** Most fixes will add/strengthen tests in `tests/core/` (no `src/`, exempt). But a survivor can expose a genuine `src/core/` bug (the best possible outcome); fixing it touches `src/` ⇒ that PR needs a fresh-session `architect-review`. Keep test-only fixes and source fixes on separate, clearly-scoped PRs so the exempt ones stay exempt.
- **Do not weaken a test to raise the score.** FD5. An equivalent mutant is a `keep`, not a provocation to over-fit a test to the implementation.
- **Sequential, evidence-carrying PRs.** T1 (tooling + baseline run) lands first so its survivor report and timings exist before T2/T3 triage and fix against them; the gate-vs-script follow-up decision is taken only after T1's real numbers are in hand.

  - **Description:** For each survivor from T1, produce a verdict. Apply FD5: behaviorally-identical mutants are `keep — equivalent mutant`. The output is the audit report — the sprint's primary deliverable — one row per survivor: `file:line`, the mutation, verdict, reason, and (for `fix`) the concrete test/source change T3 will make.
  - **Target Files:** `sprints/38_test_validity_audit/audit_report.md` (the keep/fix/delete verdict table + a short methodology note that **explicitly states the static structural guards were out of scope and why**, per FD1/Risks, so no reader mistakes their absence for a clean bill).
  - **Acceptance Criteria:**
    - **Every** survivor from T1 has exactly one verdict (`keep`/`fix`/`delete`) and a one-line reason — none dropped.
    - Each non-equivalent survivor carries its full-suite cross-check result (killed-elsewhere vs. genuinely-uncovered), per FD3.
    - The report's methodology note states the static-guard scope boundary (FD1) in plain language.
    - Docs-only task → `architect-review`-exempt.

- **Task 3: Land the fixes — implement every `fix` and `delete` verdict; document every `keep`**
  - **Description:** Execute T2's verdicts. `fix` → add/strengthen the `tests/core/` test that kills the mutant (or, where a survivor exposed a real `src/core/` defect, fix the source — the ideal outcome); `delete` → remove the orphan/vestige with its reason recorded in the PR; `keep` → no code change, rationale already in the report. Re-run mutmut afterward to confirm the `fix`ed mutants are now killed and the survivor set matches the report's residual (only `keep`s remain). Keep **test-only** fixes and any **`src/core/` source** fixes on separate PRs so the test-only ones stay `architect-review`-exempt and the source ones get a proper fresh-session review.
  - **Target Files:** `tests/core/*` (strengthened/added tests), and — only if a survivor exposed a real defect — `src/loop_engine/core/*` (source fix, its own PR). No change to the default test scripts or the ruleset.
  - **Acceptance Criteria:**
    - Every `fix` verdict has a landed test (or source fix) that provably kills its mutant; re-running `hatch run mutate` shows those mutants killed and only `keep`/equivalent survivors remaining, matching the report.
    - Every `delete` verdict's test is removed with a recorded reason; no behavioral coverage is lost in the process (a delete is for a test that constrains *nothing*, verified by mutmut, not a slow test).
    - Full suite green; lint/format clean. Any PR touching `src/core/` carries a fresh-session `**Opus/Architect HITL review (automated)**`; test-only PRs are exempt.
