### FILEPATH: /sprints/29_coder_host_run_hardening/sprint_plan.md

**Sprint Goal:** Close the two **Coder-convergence gaps** the Phase-6 V2 host runs surfaced — both orthogonal to the flip-block flags, both blocking a real Coder-ACCEPT → terminal `COMPLETED` under the target production config. (1) A stuck inner tool loop must not crash the run, and its blunt 12-iteration cap must not guillotine a legitimately-progressing increment. (2) The agentic Coder must be able to **verify `ruff check` / `ruff format --check`** the same way it already verifies `pytest` via `run_tests`, so a task carrying a "ruff clean" acceptance criterion no longer *escalates on missing tooling*. After this sprint, a container-sandboxed Ralph run over an unambiguous, escalation-free spec can reach `COMPLETED` within budget — the observation sprint 27's **V2 still needs** (held open, per the 2026-07-10 host session).

**Why this is its own sprint (not folded into 27):** the flip block (27) is *subtractive* — it deletes proven-redundant scaffolding and adds no behavior. This sprint is *additive/corrective* — it adds a new Coder tool (a new sanctioned subprocess surface) and hardens the inner tool loop. It sequences **strictly before** a re-attempt of sprint 27's V2-for-`COMPLETED`. Companion to sprint 28 (which closed F-GATE-SANDBOX): 28 made the gate *runnable* under container; 29 makes the Coder *converge* under it.

**Context (what the V2 host runs showed — `sprints/DEFERRED_VERIFICATION.md` "Phase 6 flip-block host-run results"):**
- **Run #1** reached the container gate and checked off 3 tasks, then **crashed** with `ToolLoopExceededError` when a trivial `truncate` increment blew the 12-iteration inner cap — and that exception was **uncaught** in `core/engine.py` (unlike `BudgetExceededError`/`TruncatedResponseError`), so a stuck loop aborted the whole run instead of failing the stage. → **Finding F-TOOLLOOP-CAP.**
- **Run #3** (tightened spec) converged **11 tasks** across all 6 sprints (whole `textkit` library written, `truncate` converged), and terminated cleanly at `BUDGET_EXCEEDED` one sprint short of `COMPLETED` — the container gate **ACCEPTed 11×** (F-GATE-SANDBOX host-verified live). No escalation, no crash.
- **Runs #2 / #4** never reached `COMPLETED` because they **escalated** (upstream spec ambiguity; the Coder asking for `ruff` it cannot run) and the escalation's `gh issue create` crashed on the remote-less scratch tree. Run #4's escalation was structural: *"my toolset does not expose a way to execute `ruff check` / `ruff format --check`."* → **Finding F-CODER-NO-LINT.**

**Scope / what lands:**
- Task 1 (F-TOOLLOOP-CAP) — **already implemented this session** (working-tree, green: `hatch run test` 565 passed, lint/format clean); this sprint carries its HITL review + commit.
- Task 2 (F-CODER-NO-LINT) — new `run_lint` Coder tool over `ruff check` + `ruff format --check`, exposed in-process and over MCP, with the subprocess-surface invariant amended.
- Task 3 — doc/pointer reconciliation; V2 stays **open** pending a real `COMPLETED`, now gated on Tasks 1–2 + escalation-free staging.

**Out of scope:**
- **Gate-enforced lint.** This sprint gives the Coder the `run_lint` *tool* (so it can self-verify and stop escalating). Making the Coder *gate* fail on dirty lint (beyond the existing green-`pytest` gate) is a separate decision — not here.
- **The Ralph inner-loop progress watcher** (`docs/backlog.md` BL-4) — raising the cap to 40 is a finite backstop, not the liveness detector; BL-4 remains backlogged.
- **Escalation-proof staging** (a real remote / injected non-crashing `issue_filer` for the V2 scratch tree) — a host-session harness concern, tracked with the V2 re-attempt, not a product change here.

**Security Considerations:** This sprint adds **one** new subprocess surface — `ruff` in `tools/coder_tools/run_lint.py` — taking the sanctioned count from **four to five** (`test_subprocess_surfaces.py` + the CLAUDE.md boundary contract updated in lockstep). Containment mirrors `run_tests` exactly: fixed argv, `shell=False`, hard timeout, output size-capped, target path traversal/symlink-validated via `resolve_tool_path`, executed under the sandboxed devcontainer. **`ruff check`/`ruff format --check` statically parse the target files — they do not *execute* model-generated code** (unlike `run_tests`' pytest), so this surface is strictly *lower* risk than the one it sits beside; no code is run, no network, no credentials. Invariants preserved: `tools/llm/client.py` stays the only `keyring` importer; the three MCP servers' tool-sets stay pairwise disjoint (the coder set grows to `{read_file, list_files, grep, run_tests, run_lint}`, still disjoint from github/issue); `run_lint` stays in the model's Coder tool loop only (never an orchestrator/factory verb).
