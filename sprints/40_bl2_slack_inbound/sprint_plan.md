### FILEPATH: /sprints/40_bl2_slack_inbound/sprint_plan.md

**Sprint Goal:** Land the **second surface** of BL-2's Slack "bot control plane" — the **inbound trigger**: a human types `/agent-run --budget <n> <requirements>` in the configured Slack channel and the engine kicks off a run. The transport is Slack **Socket Mode** — the app opens an *outbound* WebSocket, so Slack never has to reach us — which makes this the **first genuinely live-exercised inbound path in the repo**, sidestepping BL-24's tunnel/routable-host blocker entirely (FD1). The deliverable is an inbound Socket Mode transport in `tools/slack_io` (preserving the `slack_sdk` single-importer boundary), a new top-level orchestrator-level caller `slack_control/` (sibling of `trigger/`/`flows/`) that parses the command, enforces the channel-scoped authorization boundary, and dispatches the run fire-and-forget through `runner.run_new`, and a blocking `loop-engine slack-listen` CLI entrypoint that runs the daemon. This is BL-2 **pass 2 of 3**; the escalation round-trip (pass 3) is deliberately out of scope.

**Out of scope:**
- **The escalation round-trip (route a paused run's questions to a Slack thread and fold the reply back).** That is BL-2 **pass 3** — bidirectional and stateful, and it depends on both notify (pass 1, landed) and this inbound surface existing first. This sprint is trigger-only: Slack → *start* a run. It does **not** touch the resolver ladder, `issue_io`, or `resume --from-issue`.
- **Discharging or deleting `trigger/` (the GitHub webhook) and BL-24.** FD1 is *supersede*, not *delete*: the `trigger/` package stays in the tree, parked in place (nothing routes through it). Its live-verification debt (BL-24) is superseded **in practice** — the practical inbound need is now met by a live-exercised Slack path — but the webhook code and its owed §6 verification are **not** removed this sprint. BL-24 gets a supersede note, not a close.
- **Any authorization boundary beyond channel-scoping (FD3).** No per-user allowlist this sprint. The residual risk (channel membership *is* the ACL) is documented and a per-user allowlist is recorded as a deliberate future tightening — it is a knowingly-accepted posture, not an oversight.
- **A Slack MCP server / gateway.** Same reasoning as pass 1 FD2, and it now holds *more* firmly: Socket Mode is a first-party `slack_sdk` connection under the app's own tokens; there is no model-influenced *execution* at the transport layer to isolate (the run it launches is already sandboxed downstream exactly as every other run is). `slack_sdk` stays the standard first-party gateway at the library layer.
- **Multi-tenant / multi-channel / multi-workspace operation.** One workspace, one channel (`LOOP_ENGINE_SLACK_CHANNEL`), one bot. Concurrency across runs is serialized (see FD6 / the `_run_lock` note), consistent with `trigger/`'s single-process posture and BL-8.
- **Any change to `core/`'s decision logic, exit codes, or terminal-status semantics, or to the outbound notifier (pass 1).** The inbound path *calls* `runner.run_new`; it does not alter what a run does. `cli.py`'s existing `_EXIT_CODES` map for `run`/`resume` is untouched (the new `slack-listen` command is a long-running daemon with its own trivial exit posture, not a run-status mapping).

**Context (locked decisions — this planning pass, Opus/Architect, owner-confirmed via HITL gates 2026-07-16; do not re-open):**

- **FD1 — Slack Socket Mode *supersedes* the GitHub webhook as the active inbound path** *(locked 2026-07-16, HITL gate 1)*. Of supersede / parallel / wait-behind-BL-24, the owner chose **supersede**. Socket Mode's outbound WebSocket needs neither a tunnel nor a publicly routable address — the exact blocker that parked BL-24 — so it delivers a **live, exercised** inbound path without solving hosting. `trigger/` is parked in place (not deleted; see Out of scope). The connection posture — a long-lived, self-reconnecting outbound WebSocket — is **new to the repo** (every other surface is request/response), but it is an *operational* lifecycle to design, **not** a static-boundary violation: a WebSocket is in-process network I/O, so it adds **no** subprocess surface (the five sanctioned surfaces are unchanged), and `slack_sdk`'s own `SocketModeClient` owns the reconnect/backoff loop.

- **FD2 — Scope is the trigger surface only (BL-2 pass 2); the escalation round-trip is pass 3** *(locked 2026-07-16, HITL gate 2)*. Smallest increment that banks the FD1 supersede win: Slack → *start* a run. The escalation round-trip (pass 3) is bidirectional/stateful and strictly larger (it touches the resolver ladder and the paused-run/`resume` machinery), so it is a separate sprint built on this one.

- **FD3 — Authorization boundary is channel-scoped only** *(locked 2026-07-16, HITL gate 3)*. The daemon acts on a trigger **only** when the event's channel id equals `LOOP_ENGINE_SLACK_CHANNEL` (pass 1's existing env var, reused). Trust is delegated to Slack channel membership. **The outer boundary is already strong:** Slack only ever delivers *our* app's events, from *our* workspace, over a WebSocket authenticated by *our* app+bot tokens — an attacker cannot inject a `slash_commands` event without those credentials. The channel guard is the **inner** boundary that narrows "any channel our app is in" to "the one operating channel." **Knowingly-accepted residual risk (document it, do not silently rely on it):** anyone who can post in that channel can spend money and create GitHub repos — channel membership becomes a security control. A per-user allowlist (`LOOP_ENGINE_SLACK_ALLOWED_USERS`) is the recorded future tightening (would slot into the same guard). **Channel name↔ID footgun:** `LOOP_ENGINE_SLACK_CHANNEL` may hold a `#name` or a `Cxxxx` **ID**; `chat.postMessage` (pass 1) accepts either, but inbound event payloads carry the **ID** only. The guard MUST compare IDs — resolve a configured `#name` to its ID once at daemon startup (a `conversations.list`/lookup via the bot token) and compare against that, or require the ID form and validate at startup. A naive `event["channel_id"] == os.environ[...]` string compare silently fails-open-to-nothing (drops *every* event) if the env var is a name. Pin this in T4.

- **FD4 — Socket Mode's app-level token is a new env var, `LOOP_ENGINE_SLACK_APP_TOKEN` (`xapp-…`), Infisical-sourced like every other secret; the daemon fails closed if it (or the bot token, or the channel) is unset** *(locked 2026-07-16, HITL gate 4)*. Socket Mode needs **two** credentials: the app-level token (`xapp-…`, scope `connections:write`) to *open* the WebSocket, and pass 1's bot token (`xoxb-…`, `LOOP_ENGINE_SLACK_BOT_TOKEN`, scopes `commands` + `chat:write`) to read the command payload and post ephemeral replies. Both are env vars, **not** keyring — same reasoning as pass 1 FD3 and `trigger/`'s `LOOP_ENGINE_WEBHOOK_SECRET`: bot infrastructure that runs headless, far lower-value than the keyring-guarded Anthropic key (which alone can spend on LLM calls), and keeping the **keyring single-importer boundary** (`tools/llm/client.py`) intact. Source of truth = Infisical (`loop-engine`/`dev`), inherited via `infisical run` exactly like the pass-1 Slack vars; **not** bounced through `seed-secrets.sh` (the env var *is* its store). Neither token is ever logged (mirror `trigger/app.py`'s "no secret is ever logged"). Unlike pass 1's notify (which is *inert* by default when unset), the **daemon fails closed** — `loop-engine slack-listen` with any of `LOOP_ENGINE_SLACK_APP_TOKEN` / `LOOP_ENGINE_SLACK_BOT_TOKEN` / `LOOP_ENGINE_SLACK_CHANNEL` unset **raises and refuses to start**, mirroring `create_app`'s `LOOP_ENGINE_WEBHOOK_SECRET` check — an inbound trigger surface must never start half-configured.

- **FD5 — Trigger grammar is a registered `/agent-run` slash command with a REQUIRED budget** *(locked 2026-07-16, HITL gate 5)*. The trigger is `/agent-run --budget <n> <requirements text>`, delivered over Socket Mode as a `slash_commands` event (Socket Mode carries slash commands and interactivity, not just Events-API events). Chosen over an `app_mention` because pass 1's notifier **already posts into `LOOP_ENGINE_SLACK_CHANNEL`**, so the channel must stay usable for chatter and notifications — a registered slash command is explicit, self-documenting (Slack shows the command + usage hint), and never false-fires on ordinary messages or stray mentions. **The budget is required, and a missing/invalid `--budget` is rejected with an ephemeral usage reply — fail-closed on the money cap** (the trigger spends real money; the per-invocation cap must be explicit, never the silent `DEFAULT_BUDGET_USD`). `loop_name` defaults to `"default"`; the requirements text becomes `run_new`'s `human_input`. One-time Slack app-config setup (register the `/agent-run` command, enable Socket Mode, mint the app-level token) is an operator step documented in T6.

- **FD6 — Module layout (owner-reviewable at PR; faithful to `trigger/`'s proven posture)** *(this planning pass)*. Two homes, mirroring how pass 1 split the pure contract from the `slack_sdk` transport:
  - **`tools/slack_io/inbound.py` (new)** — the Socket Mode transport: a thin wrapper over `slack_sdk.socket_mode.SocketModeClient` that opens the WebSocket under the app+bot tokens, registers a request listener, acks each envelope, and exposes `build_listener_from_env()` (fails closed if a token is unset). `slack_sdk` is imported **function-scoped** here, exactly as `notifier.py` does — so `tools/slack_io` keeps its **single-importer** ownership of `slack_sdk` and the boundary test stays green. This is the sole home of the "new connection posture."
  - **`slack_control/` (new top-level package, sibling of `trigger/`/`flows/`)** — the orchestrator-level caller, imports **no** `slack_sdk`:
    - `command.py` — pure parser: a `slash_commands` payload → a validated `SlackRunCommand` (pydantic `extra="forbid"`: `human_input`, `budget_usd` **required**, `channel_id`, and the Socket Mode `envelope_id` for dedupe), or a typed rejection carrying a reason for the ephemeral reply. Mirrors `trigger/parse.py`'s "locked grammar, never raise" posture.
    - `dispatch.py` — `SlackRunDispatcher`: fire-and-forget on a worker thread, in-memory dedupe keyed on `envelope_id` (Socket Mode redelivers unacked events, so idempotency matters), serialized by an `asyncio.Lock`/`_run_lock` because `worktree_run`'s `os.chdir` is process-global (BL-8) — a **direct mirror** of `trigger/dispatch.py`'s `InProcessDispatcher`. Calls `runner.run_new(human_input, budget_usd=…, loop_name="default")`.
    - `daemon.py` — wires transport → **FD3 channel-scope guard** → `command.parse` → on success `dispatch()` + ephemeral "run accepted" reply, on rejection an ephemeral usage error; acks the socket envelope within Slack's ~3 s window regardless; fails closed on unset creds. Outbound replies reuse the existing bot-token `WebClient` path.
  - **Parked `trigger/` is not touched.** `SlackRunDispatcher` deliberately *mirrors* `InProcessDispatcher` (~20 lines) rather than refactoring parked `trigger/` code into a shared seam — keeping the two inbound paths independent is consistent with "supersede, not unify," and avoids coupling live code to a parked module. A future consolidation (a shared dispatcher seam) is a reasonable backlog note, not this sprint's work. `RunRequest` is **not** reused: its `repo_full_name`/`issue_number` are GitHub-webhook-specific and required, and don't map to a Slack trigger — hence the separate `SlackRunCommand`.
  - **CLI:** `loop-engine slack-listen` — a new blocking subcommand that constructs the listener + dispatcher and runs the daemon until interrupted. It is **not** a run-status command; the `run`/`resume` `_EXIT_CODES` map is untouched.

**Security Considerations:** This sprint adds the repo's **first live-exercised inbound trigger** — an untrusted-input surface whose sink is **model-driven code execution + real spend** (the same trust class as the parked GitHub webhook). Diligence items, all keyed to `docs/architecture_definition.md`'s threat model:
1. **No new sink class versus the webhook.** The command's requirements text flows `SlackRunCommand.human_input → runner.run_new → State.artifacts["human_input"] → personas` — the *identical* "untrusted text → `human_input` → model input" flow the webhook already feeds from an issue title/body. There is **no** shell/`gh` arg, filesystem path, or credential sink reached from the Slack text. The `--budget` is parsed to a float and is the **spend bound** per invocation (required — FD5).
2. **The trust boundary is the app+bot token pairing (outer) + the FD3 channel-scope guard (inner).** Slack delivers only our app's events from our workspace over a token-authenticated socket; the guard narrows to the one operating channel. Accepted residual risk: channel membership is the ACL (FD3). Compare channel **IDs**, not names (FD3 footgun).
3. **Fail-closed on start, fail-safe on parse.** The daemon refuses to start if any credential is unset (FD4) — the opposite of pass 1's inert-when-unset, correct for an inbound surface. A malformed/unauthorized command is dropped (channel guard) or answered with an ephemeral usage error (grammar) — never raises, never 500-equivalents, never starts a run.
4. **Tokens never logged**, never placed in a State field, artifact, or message body (mirror `trigger/app.py`). Boundary test asserts `slack_control/` imports no `keyring`.
5. **No new subprocess surface** — a WebSocket is in-process network I/O; `tests/tools/test_subprocess_surfaces.py` stays at **five**. `slack_sdk` stays function-scoped/single-importer in `tools/slack_io`.
This sprint **touches `src/`** (`tools/slack_io/inbound.py`, the whole `slack_control/` package, `cli.py`) ⇒ every such PR needs a fresh-session **`architect-review`**, and the `/critic-gate` pass applies with force here — **security-critic** (untrusted-input taint to a model-execution + spend sink; the new inbound trust boundary) and **architect** (the new top-level caller's boundary posture) are both indicated, matching the pass-1 T3 critic-routing precedent. `slack_sdk` is already a pinned runtime dep (pass 1), so **no** new dependency / `sbom` / `audit` churn this sprint.

**Risks & Blockers:**
- **Fail-closed is the inbound analogue of pass 1's fail-open — get the polarity right.** Notify (pass 1) is inert when unconfigured; the trigger daemon must **refuse to start** when unconfigured (FD4). Test both: `slack-listen` with a missing token raises before opening any socket; a fully-configured daemon opens exactly one connection.
- **Channel guard must compare IDs (FD3).** A name-vs-ID mismatch silently drops every event (fail-closed-to-useless) or, worse, if only partially checked, could widen scope. Resolve/validate the channel at startup and unit-test the guard with both a matching and a foreign `channel_id`.
- **Socket Mode's ~3 s ack window.** The handler must `send_socket_mode_response(envelope_id)` promptly and run the loop on a worker thread (fire-and-forget), exactly as `trigger/` acks 202 then runs in the background. An un-acked envelope is **redelivered** — hence dedupe on `envelope_id` (mirror `InProcessDispatcher`'s in-memory dedupe). Test the redelivery/dedupe path.
- **Process-global `os.chdir` (BL-8) still applies.** `SlackRunDispatcher` must serialize actual loop execution with a lock, same as `InProcessDispatcher`; two concurrent `/agent-run`s must not race their worktree chdirs. A per-instance lock suffices because only one daemon runs (parked `trigger/` never runs concurrently).
- **`src/`-touching ⇒ `architect-review` on each PR, in a FRESH session.** Keep tasks on clearly-scoped PRs (see PR structure below); none is docs-only except T6. Do **not** `/model opus` mid-session to "review" — `/handoff` → new session → `/resume` → `/code-review` → post the verbatim-header review.
- **New top-level package registration.** `slack_control/` must be picked up by the package config (like `trigger/`/`flows/`) and get its own `tests/slack_control/test_boundaries.py` mirroring `tests/trigger/test_boundaries.py` (no `keyring`, no direct file write, no subprocess surface). Confirm the boundary/subprocess-surface tests enumerate the new package.
- **`slack_sdk` not installed in a planning-only env.** This Opus session's interpreter lacks `slack_sdk` (it is a pinned dep but this env is unsynced); the coder session must run under the synced project env. `slack_sdk==3.43.0` ships `slack_sdk.socket_mode.SocketModeClient` — confirm at implementation time.
