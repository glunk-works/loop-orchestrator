### FILEPATH: /sprints/18_execution_isolation_container/sprint_plan.md

**Sprint Goal:** Phase 3b of the MCP/LangGraph migration — the *inert seam* for sandboxed tool execution. Extend `LOOP_ENGINE_ISOLATION` from `none|worktree` to `none|worktree|container|sandbox`, and build (but do not execute) the launch-parameter builders that make untrusted execution (`run_tests` → pytest → model-generated code) run inside a throwaway container/sandbox that mounts **only** the active worktree instead of in the orchestrator process. The `container` path (docker/podman on a daemon-bearing host) is the tested-primary target and reuses the `dev` Dockerfile stage; `sandbox` (bwrap/nsjail, daemon-free) is a secondary inert builder. Everything is wired, preflighted, and argv-shape-tested; **nothing untrusted runs on this branch** — the real `docker run` end-to-end path and its parity test are explicitly deferred to a daemon-bearing host (they cannot execute in this devcontainer: no runtime present, `unshare -Ur` returns EPERM). The load-bearing invariant this sprint enforces: **selecting `container`/`sandbox` must NEVER silently degrade to running untrusted code in-process — it fails honestly at every untrusted-execution surface until the deferred host-side build replaces those guards with real sandbox execution.**

**Dependencies:** Phase 3a (`951e377`) — `tools/worktree/` and `worktree_run` (this builds on `LOOP_ENGINE_ISOLATION` and reuses the worktree the container/sandbox mounts). Phase 2 (`7368411`) — the MCP provider seam (`tools/mcp/provider.py`, `coder_tools_server_params`, `build_coder_tool_provider`, `MCPToolProvider`) that this sprint extends. **Read `docs/phase3_execution_isolation_plan.md` (Phase 3b section) and `docs/migration_roadmap.md` first** for the locked decisions (D1–D3) and the honest scoping note. No new third-party dependency (no docker/podman/bwrap Python SDK — launches go through the existing `mcp` stdio client), so `sbom.json` and `hatch run audit` are untouched.

**Security Considerations:**
- **This is the actual security boundary of Phase 3** (3a was blast-radius only). The whole point is that untrusted, model-generated code stops executing in the orchestrator process. Two orchestrator-side surfaces execute model code today and BOTH must be accounted for, or the boundary is a lie: (1) the Coder's `run_tests` tool (via the MCP server), and (2) the **Coder gate's own deterministic pytest** at `core/coder_gate.py:69` (`run_tests_tool.run_pytest`). This sprint sandboxes (1)'s launch params and **hard-guards (2)** — under `container`/`sandbox` the gate must refuse to run pytest in-process and raise clearly, because sandboxing the gate's pytest is part of the deferred host-side e2e task. A container-mode run that silently let the gate execute model tests in-process would defeat the entire phase.
- **No new sanctioned subprocess surface.** The container/sandbox is spawned by the existing `mcp.client.stdio.stdio_client(params)` (same mechanism that launches today's `python -m ...coder_tools_server`) — we only change the `command`/`args` in the `StdioServerParameters`. We do **not** add a `subprocess.run`/`Popen` call for docker anywhere, so the "exactly three sanctioned subprocess surfaces" rule in CLAUDE.md holds and no new `# noqa: S603/S607` appears. Runtime detection uses `shutil.which` (a PATH lookup, not a subprocess, not a file write).
- **Boundary tests must stay green:** `tools/mcp/` still imports no `keyring` (`tests/tools/test_keyring_boundary.py`) and writes no files (`tests/tools/test_state_io_boundary.py` — `shutil.which` and building a `StdioServerParameters` are neither `open`/`write_text`/`write_bytes`). Do not introduce file I/O in the provider.
- **Mount surface is the security contract.** The container/sandbox mounts **only** the worktree (`-v {worktree}:{worktree}:rw`), `--network none`, `--read-only` rootfs + `--tmpfs /tmp`, `--cap-drop ALL`, `--security-opt no-new-privileges`, non-root `--user`. No repo root, no `state/`, no keyring, no host paths, no network. A test asserts the argv contains exactly one bind and it is the worktree — a regression that mounts anything else is a security regression, not a cosmetic one.
- **`container` requires `LOOP_ENGINE_TOOLS=mcp`.** Sandboxing only exists on the MCP tool path (it *is* the server launch). On the in-process tool path there is no server to sandbox, so `container`/`sandbox` + in-process tools is a misconfiguration that would run untrusted code in-process. It must raise at selection time, never be silently honored.

**Risks & Blockers:**
- **Nothing here can be executed or e2e-verified in this devcontainer** — there is no docker/podman/bwrap and unprivileged user namespaces are denied (`unshare -Ur` → EPERM). Verification on this branch is therefore limited to **deterministic argv-shape unit tests and the honest-failure guards**; do not write any test that spawns a container or an MCP server in a container. The commit is green because the untrusted paths are guarded to raise, not because they were exercised. This is intended (locked decision D3).
- **DooD host-path footgun (why `container` targets a native-daemon host).** `-v {worktree}:{worktree}` requires `{worktree}` to be a path the docker *daemon* can see. Inside a nested/devcontainer talking to a mounted host socket, the host and container disagree on that path and the bind silently mounts the wrong thing or nothing. The `container` path is specced for the Phase 5 factory host with a **native** daemon, where worktree paths are real host paths. Do not attempt to make it work via a mounted host socket here.
- **`docker run -i`, not `-t`.** The MCP stdio transport speaks JSON-RPC over the child's stdin/stdout pipes that `stdio_client` creates. The container must keep stdin attached (`-i`) and must NOT allocate a TTY (`-t` would corrupt the framing). Getting this wrong is the most likely e2e failure when the deferred host-side task runs — it is called out in that task's notes, not tested here.
- **Bind-mount file ownership.** pytest writes caches (`.pytest_cache`, `__pycache__`) into the worktree. If the container runs as root, those land root-owned on the host worktree and later in-process writes fail. Hence `--user {uid}:{gid}` defaulting to the orchestrator's own uid/gid. This is a real e2e hazard, deferred with the e2e task, but the argv builder must emit `--user` now so the seam is correct.
- **Do not weaken the 3a guarantees.** `container`/`sandbox` still imply a worktree (the thing being mounted) and still keep State snapshots in the main checkout. `worktree_needed()` must be true for all three isolation modes; the CLI's `worktree_run` wrapping is unchanged.

- **Task 2: `container_server_params(worktree)` + runtime preflight (docker/podman — primary, inert)**
  - **Description:** In `src/loop_engine/tools/mcp/provider.py` add `IsolationUnavailableError(Exception)` and `container_server_params(worktree: str | Path) -> StdioServerParameters`. It must: (a) resolve the runtime via `_container_runtime()` = `LOOP_ENGINE_CONTAINER_RUNTIME` (must be `docker` or `podman` if set) else `shutil.which("docker") or shutil.which("podman")`; **raise `IsolationUnavailableError` if none found** (clear message: container isolation selected but no docker/podman on PATH — never fall back); (b) require `LOOP_ENGINE_DEV_IMAGE` (the `dev` stage image tag) and **raise `IsolationUnavailableError` if unset**; (c) resolve `--user` from `LOOP_ENGINE_SANDBOX_UID` else `f"{os.getuid()}:{os.getgid()}"`. Build the argv exactly:
    ```
    [runtime, "run", "--rm", "-i",
     "--network", "none",
     "--read-only", "--tmpfs", "/tmp",
     "--cap-drop", "ALL",
     "--security-opt", "no-new-privileges",
     "--user", user,
     "-v", f"{wt}:{wt}:rw", "-w", wt,
     image,
     "python", "-m", _CODER_TOOLS_SERVER_MODULE]
    ```
    where `wt = str(Path(worktree).resolve())`. Return `StdioServerParameters(command=runtime, args=[...], cwd=None)` (the container's `-w` sets the working dir; the host cwd is irrelevant). Note in the docstring: `-i` (no `-t`) is required for the stdio JSON-RPC framing, and the single bind is the whole security contract. **Do not** call `subprocess`/`docker` here — only `shutil.which` for detection.
  - **Target Files:** `src/loop_engine/tools/mcp/provider.py`, `tests/tools/test_mcp_provider.py`
  - **Acceptance Criteria:** With runtime + `LOOP_ENGINE_DEV_IMAGE` monkeypatched present, `container_server_params("/wt/run1")` returns the exact argv above (assert the full list) with `command` = the runtime. A test asserts the argv contains **exactly one `-v`** and it is `"{wt}:{wt}:rw"` (no other bind, no repo root, no `state/`). Missing runtime → `IsolationUnavailableError`; missing `LOOP_ENGINE_DEV_IMAGE` → `IsolationUnavailableError`. `--network none`, `--cap-drop ALL`, `--read-only`, `--user` all present. No `subprocess.run`/`Popen` added to the module (the file-write and subprocess boundary tests stay green; no new `# noqa`).

  - **Target Files:** `src/loop_engine/tools/mcp/provider.py`, `src/loop_engine/personas/coder_iac/persona.py`, `tests/tools/test_mcp_provider.py`, `tests/personas/coder_iac/` (the persona-backend test module)
