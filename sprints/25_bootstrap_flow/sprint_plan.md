### FILEPATH: /sprints/25_bootstrap_flow/sprint_plan.md

**Sprint Goal:** Build **Phase 5 piece 4 — the bootstrap flow**: the factory verb that **brings a new, conventions-conformant `glunk-works` repo into existence**. `flows/bootstrap` creates a fresh private repo, clones the (empty) tree, **scaffolds a minimal Python skeleton into it from bundled templates**, **injects the Global Conventions as the managed repo's `CLAUDE.md`**, commits the scaffold as the repo's first commit, pushes it to `main`, and creates a `develop` integration branch off `main` — leaving a repo the **maintenance flow (24) can immediately PR into**. This is a **capability slice** (mirroring 22b/23/24): it lands a new **`tools/scaffold`** module (the system's **second sanctioned file-write surface** — carved out as its own named/tested module exactly as `tools/git_io` was carved out as the fourth subprocess surface in 24), a new top-level **`src/loop_engine/flows/bootstrap/`** package behind injectable collaborators, bundled skeleton templates + a byte-identical conventions sync-guard, hermetic tests, boundary-test coverage, and docs. It **does NOT run the inner default loop** (no `run_in_tree`, no LLM run, no green gate — bootstrap makes a *ready-to-work* repo; the maintenance flow / trigger does the feature work), does **NOT** open a PR (`create_repository` and `push` are the terminal GitHub ops; there is nothing to review into on a brand-new repo — **auto-merge stays impossible**), does **NOT** add a CLI subcommand, does **NOT** wire the trigger surface, and ships **Python only** (the IaC/OpenTofu template set is deferred behind the `kind` parameter). Live create→clone→push verification is deferred to a daemon-bearing host.

**Dependencies:** Sprint 22b (`7b46227`→`5bc3811`) landed `tools/repo_io`'s four factory verbs — `create_repository`, `clone_repo`, `create_branch`, `open_pr` — the remote-GitHub foundation this flow chains (`bootstrap` uses the first three; not `open_pr`). Sprint 24 (`6172ad1`→`f8d388a`) landed `tools/git_io` (`checkout_branch`/`commit_all`/`push_branch` over a foreign clone) and the `src/loop_engine/flows/` top-level package with its boundary static test — `bootstrap` is a **sibling of `flows/maintenance`** and reuses `git_io` verbatim. Precedents to mirror exactly: `src/loop_engine/flows/maintenance/flow.py` (the injectable-collaborators flow shape, `Pydantic extra="forbid"` request, `Enum` status result); `tools/repo_io/github.py::_validate_clone_dest` (the tree-path validation `tools/scaffold` reuses on every write, just as `git_io` does); `tests/flows/test_boundaries.py` (the no-keyring / no-direct-write / no-own-subprocess-surface `flows/` boundary test, which already rglobs `flows/` and so auto-covers `bootstrap`); `tests/tools/test_state_io_boundary.py` (the file-write invariant this sprint **widens** from `state_io`-only to `{state_io, scaffold}`); `tests/tools/test_subprocess_surfaces.py` (the four-surface invariant this sprint must keep at **four** — `scaffold` writes files, it does **not** shell out). **Read the "Context (locked decisions)" section before starting** — the Architect (Opus) settled all gating decisions in the 25 planning pass; do not re-open them.

**Context (locked decisions — do not re-open):**
- **Deliverable = a skeleton-only capability slice; NO inner loop run** *(locked 2026-07-10, Opus/Architect, 25 planning pass)*. Bootstrap's job is to make a conventional, ready-to-work repo *exist* — `create_repository` → `clone_repo` (empty tree) → scaffold skeleton files → inject conventions `CLAUDE.md` → `commit_all` → `push` → create `develop`. It does **NOT** dispatch `runner.run_in_tree`, does **NOT** run an LLM/default-loop pass, and has **no LLM-driven green gate**. The actual feature work on the new repo is a *later* maintenance-flow (24) or trigger dispatch. Rationale (do not re-litigate): clean separation — "create the repo" and "build in the repo" are different responsibilities; keeping bootstrap loop-free keeps the slice minimal and its cost model trivial (no `LLMClient`, no budget).
- **Scaffold mechanism = a NEW dedicated `tools/scaffold` module; this is honestly a SECOND sanctioned FILE-WRITE surface — NOT a fifth subprocess surface** *(locked 2026-07-10, Opus/Architect)*. Skeleton files are written into the clone from **bundled in-repo template assets** via `write_text`; the module reuses `tools/repo_io/github.py::_validate_clone_dest` to validate the tree on every write. We deliberately do **NOT** shell out to `hatch new` (that would add a *fifth* subprocess surface, couple the skeleton to hatch's output, and be inconsistent with IaC which has no generator). It lives in its **own** module — NOT bolted onto `state_io` (which owns *state/artifact* persistence keyed to the run's state root; a foreign clone's skeleton is a different responsibility) and NOT written inline in `flows/` (the `flows/` boundary test forbids direct file writes there). The "`state_io` is the only module with direct file-write calls" invariant becomes **`state_io` and `scaffold` are the file-write-owning modules** — update `CLAUDE.md` and `tests/tools/test_state_io_boundary.py` to say so truthfully. Rationale (do not re-litigate): the invariant exists to make file-write surfaces conspicuous and reviewed; a clean, named, tested `tools/scaffold` serves that better than hiding writes inside a flow or overloading `state_io`. **This is the exact precedent-mirror of 24's `git_io` carve-out — a new surface gets its own module, its own tests, and an honest count bump, not a stretched existing one.**
- **Templates are bundled package data; the injected `CLAUDE.md` is byte-identical to `.ai/context/conventions.md`, enforced by a sync-guard test** *(locked 2026-07-10, Opus/Architect)*. The skeleton templates (pyproject/src/tests/README/.gitignore) and the conventions `CLAUDE.md` live under `src/loop_engine/tools/scaffold/templates/` as package data, accessed via `importlib.resources` so they resolve in an installed wheel (no runtime dependency on the dev-workflow `.ai/` layer). The conventions template is the **portable Global Conventions block**; a test asserts it stays **byte-identical** to `.ai/context/conventions.md` so the managed-repo copy and the dev-doc never drift (drift fails CI). `.ai/context/conventions.md` is the human-facing doc; the bundled copy is the runtime source — one logical source, parity-enforced.
- **Repo type = Python-only this sprint; `kind`-parametrized so IaC drops in later** *(locked 2026-07-10, Opus/Architect)*. Ship exactly one template set — `kind="python"` (a minimal `src/`-layout repo: `pyproject.toml`, `src/<pkg>/__init__.py`, a trivial passing `tests/` smoke test, `README.md`, `.gitignore`) plus the injected `CLAUDE.md`. The writer takes a `kind` argument and resolves `templates/<kind>/`, so a second set (`kind="iac"`: OpenTofu `main.tf`/`variables.tf`/`outputs.tf`/`versions.tf` with pinned providers + a `.tfstate`-excluding `.gitignore`) drops in later **with no flow change**. IaC is deferred (below).
- **GitHub end-state = push `main` (first commit) → create `develop`; NO PR** *(locked 2026-07-10, Opus/Architect)*. A brand-new repo has no commits and no branches, so there is nothing to open a PR *into* — the scaffold is committed as the first commit, pushed to `main` (which sets the repo's default branch), and then `repo_io.create_branch(owner, repo, "develop", base="main")` establishes the integration branch (a remote ref via `gh`, no new surface). **No `open_pr`.** This satisfies the roadmap's "commit/push" wording and its explicit note that **branch-topology setup is the bootstrap flow's concern** (deferred from 24). The repo is left with `main` + `develop` so a subsequent maintenance run's `base="develop"` resolves immediately. Auto-merge stays impossible — no merge verb exists and bootstrap never even opens a PR.
- **Placement = new `src/loop_engine/flows/bootstrap/` package, sibling of `flows/maintenance`; orchestrator-level caller** *(locked 2026-07-10, Opus/Architect)*. Like `flows/maintenance` it may import `core`/`State`/`repo_io`/`git_io`/`scaffold`; it is **not** a `tool` and not an MCP server. Its enforced boundary posture, asserted by the existing `tests/flows/test_boundaries.py` (which rglobs `flows/`): imports no `keyring`, writes no files directly (all skeleton writes go through `tools/scaffold`; all git writes through `tools/git_io`), and introduces **no subprocess surface of its own**. Unlike `flows/maintenance`, `bootstrap` introduces **no new subprocess surface at all** — its one genuinely new surface, `tools/scaffold`, is a **file-write** surface, not a subprocess one, so the four sanctioned subprocess surfaces are **unchanged**.

**Security Considerations:** The bootstrap flow **writes model-/caller-influenced files into a cloned tree and then pushes them to a newly created GitHub repo**, so three boundaries are load-bearing. (1) **Tree confinement:** `clone_repo`'s destination and every `tools/scaffold` write target are validated with `repo_io._validate_clone_dest` (relative, no `..`, no symlink escape out of the run tree); the template *file list* is first-party bundled data (not caller-controlled), so only the `tree` root and the derived package name are untrusted inputs. (2) **Derived package-name sanitization:** the Python package directory name is derived from the caller-supplied repo `name`; it MUST be sanitized to a safe Python identifier (e.g. reject/replace anything outside `[A-Za-z0-9_]`, reject a leading digit) so a crafted `name` can never inject a `../` path segment or a non-identifier into the written tree — cover this with a negative test. (3) **Push is non-merging, no PR:** the terminal GitHub ops are `push` (of the initial commit) and `create_branch` (a `develop` ref); there is no merge verb and no `open_pr`, so bootstrap can never merge anything. No credential enters `flows/` or `tools/scaffold` (no `keyring` import); `git push` rides `gh`'s clone-established credential helper. `BootstrapRequest` is a Pydantic model with `extra="forbid"` and has an invalid-input-rejection test (DoD). `tools/scaffold` shells out to **nothing** (it is a file-write surface only — asserted by `test_subprocess_surfaces.py` continuing to pass with four surfaces). No secret appears in a source file, template, config, or test fixture (tests use obviously-fake slugs/paths and a local temp git repo, so `gitleaks` stays clean); templates contain no credentials by construction.

**Risks & Blockers:**
- **The second file-write surface is a real boundary change — the doc and the test must move together.** `CLAUDE.md`'s "`tools/state_io` is the only module with direct file-write calls" clause and `tests/tools/test_state_io_boundary.py` (`ALLOWED_DIR = tools/state_io`, `test_only_state_io_writes_files`) both currently assert **state_io-only**. Both must move to **`{state_io, scaffold}`** in the same sprint, naming `tools/scaffold`. Do not leave the doc and the test disagreeing. (Mirror exactly how 24 moved the subprocess count three→four in lockstep.)
- **The four sanctioned subprocess surfaces must stay four.** `tools/scaffold` writes files; it must import **no** `subprocess`/`Popen`/`os.system`/`os.exec*`. `tests/tools/test_subprocess_surfaces.py` already forbids subprocess outside its sanctioned set, so it **auto-guards** this — but confirm `scaffold` is NOT added to `_SANCTIONED_SUBPROCESS_MODULES` and the four-count assertion/`CLAUDE.md` "exactly four" clause are untouched. If scaffold ever needs a subprocess, that is a *different* sprint's decision.
- **Bundled non-`.py` templates must actually ship in the wheel.** The wheel target is `packages = ["src/loop_engine"]` with no explicit package-data config. Verify that `importlib.resources.files("loop_engine.tools.scaffold").joinpath("templates")` resolves both in-tree (editable) **and** from a built wheel; if hatchling does not include the non-`.py` template files by default, add the minimal `[tool.hatch.build.targets.wheel.force-include]` (or `artifacts`) entry so they are packaged. A `.tmpl` extension keeps templates from being mistaken for importable modules / collected by pytest. **No new runtime dependency** — `importlib.resources` + `string.Template`/token-replace are stdlib; **`sbom.json` unchanged**.
- **Empty-clone branch mechanics.** Cloning a freshly `gh repo create`d repo yields an **empty tree on an unborn branch** whose name follows the cloner's `init.defaultBranch` (may be `master`, not `main`). To make the pushed default branch deterministic, `checkout_branch(dest, "main")` (via `git_io`, `git checkout -b main` — succeeds on an unborn HEAD) **before** `commit_all`, then `push_branch(dest, "main")`. Only **after** the push does `main` exist remotely, so `create_branch(owner, repo, "develop", base="main")` must come **after** the push (ordering is load-bearing — `create_branch` reads the base ref's SHA over the API). Cover the ordering in the flow test.
- **`scaffold` tests are hermetic against a `tmp_path` tree; flow tests fake `repo_io`.** Render + write the skeleton into a `tmp_path` dir and assert the file set/contents directly; the flow-level end-to-end proof uses a **real** `tools/scaffold` + **real** `tools/git_io` against a `tmp_path` `git init` repo with a local bare remote, with `repo_io` **faked** (`create_repository`→a canned `RepoRef`, `clone_repo`→the pre-made `tmp_path` tree, `create_branch`→a recorded call) so no real GitHub/network runs in CI.
- **No live end-to-end verification on this branch.** No host here can `gh repo create` in `glunk-works`, clone it, push an initial commit, and create `develop` with real `gh` auth + network. Coverage is hermetic only. Append the live check — real `create_repository` → real clone → scaffold → real push `main` → real `create_branch develop` — to `sprints/DEFERRED_VERIFICATION.md`. (`glunk-works` org access is still an open hosting question, deferred with the live check.)
- **Do not scope-creep.** No inner loop run / `run_in_tree` call, no green gate, no `open_pr`, no CLI subcommand, no trigger wiring, no IaC template set, no new dependency (`scaffold` adds none — **`sbom.json` unchanged**). The carried-forward `python` vs `sys.executable` `loop_engine.mcp.json` nit and the Sprint-24 `_default_run_tests` coverage nit are both orthogonal to bootstrap — leave them (carry-forward, below).
