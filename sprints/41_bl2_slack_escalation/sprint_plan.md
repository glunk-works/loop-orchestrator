### FILEPATH: /sprints/41_bl2_slack_escalation/sprint_plan.md

**Sprint Goal:** Land the **third and final surface** of BL-2's Slack "bot control plane" — the **escalation round-trip**: when a run pauses on questions no automated layer could resolve, post them to the configured Slack channel, let a human answer **in the message's thread**, and fold that reply back into the run exactly as `resume --from-issue` folds a GitHub-issue answer. This closes BL-2 and gives the resolver ladder a Slack terminus **as an alternative to** the GitHub-issue round-trip (selected per-deployment; the issue path stays the default). The deliverable is: a **transport-agnostic escalation seam** in `core/` (an `EscalationFiler` returning a union ref, the issue filer unchanged behind it); a **Slack escalation filer** in `tools/slack_io` that posts the questions and records the thread as the correlation key; a **new `State.pending_slack` field** (schema bump v4→v5); a **shared resume-execution seam** in `runner.py` that both `cli.resume` and the Slack path call (one fold-back implementation, not two); and a **`slack_control/` daemon extension** that receives thread replies over the existing Socket Mode connection, correlates each to a paused run by an **on-demand snapshot scan**, resumes it **in-process** under the pass-2 `_run_lock`, and posts the outcome back into the thread. This is BL-2 **pass 3 of 3**; with it, BL-2 is complete.

**Out of scope:**
- **Dual-channel escalation (Slack *and* issue for the same pause).** The transport is **exclusive per run** (HITL gate 3): an env selector picks exactly one. Posting to both would put two correlation refs on one snapshot and force the resume path to reconcile "answered on Slack vs. answered on the issue" (a race + double-fold guard) for marginal benefit. A `both` mode is a recorded future option, not this sprint.
- **Slack escalation for `flows/maintenance` / foreign-clone runs.** Pass 3 covers the orchestrator's **own** worktree-based runs, whose snapshots live under the orchestrator's `state/` tree the daemon can scan. A `flows/maintenance` run's snapshots live inside the *foreign clone* the daemon has no index of; wiring the correlation scan across foreign clones is a separate concern (mirrors pass 2's single-workspace scoping). The issue path remains the escalation route for those runs.
- **A per-user authorization allowlist.** Same posture as pass 2 FD3: the FD3 **channel-scope guard** is the boundary; channel membership is the ACL. A thread reply that resumes a run has the *same* trust class as a `/agent-run` that starts one (both reach a model-execution + spend sink), so no new authorization control is added here. `LOOP_ENGINE_SLACK_ALLOWED_USERS` remains the recorded future tightening, and would slot into the same guard for both surfaces.
- **A broader terminal-status / exit-code redesign.** Pass 3 adds **exactly one** new terminal status, `AWAITING_SLACK`, and **one** new exit code for it (FD5, owner-directed) — a distinct Slack pause, not a generalized rename of `AWAITING_ISSUE`. The existing `AWAITING_ISSUE` status, its exit code 2, and every other status/code are **unchanged**; the wire value `awaiting_issue` is frozen (renaming it is a migration cost with zero benefit). No other status or exit-code semantics are touched.
- **Retiring `trigger/` / closing BL-24.** Unchanged from pass 2's Out of scope — `trigger/` stays parked; BL-24 stays an owner decision (retire-or-verify). Pass 3 shares no code with `trigger/`.
- **Any change to what a run *does*, to the resolver-ladder decision logic, or to `reentry_index`/`fold_answers` themselves.** Pass 3 adds a *transport* for the escalation's two ends and a *shared execution seam*; the blast-radius classification, the PM answer-folding, and the re-entry math are reused verbatim from the issue path.

**Context (locked decisions — this planning pass, Opus/Architect, owner-confirmed via HITL gates 2026-07-17; do not re-open):**

- **FD1 — Correlation is the Slack message thread (`thread_ts`)** *(locked 2026-07-17, HITL gate 1)*. Of thread-reply / slash-command-with-run-id / post-only, the owner chose **thread reply**. The escalation filer posts the questions as a channel message; that message's `ts` becomes the correlation key, recorded on the snapshot. The human answers by **replying in that thread**, and the daemon ties the reply back by matching its `thread_ts`. This is the Slack-native UX that justifies the feature over the already-working issue path — but it is the most surface-heavy option: it requires the daemon to receive `message` events (a second Socket Mode event type beyond pass 2's `slash_commands`) and to correlate a reply to a paused run it **may not have started**.

- **FD2 — Correlation is resolved by an on-demand snapshot scan, and the run is resumed in-process** *(locked 2026-07-17, HITL gate 2)*. Of on-demand-scan+in-process-resume / in-memory-tracked-threads / listener-shells-CLI, the owner chose **on-demand scan + in-process resume**. A thread reply lands on the live daemon, which scans the `state/` snapshot tree for the one snapshot whose `pending_slack.message_ts` matches the reply's `thread_ts`, loads it, and resumes it **in-process** via the pass-2 dispatcher under the existing `_run_lock`. This preserves the issue path's stateless "read when the human acts" property — there is **no** in-memory index to lose on a daemon restart, and a run paused by the CLI or a prior daemon instance is still answerable — at the cost of a new capability for `slack_control/`: it now **reads the run-state tree** and **resumes** runs (pass 2 only *started* them). The in-memory-index option was rejected because it cannot answer runs the daemon did not itself start; the shell-out option was rejected because it either adds a sixth subprocess surface (breaking `slack_control/`'s boundary) or collapses into this one with an extra CLI seam to maintain.

- **FD3 — Transport is selected by an exclusive env var, defaulting to the issue path** *(locked 2026-07-17, HITL gate 3)*. Of env-selector-exclusive / env-selector-both / per-loop-config, the owner chose **env selector, exclusive, default = issue**. A new env var **`LOOP_ENGINE_ESCALATION_TRANSPORT`** (`issue` | `slack`; unset ⇒ `issue`) picks **exactly one** transport per run. Default `issue` means **zero behavior change** until a deployment opts in — the issue round-trip stays the out-of-the-box escalation path. This mirrors the repo's env-var config convention (every `LOOP_ENGINE_SLACK_*` var, `LOOP_ENGINE_WEBHOOK_SECRET`) and the `build_notifier_from_env()` seam, and keeps `core/` transport-agnostic exactly as it already is for the notifier and the `IssueFiler`.

- **FD4 — `core/` learns a transport-agnostic `EscalationFiler` seam; the issue filer moves behind it unchanged** *(this planning pass)*. The engine's pause path (`_pause_for_issue` in `core/engine.py`) generalizes to `_pause_for_escalation`, calling a single `EscalationFiler` protocol `(state, questions, snapshot_hint) -> EscalationRef`, where `EscalationRef = IssueRef | SlackRef`. `core/` depends only on the **protocol** and the **union type in `State`** — never on `tools/slack_io` or `tools/issue_io` concretely — mirroring how it already depends on the `Notifier` shape and the `IssueFiler` shape, not their transports. The concrete filer is chosen at the wiring layer (`runner`/`graph_engine`, from the FD3 env var), via a `build_escalation_filer_from_env()` that returns the issue filer by default and the Slack filer when `LOOP_ENGINE_ESCALATION_TRANSPORT=slack`. `default_issue_filer` is unchanged; it simply becomes the default implementation *behind* the new seam. The persisted question-ordering contract (`unresolved_questions`) is shared by both transports verbatim.

- **FD5 — `State.pending_slack: SlackRef | None`; schema bump v4→v5; a distinct `AWAITING_SLACK` terminal status** *(owner-directed 2026-07-17; overrides the drafted status-reuse proposal)*. A new optional field `pending_slack` sits alongside `pending_issue` (both `None` on a running or non-escalated run; exactly one set on a paused run — FD3 exclusivity). `SlackRef` is a pydantic model (`extra="forbid"`) carrying `channel_id` and `message_ts` (the posted escalation message's `ts`, which is the `thread_ts` replies will carry). `CURRENT_SCHEMA_VERSION` bumps **4→5**; `migrate_state_payload` extends so a v4 snapshot upgrades by defaulting `pending_slack` to `None` (a pure additive field — no data to move, `extra="forbid"` preserved). A Slack-escalated pause is its **own terminal status `AWAITING_SLACK`** (owner chose separation over reuse), which ripples deliberately and minimally: `RunStatus.AWAITING_SLACK` (T1), a matching `EventKind.AWAITING_SLACK` + a `_TERMINAL_EVENT_KINDS` entry (T2), and a **new distinct CLI exit code** for it (T2) — chosen distinct from `AWAITING_ISSUE`'s code 2 because the resume mechanism differs (a Slack pause is resumed by the *daemon* via a thread reply, **not** by `resume --from-issue`), so a wrapping script must be able to tell the two paused states apart. Proposed value: **exit code 5** (next free after 0/1/2/3/4); the CLI's paused-run message for `AWAITING_SLACK` points at the Slack thread, not the issue-resume command. The paused snapshot's filename uses the status value (`…_awaiting_slack.json`), so the FD2 correlation scan filters precisely on `status == AWAITING_SLACK`.

- **FD6 — Module layout (owner-reviewable at PR; faithful to the pass-2 posture and the issue-path precedent)** *(this planning pass)*:
  - **`core/state.py`** — `SlackRef` + `pending_slack` + `RunStatus.AWAITING_SLACK` + the schema bump/migration (FD5). The only `core/` schema change.
  - **`core/engine.py`** — `_pause_for_issue` → `_pause_for_escalation` calling the FD4 `EscalationFiler` protocol; the `IssueFiler` type generalizes to `EscalationFiler` returning the union. The finalized terminal status is chosen by the ref type (`SlackRef` ⇒ `AWAITING_SLACK` + `pending_slack`; `IssueRef` ⇒ `AWAITING_ISSUE` + `pending_issue`), and the snapshot-hint filename follows the chosen status value. No new imports of a concrete transport into `core/`.
  - **`core/notify.py` / `core/graph_engine.py`** — add `EventKind.AWAITING_SLACK` and its `_TERMINAL_EVENT_KINDS` entry so a Slack pause emits its own lifecycle event (the pass-1 notifier fires on it like any other terminal kind).
  - **`cli.py`** — add the `AWAITING_SLACK` → exit-code-5 mapping and a paused-run message that points at the Slack thread (not `resume --from-issue`); `AWAITING_ISSUE`/code 2 and all other codes unchanged.
  - **`tools/slack_io/escalation.py` (new)** — the **Slack escalation filer** (outbound): posts the questions as a channel message via the pass-1 bot-token `WebClient`, using the same numbered rendering the issue body uses, and returns `SlackRef(channel_id, message_ts=<posted ts>)`. `slack_sdk` stays **function-scoped** here — `tools/slack_io` remains the sole importer, both directions. A pure `render_question_message(state, questions)` mirrors `issue_io.render_question_issue`.
  - **`tools/slack_io/inbound_parse.py` (new, pure) or extend `command.py`** — a pure `parse_thread_answers(text)` that reads the human's thread reply into `{question_number: answer}`, reusing the issue path's numbered `N: answer` convention; for a single unresolved question a bare message body is the answer. No I/O, no `slack_sdk`.
  - **`runner.py`** — a new **shared resume-execution seam** `resume_run(state, resolved_answers, *, budget_usd, ...)` that folds answers via the stage-0 PM `fold_answers`, computes `reentry_index`, seeds carried findings, and calls `run_graph_loop(..., resuming=True, start_index=…)` inside `worktree_run(run_id, reuse=True)`. **`cli.resume` is refactored to call it** so there is exactly **one** resume-execution implementation shared by the issue and Slack paths (the CLI keeps its issue-specific *reading*/verification; only the post-parse *execution* moves into `runner`).
  - **`tools/state_io` (reader seam)** — `find_paused_snapshot_by_slack_thread(thread_ts) -> Path | None`: the on-demand FD2 scan, living in the **state-owning** module so `slack_control/` performs **no raw file I/O** and its boundary test (writes no files directly) stays honest. Reads only; the write-owner invariant is untouched.
  - **`slack_control/dispatch.py`** — `SlackRunDispatcher` gains a `dispatch_resume(...)` sibling to `dispatch(...)`: same in-memory `envelope_id` dedupe, same `_run_lock` serialization (a start and a resume must not race their process-global `os.chdir` — BL-8), calling `runner.resume_run(...)` on the worker thread. A raising resume is logged and swallowed (a bad answer must not kill the daemon).
  - **`slack_control/daemon.py`** — `_handle_request` extends to also handle `request.type == "events_api"` carrying a `message` event: apply the **FD3 channel-scope guard**, ignore the bot's own messages and non-thread messages, then on a thread reply call `state_io.find_paused_snapshot_by_slack_thread(thread_ts)`; if a paused snapshot is found, `parse_thread_answers` → `dispatcher.dispatch_resume(...)` and post an ephemeral/threaded acknowledgement; if none is found, drop silently (an ordinary channel/thread message). The pass-2 `slash_commands` path is unchanged.
  - **`slack_control/` imports no `slack_sdk` and no `keyring`** — all Slack I/O goes through `tools/slack_io` helpers (the daemon's ephemeral replies via `send_ephemeral_reply`, the threaded outcome/hint posts via the new `send_thread_message`; the escalation *post* is done by the filer inside `tools/slack_io`, invoked by the engine, not by `slack_control/`). It now **reads** the state tree via `tools/state_io` (not via raw `open`) and **resumes** runs via `runner`.
  - **No new CLI subcommand.** The daemon (`loop-engine slack-listen`) already runs the listener; pass 3 only widens what its handler accepts. `run`/`resume` exit codes are untouched.

**Security Considerations:** Pass 3 adds a **second inbound trust surface** — a thread reply that **resumes** a run — of the **same trust class** as pass 2's `/agent-run` (both reach model-driven execution + real spend). Diligence items, keyed to `docs/architecture_definition.md`'s threat model:
1. **No new sink class.** A thread reply's text flows `parse_thread_answers → resolved_answers → PM.fold_answers → State.questions[].resolution → personas` — the *identical* "untrusted text → answer → model input" flow the issue path already feeds from an issue comment. There is **no** shell/`gh` arg, filesystem path, or credential sink reached from the reply text. The resume runs under the same per-run `--budget` cap the original run carried.
2. **The correlation scan reads only, and only within the state tree.** `find_paused_snapshot_by_slack_thread` matches an exact `pending_slack.message_ts` and returns a path under `state/`; it opens snapshots for **read** (the state-owning module keeps the write monopoly). A reply whose `thread_ts` matches no paused snapshot is dropped — the daemon never resumes on an unmatched or already-resumed thread (after resume the snapshot is no longer `AWAITING_SLACK`, so a redelivered or duplicate reply finds nothing).
3. **Same boundary as pass 2 (outer app+bot tokens, inner FD3 channel guard), applied to `message` events too.** The `message`-event path enforces the **same** channel-ID guard as the slash-command path (compare IDs, resolved once at startup — the FD3 footgun still applies), and additionally ignores the bot's own messages (no self-trigger loop) and non-thread messages. Accepted residual risk unchanged: channel membership is the ACL.
4. **Idempotency on redelivery.** Socket Mode redelivers unacked `message` envelopes; `dispatch_resume` dedupes on `envelope_id` exactly as `dispatch` does, and the snapshot-state check in (2) is the backstop. Two replies to the same thread cannot double-resume.
5. **Tokens never logged**, never placed in a `State` field, artifact, or message body. `pending_slack` carries only a channel ID and a message ts — no credential. Boundary test asserts `slack_control/` imports no `keyring` and no `slack_sdk`, writes no files directly (the state read goes through `tools/state_io`), and adds no subprocess surface.
6. **No new dependency.** `slack_sdk` is already pinned (pass 1); the new `message.channels` event subscription + `channels:history` bot scope are **Slack-app-config** changes (operator step, T6), not a code dependency — so **no** `sbom`/`audit` churn.
This sprint **touches `src/`** (`core/state.py`, `core/engine.py`, `runner.py`, `tools/slack_io/*`, `tools/state_io`, `slack_control/*`, `cli.py`) ⇒ every such PR needs a fresh-session **`architect-review`**, and the `/critic-gate` pass applies with force — **security-critic** (untrusted thread-reply taint to a model-execution + spend sink; the new correlation-scan read path) and **architect** (the `core/` `EscalationFiler` seam, the `State` schema bump, `slack_control/`'s widened boundary) are both indicated, matching the pass-1/pass-2 critic-routing precedent.

**Risks & Blockers:**
- **`State` schema bump is a load-bearing invariant.** `CURRENT_SCHEMA_VERSION` 4→5, `migrate_state_payload` must upgrade a v4 payload (default `pending_slack=None`), and `extra="forbid"` must stay intact. The migration test must cover a v4→v5 round-trip and that a v5 snapshot with `pending_slack` set validates. Get this in T1 alone, first, on its own PR — every later task builds on the field existing.
- **The scan root is the main-checkout `state/`, NOT the worktrees (verified against `tools/state_io/writer.py`).** Snapshots are anchored to `state_root()`, which `worktree_run` pins to the **original main-checkout CWD before chdir'ing** — the writer's own docstring: "snapshots must stay in the orchestrator's main checkout." So **every** run's snapshots (worktree isolation on or off) land at `<main-checkout>/state/<run_id>/*.json`; they do **not** live inside `.worktrees/<run_id>/`. `find_paused_snapshot_by_slack_thread` is therefore a plain `state/*/*.json` glob — **do not** build worktree traversal. The real operational constraint: the **daemon must run from the same checkout the runs are started in**, since the scan is relative to the daemon's CWD (state this in T6's operator docs). The scan must also **tolerate old/foreign snapshots**: `state/` holds v4 and non-Slack snapshots, so read each candidate's `status` + `pending_slack.message_ts` from raw JSON (or migrate defensively) rather than hard-`model_validate`-ing every file (finding #9). (`flows/maintenance`/foreign-clone runs use `run_in_tree`, which does *not* pin `state_root`, so their snapshots live in the clone — exactly why those runs are out of Slack-escalation scope.) Unit-test the scan against a fixture `state/` tree.
- **Two Socket Mode event types now share one listener.** The pass-2 `inbound.py` listener already forwards **all** request types to the handler, so no transport change is needed — but the app must be **subscribed to `message.channels`** (Events API) and the bot must hold **`channels:history`** (operator step, T6). Without them no `message` event ever arrives and the round-trip silently never fires. The daemon must branch cleanly on `request.type` (`slash_commands` vs `events_api`) and never let a `message` event fall into the slash-command path or vice versa.
- **Self-trigger loop.** The daemon posts the escalation message and the outcome reply as the bot; it must ignore `message` events whose author is the bot itself (and `message_changed`/`bot_message` subtypes), or it will parse its own posts as answers. Test with a bot-authored thread message ⇒ dropped.
- **Fold-back parity.** The Slack resume must produce the *same* re-entry as the issue resume for the same answers: reuse `apply_answers`-style marking, `PM.fold_answers`, and `reentry_index` verbatim through the shared `runner.resume_run` seam. Refactoring `cli.resume` to call the seam risks changing the issue path — cover both callers with tests and confirm the issue-path resume behavior is byte-for-byte unchanged.
- **`_run_lock` now serializes starts *and* resumes.** A `/agent-run` start and a thread-reply resume must not race their process-global `os.chdir` (BL-8). `dispatch_resume` must take the **same** `_run_lock` as `dispatch`; test that a start in flight blocks a resume (and vice versa).
- **`src/`-touching ⇒ `architect-review` on each PR, in a FRESH session.** Keep tasks on clearly-scoped PRs (see PR structure). Do **not** `/model opus` mid-session to "review". `/handoff` → new session → `/resume` → `/code-review` → post the verbatim-header review. Watch the **BL-35 stale-red** trap on every `src/` PR here (BLOCKED + rollup FAILURE ⇒ `gh run rerun` the old run).
- **`slack_sdk` not installed in a planning-only env.** As in pass 2, this Opus session's interpreter may lack `slack_sdk`; the coder session must run under the synced project env.
