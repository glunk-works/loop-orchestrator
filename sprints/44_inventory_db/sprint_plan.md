**Sprint Goal:** Stand up **`tools/inventory_db`** — the bounty loop's cross-run asset/finding inventory and the **sole module that opens a Postgres connection** (the DB-connection analog of `tools/llm/client.py` being the only `keyring` importer, and of `state_io`/`scaffold` being the only file-writers) — together with the **§4 Postgres schema** (`targets`/`assets`/`endpoints`/`findings` + run linkage). This is Phase 0 **sprint 2 of 3** (`docs/bounty_loop_architecture.md` §8): 43 (BL-5 routing, done) → **44 (this plan)** → 45 (scope validator §5 + ingestion seam §10, which reads 44's `targets` rules-of-engagement). Per **micro-gate 1 (P0-D7)** the deliverable is **schema + the write surface only**: the versioned idempotent `.sql` DDL, a typed `InventoryRepository` write interface (upsert `targets`/`assets`/`endpoints`, append-insert `findings`), an **in-memory fake** as the default hermetic test double, and a **real psycopg3 (sync)** implementation exercised by a **skip-when-no-DSN** integration test. Read/query/dedup/join methods are **deferred to Phase 1** when the Recon stage becomes the first consumer — there is no in-repo consumer of this module until then (P0-D2), so building the read surface now would be unvalidated speculative code (same YAGNI logic as P0-D2/D3). This sprint touches **no `State` shape** and does **not** bump `schema_version` (P0-D2).

**Out of scope:**
- **The read/query surface** (`get_*`, `list_*_for_*`, dedup lookups, referential-join reads). Deferred to Phase 1's Recon stage, the first consumer (micro-gate 1 / P0-D7). This sprint ships only what makes the schema real and writable.
- **Any `State` change / `schema_version` bump.** The 5→6 bump ships with the first *bounty* `State` field in Phase 1 (P0-D2), not with pure non-`State` infra. `inventory_db` rows are referenced from `State` **by ID** only in Phase 1+; this sprint adds no `State` field and no `migrate_state_payload` branch.
- **The scope validator (§5) and the ingestion-sanitization seam (§10).** Those are **sprint 45** (P0-D1/P0-D6). This sprint stores the `targets` rules-of-engagement faithfully but performs **no scope enforcement** and **ingests no scanner output** — no untrusted scanner text reaches this module yet.
- **Docker / a live Postgres in CI.** P0-D4: the default `hatch run test` suite is hermetic against the in-memory fake; the real psycopg3 + DDL path is a **marked integration test that SKIPS when `LOOP_ORCHESTRATOR_INVENTORY_DSN` is unset**, discharged via the existing `live-verify`/deferred-verification posture. No devcontainer/CI Postgres service is added.
- **Alembic / any migration tool.** P0-D5: the schema ships as a versioned idempotent `.sql` applied via `CREATE … IF NOT EXISTS`. A migration tool is deferred to the first schema *evolution*.
- **Connection pooling, async psycopg, a bounty `Stage`/persona, or any `bounty-infra`/S3 ingestion wiring.** Per-operation sync connections are sufficient for a module with no consumer; pooling and async are YAGNI until a consumer's load profile exists. No Phase 1+ stage/persona is created here.

- **P0-D7 (micro-gate 1) — Scope = "schema + writes only".** Ship the `.sql` schema, the connection-owner, the idempotent bootstrap, and the upsert/insert **write** surface (§4's multi-writer recon path) + the in-memory fake + the skip-when-no-DSN psycopg3 integration test. **Defer** the read/query/join surface to Phase 1's Recon consumer. Not full-CRUD (unvalidated, no consumer), not schema-only (leaves the module unable to write a row and the fake/boundary pattern nothing to pin).
- **P0-D8 (micro-gate 2) — UUID primary keys + explicit natural keys.** Every table `id UUID PRIMARY KEY DEFAULT gen_random_uuid()` (built-in, PG13+): opaque IDs travel cleanly into JSON run snapshots (the §4 by-ID boundary discipline), no sequence contention under multi-writer recon, and the in-memory fake mints identical IDs with `uuid4()` so fake↔psycopg3 stay behaviorally aligned. Natural-key uniqueness for idempotent upsert: `targets.program_name` UNIQUE; `assets UNIQUE(target_id, asset_identifier)`; `endpoints UNIQUE(asset_id, url_path)`. **`findings` is append-only `insert`, not upsert** — each finding is a distinct observation — carrying its producing `run_id` as a **plain-text soft reference (no FK)**, because run snapshots live in JSON, never Postgres (§4).
- **P0-D9 (micro-gate 3) — psycopg3 is a hard-pinned dependency.** Add **`psycopg[binary]==<current 3.x>`** (bundled libpq wheel, no system libpq dependency) to the flat `dependencies` list in `pyproject.toml` — consistent with the existing all-deps-pinned-in-one-list convention (there is no `optional-dependencies` group, and this sprint does not introduce one). psycopg is imported **module-level only inside `tools/inventory_db`'s real impl**, pinned by a new sole-importer boundary test. Every install ships DB-capable; `sbom` regen + `dependency-audit` (pip-audit) run this sprint.
- **P0-D10 (micro-gate 4) — `validation_status` is a `TEXT` column with a `CHECK` constraint**, not a native ENUM: `CHECK (validation_status IN ('unverified','ai_verified','human_verified'))` rides inline in `CREATE TABLE IF NOT EXISTS` with zero extra idempotency machinery (a native `CREATE TYPE … AS ENUM` has no `IF NOT EXISTS` form and would need a `DO $$ … EXCEPTION WHEN duplicate_object $$` wrapper — friction against P0-D5's clean idempotent DDL). The in-memory fake validates the same allowed set in Python. Following §4: `in_scope_regex`/`out_of_scope_regex`/`banned_actions`/`http_methods` ship as `TEXT[]`, `open_ports` as `INT[]`, and `raw_scan_data`/`tech_stack` as `JSONB`.

**Sprint-44 specifics (this plan):**

- **The §4 schema (`inventory.sql`, applied idempotently — P0-D5/P0-D8/P0-D10).** One versioned `.sql` resource shipped inside the package, all statements `CREATE … IF NOT EXISTS`, applied by `bootstrap()`. The four tables, exactly per §4 + the locked identity model:
  - **`targets`** — `id UUID PK`, `program_name TEXT NOT NULL UNIQUE`, `in_scope_regex TEXT[]`, `out_of_scope_regex TEXT[]`, `banned_actions TEXT[]`. The rules-of-engagement and root of scope enforcement (§5, consumed in sprint 45).
  - **`assets`** — `id UUID PK`, `target_id UUID NOT NULL REFERENCES targets(id)`, `asset_identifier TEXT NOT NULL`, `asset_type TEXT`, `open_ports INT[]`, `raw_scan_data JSONB`, `UNIQUE(target_id, asset_identifier)`.
  - **`endpoints`** — `id UUID PK`, `asset_id UUID NOT NULL REFERENCES assets(id)`, `url_path TEXT NOT NULL`, `http_methods TEXT[]`, `tech_stack JSONB`, `requires_auth BOOLEAN`, `UNIQUE(asset_id, url_path)`.
  - **`findings`** — `id UUID PK`, `endpoint_id UUID NOT NULL REFERENCES endpoints(id)`, `run_id TEXT NOT NULL` (soft ref to the producing snapshot — **no FK**), `finding_type TEXT`, `severity TEXT`, `reproduction_steps TEXT`, `validation_status TEXT NOT NULL DEFAULT 'unverified' CHECK (validation_status IN ('unverified','ai_verified','human_verified'))`.
  - LangGraph's own checkpoint tables are **not** created (§4 / §9-D1) — execution state stays on JSON snapshots.
- **The typed write interface + two implementations (P0-D4/P0-D7).**
  - Pydantic domain models (`extra="forbid"`) for `Target`/`Asset`/`Endpoint`/`Finding` and thin ID newtypes (`TargetId`/`AssetId`/`EndpointId`/`FindingId` over `UUID`/`str`), so callers pass typed rows and receive typed IDs — the §4 by-ID boundary is expressed in the type system.
  - An `InventoryRepository` **Protocol** (write surface only): `bootstrap() -> None`; `upsert_target(...) -> TargetId`; `upsert_asset(...) -> AssetId`; `upsert_endpoint(...) -> EndpointId`; `insert_finding(...) -> FindingId`. Upserts key on the natural keys (P0-D8) via `ON CONFLICT … DO UPDATE`; `insert_finding` is a plain append.
  - `InMemoryInventory` — the hermetic fake: dict-backed, mints `uuid4()` IDs, enforces the same natural-key upsert semantics and the same `validation_status` allowed set, raising the same typed errors. This is the **default** test double and the thing the whole hermetic suite runs against.
  - `PsycopgInventory` — the real impl: constructed from the `LOOP_ORCHESTRATOR_INVENTORY_DSN` env var (via a small `build_inventory_from_env()` that **fails closed** when the transport is requested but the DSN is unset — same posture as the other env-var credential classes), holds a sync `psycopg` connection, `bootstrap()` applies `inventory.sql`, each write runs in its own transaction (per-op, no pool — YAGNI). **All SQL is parameterized** (psycopg parameter binding); the only non-parameterized SQL is the static `inventory.sql` DDL. This is the sole module that imports `psycopg`.
- **The new module boundary (mirrors the keyring single-importer test).** `tools/inventory_db` is the **only** module permitted to import `psycopg` — pin it with an AST test modeled exactly on `tests/tools/test_keyring_boundary.py` (walk every `src/` module, assert `psycopg`/`psycopg.*` is imported only by `PsycopgInventory`'s file). It imports **no `keyring`** (the keyring boundary test stays green), **no `subprocess`** (psycopg is a library over a socket — the **five** sanctioned subprocess surfaces in `tests/tools/test_subprocess_surfaces.py` are **unchanged**; this adds no sixth), and writes no files (reading the packaged `inventory.sql` via `importlib.resources` is a read, not a `state_io`/`scaffold` write-surface concern). No existing module imports `inventory_db` this sprint (no consumer), so `core/`'s import graph is untouched.

**Security Considerations:** Moderate — this sprint introduces the loop's **first SQL sink** and the **first DB credential**, but **no untrusted-input path yet** (scanner-output ingestion is sprint 45). Diligence items:
1. **Parameterized queries are the security-relevant invariant here — the SQL-sink analog of fixed-argv/`shell=False` on the subprocess surfaces.** Every value-bearing query uses psycopg parameter binding; **no** SQL is built by f-string/`%`/`.format()`/`+` on caller data. This matters because in Phase 1+ these exact write methods receive **scanner-derived, attacker-influenceable** data (§10 threat-model delta) — building the injection-proof seam now, before any untrusted data flows, is the point. The static `inventory.sql` DDL is the only non-parameterized SQL and contains no interpolation. Add a test/lint angle asserting the impl builds no SQL string from arguments (a focused AST/grep check on the impl module is acceptable).
2. **The DSN is an env var, not keyring (P0-D4 / §9).** `LOOP_ORCHESTRATOR_INVENTORY_DSN` is a distinct credential class — same posture as `LOOP_ORCHESTRATOR_WEBHOOK_SECRET` and the Slack tokens — authenticating an outbound DB connection, not the Anthropic LLM call. `inventory_db` must **not** import `keyring`; `build_inventory_from_env()` **fails closed** when Slack-style requested-but-unconfigured. The DSN (which may embed a password) must never be logged or surfaced in an error message.
3. **`extra="forbid"` on every new Pydantic domain model** (the same discipline `State` and `ModelRates` hold) — a typed row is a fixed shape, not an open bag.
4. **`targets` integrity matters because sprint 45's scope validator reads it as the rules-of-engagement** — but scope *enforcement* is explicitly sprint 45, not here. This sprint's job is faithful storage: the CHECK constraint, the NOT NULLs, and the FKs are the storage-integrity guards.

This sprint **touches `src/`** (new `tools/inventory_db/` package, `pyproject.toml` dep, `sbom.json` regen) ⇒ **every such PR needs a fresh-session `architect-review`**, and the `/critic-gate` pass applies. Indicated critics: **architect** (the new module-boundary placement + the schema/interface shape), **security-critic** (the parameterized-query invariant and the env-var-not-keyring credential posture — genuinely worth a pass since this is the first SQL sink + first DB credential), and **guard-adversary** on the **new psycopg single-importer boundary test** (verify it goes RED under a planted `import psycopg` in a disallowed module — the BL-15 "a guard that never fails" failure mode). Docs-only tasks are `architect-review`-exempt.

**Risks & Blockers:**
- **A loose SQL construction re-opens injection.** The single highest-value check. All value-bearing SQL parameterized; the AST/grep guard in Security item 1 is what makes a regression fail a test rather than ship silently. Do not "optimize" a query into an f-string.
- **The integration test must SKIP, not ERROR, when no DSN is set.** In CI (`LOOP_ORCHESTRATOR_INVENTORY_DSN` unset) a `pytest.mark`/skip keyed on the env var must cause a clean skip; a test that instead errors on a missing DSN or a failed `psycopg.connect` breaks the hermetic gate. Verify the skip fires in a no-DSN run locally before pushing (`hatch run test -k inventory` shows it skipped, not errored).
- **`psycopg[binary]` pin trips the supply-chain gates.** Adding the dep **requires** `hatch run sbom` (regenerate `sbom.json`) and a clean `hatch run audit` (pip-audit) — both are CI gates. Pin the current stable psycopg 3.x release and confirm pip-audit is clean on it; a CVE in the chosen version blocks the PR until re-pinned. `psycopg[binary]` (bundled libpq) is deliberate — no system libpq install, keeps the hermetic suite and container portable.
- **The psycopg boundary test must actually constrain (BL-15).** Land it in the **same PR** as the psycopg import it guards (T2), and have guard-adversary confirm it fails under a planted violation — a boundary test that passes vacuously is worthless.
- **`src/`-touching ⇒ `architect-review` on each `src/` PR, in a FRESH session.** Do **not** `/model opus` mid-session to "review". `/handoff` → new session → `/resume` → `/code-review` → post the verbatim-header review. Watch the **BL-35 stale-red** trap on every `src/` PR (BLOCKED + rollup FAILURE ⇒ `gh run rerun` the OLD run — `architect-review` fires on both `pull_request` and `pull_request_review`).
- **Full local gate before push.** Run lint → format → test (or `/ship`) — a new package + a `.sql` resource + a dep pin is exactly the diff that trips `format-check` or a packaging include if only `hatch run test` was run (memory: #141). Confirm the `.sql` is packaged (wheel `include`/`force-include`) so `importlib.resources` finds it at runtime. **PR title ≤72 bytes** — `wc -c` first. **Never commit to `main`, merge, or force-push.**
