### FILEPATH: /sprints/37_test_suite_velocity/sprint_plan.md

**Sprint Goal:** Make the test suite cheap to run constantly (the frame the repo owner set for BL-22: the goal is *not* "run the tests less" — running them on every change is essential to TDD — it is to **stop paying for runs that cannot fail** and to **make the run itself cheap**). Two independent levers, landed as separate PRs: (1) cut the number of times the suite pays the fixed ~5s cost of spawning a stdio MCP server, and (2) stop docs-only PRs from paying the ~330s pytest run they cannot possibly fail. A measured `pytest-xdist` spike closes the sprint.

**Out of scope:**
- **BL-23** (the test-validity audit — mutation-testing the boundary guards, hunting orphan/weak tests). Its own sprint, next.
- **Reducing the per-spawn cost itself** (the ~5s is import-bound cold-start, paid by every *real* coder session too). Filed as **BL-31** and deliberately deferred (owner decision, this planning pass) — it touches `src/` runtime and deserves its own framed unit; this sprint only reduces the *number* of spawns.
- The `push: [main]`-re-runs-everything thread and the `dependency-audit`/`sbom`-on-every-PR thread from BL-22. Left alone this sprint: the post-merge `main` run is the only thing that catches a semantic conflict between two independently-green PRs (BL-22 says evaluate, don't assume waste), and the docs short-circuit (Task 3) is scoped to the one job that dominates — `test`.
- Any change to the eight required checks, the ruleset, the four workflows' *trigger* logic, or `tests/test_ci_config.py`'s guards. Task 3 is explicitly designed to need **none** of these (see FD3).

**Context (locked decisions — measured this planning pass, do not re-open):**

- **FD1 — The dominant cost is a fixed ~5s-per-spawn MCP server cold-start, measured not estimated** *(locked 2026-07-15, Opus/Architect; owner-confirmed)*. Timed in isolation on this host: `build_coder_tool_provider().__enter__()` costs **~5.0s** (spawn the stdio subprocess + `session.initialize()` + `list_tools()`); teardown is a cheap **~0.25s**. The cost is import-bound — a fresh subprocess re-imports `loop_engine` + the MCP/pydantic/anthropic stack. The suite is 278s locally / ~380s in CI, and **~20 tests each pay this penalty at least once — roughly 100-115s, ~40% of the suite**. The lever this sprint pulls is *the number of spawns*, not the per-spawn cost (FD5 / BL-31). `_SHUTDOWN_TIMEOUT_S = 10.0` is **not** implicated — teardown is fast; do not chase it.

- **FD2 — The backlog was right about the symptom and wrong about the location for `test_ralph_coder`** *(locked 2026-07-15, Opus/Architect; owner-confirmed)*. BL-22 attributed the ~75s in `tests/personas/test_ralph_coder.py` (~9 tests) to fixture-spawned MCP subprocesses. It is not a fixture. These tests use a `MagicMock` LLM whose `run_tool_loop` is fully stubbed, yet `RalphCoderPersona.run` still calls `_CoderToolBackend.resolve()` (`src/loop_engine/personas/coder_iac/shared.py:154`), which **spawns a real 5s MCP server purely to enumerate tool schemas for a mock that never dispatches to them**. `test_ralph_coder.py` does not mock the provider the way `tests/personas/test_coder_tool_backend.py` already does. This is the single largest, lowest-risk reclaim and it is a *mock*, not a fixture-scoping change (Task 1). The direct-spawner cluster (`test_mcp_provider`, `test_github_server`, `test_issue_io_server`, `test_issue_provider`) is the fixture-scoping target (Task 2).

- **FD3 — The docs-only exemption is an in-job short-circuit, NOT the backlog's aggregator-with-conditional-jobs** *(locked 2026-07-15, Opus/Architect; owner asked for a recommendation, this is it)*. BL-22 sketches an `if: always()` aggregator job with heavy jobs conditional behind it. That pattern requires job-level `if:`, which would force **rewriting `tests/test_ci_config.py::test_no_job_in_ci_workflow_can_report_skipped` (line 151)** — the codified BL-10 lesson — to *permit* what it exists to forbid, and **re-pointing the ruleset's required contexts** from `test` to the aggregator. On a solo repo where that ruleset is the only thing between a mistake and `main`, that reopens the blast radius of the two most expensive bugs in this repo's history to save ~40s of job *setup* on docs PRs. **We do not.** Instead the `test` job stays unconditional (no job-level `if:`; it always reports a check-run named `test`, green or red only for real reasons), and a *step* decides whether to run pytest. "Skipped" and "absent" — the exact BL-10 and BL-12 failure modes — become **structurally impossible**, not merely handled. `architect-review` in `hitl-review.yml` is the working precedent: an always-running required check that internally runs `gh api .../files` and decides whether to do work.

- **FD4 — Task 3 borrows `architect-review`'s mechanism but MUST invert its predicate, carefully** *(locked 2026-07-15, Opus/Architect)*. Three things carry over from `hitl-review.yml`: (a) get the changed-file list from the **API** — `gh api "repos/$REPO/pulls/$PR/files" --paginate --jq '.[].filename'` — so there is no `git diff` and no `fetch-depth` dependency; (b) obey the **SIGPIPE lesson** at `hitl-review.yml:70-74` — capture the list into a variable first, match second; **never** pipe `gh --paginate` into `grep -q`, or a large diff dies of SIGPIPE 141 under `pipefail` and the gate silently exempts itself on exactly the big PRs. What does **not** carry over is the predicate: `architect-review` skips when *no* file matches `^src/`. The test suite also covers `tests/`, `pyproject.toml`, `.github/`, `conftest.py`, `loop_engine.mcp.json`, etc., so Task 3 must skip pytest **only when *every* changed file is docs**, and **run on anything else and on any error** (fail-safe). Occasionally running pytest on a docs PR is fine; skipping it on a code PR is the bug. Docs set: `**/*.md`, `.ai/**`, `sprints/**`, `docs/**` — nothing else.

- **FD5 — This sprint reduces spawn *count*, never spawn *cost*, and never real-server *coverage*** *(locked 2026-07-15, Opus/Architect; owner-confirmed)*. Task 1 removes *redundant* spawns from persona unit tests (which assert nothing about the real server). It must **not** reduce the repo's end-to-end coverage of the real coder-tools server — that coverage lives in `tests/tools/test_mcp_provider.py` (dispatch-matches-in-process) and stays. After Task 1, at least one test must still spawn a real coder-tools server and assert real dispatch. The per-spawn cost reduction is BL-31, a separate unit.

**Security Considerations:** No new dependency, no new credential path, no new subprocess surface. Task 3 adds a `gh api` call inside the existing `test` job — the same already-sanctioned `gh` surface `hitl-review.yml` uses, reading `repos/$REPO/pulls/$PR/files` under the workflow's `GITHUB_TOKEN`; it introduces no new secret and no `src/` change. Tasks 1-2 are test-only. The one security-relevant property is Task 3's **fail-safe polarity** (FD4): the short-circuit must default to *running* the suite, so a detection error can never let an unreviewed code change merge without its tests — the same failure-open discipline that `architect-review` gets right.

**Risks & Blockers:**
- **Task 3 fail-safe is the whole ballgame.** Any path where the changed-file detection errors, returns empty, or is ambiguous MUST run pytest. Test the polarity explicitly (a PR touching `tests/` or `pyproject.toml` must run; only an all-docs PR skips).
- **The SIGPIPE trap is real and size-dependent** (FD4). It only bites on diffs long enough that `gh` is still paginating — i.e. the big PRs — so a naive `gh ... | grep -q` will pass every small-PR test and fail silently in production. Capture-then-match; add a note pointing at `hitl-review.yml:70-74`.
- **Task 2 shared-state collisions.** Session-scoping a provider is only safe for read-only/discovery tests. Any test that writes into the seeded tree, or asserts on a fresh tree, stays function-scoped. Audit before scoping; a flaky test here is worse than a slow one.
- **Do not add a job-level `if:` to any `ci.yml` job** (FD3). `test_no_job_in_ci_workflow_can_report_skipped` will fail, and rightly — that is the guard, not an obstacle. Task 3's `if:` is **step-level**, which that test does not (and must not) forbid.
- **Absolute-second thresholds are flaky** as pass/fail gates (CI-runner variance). Acceptance is "measurable reduction on the target files + zero assertions weakened + real-server coverage preserved," with numeric deltas recorded as *evidence*, not gates.
- **Sequential PRs** (T1 → T2 → T3 → T4) so each carries a clean before/after `--durations` delta and T4's xdist measurement is taken *after* the spawn count is already cut. Each of T1-T3 touches `src/`-adjacent test/CI files → each needs its own fresh-session `architect-review` if it touches `src/`; T1/T2 touch only `tests/`, T3 touches only `.github/workflows/ci.yml`, so **none touches `src/`** and all are `architect-review`-exempt — but each still needs the full green suite.

**Tasks:**

- **Task 1: Stop `test_ralph_coder.py` from spawning a real MCP server**
  - **Description:** Per FD2, the ~9 ralph tests each spawn a real ~5s coder-tools server only to hand tool schemas to a `MagicMock` LLM that never uses them. Mock the boundary the way `tests/personas/test_coder_tool_backend.py:35` already does — patch `loop_engine.personas.coder_iac.shared.build_coder_tool_provider` (or inject a `_FakeProvider`) so `_CoderToolBackend.resolve()` returns a stub `(tools, execute)` without spawning a subprocess. These are unit tests of the persona's task-selection / report / escalation logic; the real server is irrelevant to every assertion they make. Per FD5, do **not** touch the real-server coverage in `test_mcp_provider.py`.
  - **Target Files:** `tests/personas/test_ralph_coder.py` (and a shared fake/fixture if one is cleaner than per-test patching)
  - **Acceptance Criteria:**
    - No test in `test_ralph_coder.py` spawns a real MCP server (verify: the `select.epoll`/subprocess wall time is gone; `--durations` for the file drops from ~75s toward single digits).
    - Every existing assertion in the file is preserved unchanged — the mock changes *how the tools list is obtained*, nothing about what the tests check.
    - `tests/tools/test_mcp_provider.py` still exercises a real coder-tools server end-to-end (FD5) — untouched.
    - Full suite green; `hatch run lint` / `hatch run format` clean.

- **Task 2: Session-scope the direct-spawner MCP tests**
  - **Description:** The four files `test_mcp_provider.py`, `test_github_server.py`, `test_issue_io_server.py`, `test_issue_provider.py` each open a provider per test via a context-manager fixture, paying ~5s every time. Amortize the spawn with **session-scoped** fixtures: a shared read-only seeded tree and a shared provider for the coder-tools dispatch tests; a shared discovery provider for the github/issue tests (these take no `cwd` and only list tools). Audit every test in these files for state it writes into the tree or an assumption of a fresh spawn (per Risks) — anything that mutates stays function-scoped. This weakens no assertion: the same real server is exercised, just launched once per session instead of once per test.
  - **Target Files:** `tests/tools/test_mcp_provider.py`, `tests/tools/test_github_server.py`, `tests/tools/test_issue_io_server.py`, `tests/tools/test_issue_provider.py` (and `tests/conftest.py` if a shared session fixture belongs there)
  - **Acceptance Criteria:**
    - Providers in these files are spawned at **session** scope where the tests are read-only/discovery-only; the per-file `--durations` drops materially (record the before/after).
    - No assertion is weakened; the real-server dispatch/discovery/disjointness checks (incl. `test_coder_github_and_issue_providers_are_pairwise_disjoint`) all still run against a real server.
    - No new cross-test order-dependence or shared-state flake — the four files pass in isolation and in a full run, and pass on a repeat run.
    - Full suite green; lint/format clean.

- **Task 3: Short-circuit the `test` job for docs-only PRs (Option A, in-job)**
  - **Description:** Per FD3/FD4, keep the `test` job unconditional (no job-level `if:`) and add a step that computes whether the PR is docs-only and, if so, skips `hatch run test` while the job still reports a green `test` check-run. Use the API for the changed-file list, capture-then-match to dodge SIGPIPE, and **fail safe**: run pytest unless *every* changed file matches `{**/*.md, .ai/**, sprints/**, docs/**}`, and run pytest on any API error or empty result. Scope to the `test` job only; leave `lint`/`format-check`/`secrets-scan`/`dependency-audit`/`sbom` running fully (they are cheap relative to pytest, and widening the exemption is a separate decision). The step's `if:` guards only the `hatch run test` invocation; the job carries no `if:`, so `test_no_job_in_ci_workflow_can_report_skipped` stays green. Add a comment pointing at `hitl-review.yml:70-74` for the SIGPIPE rationale.
  - **Target Files:** `.github/workflows/ci.yml` (and a comment; possibly a tiny assertion in `tests/test_ci_config.py` that the `test` job still carries no job-level `if:` and that the short-circuit is step-level — additive, not a rewrite of line 151)
  - **Acceptance Criteria:**
    - The `test` job has **no job-level `if:`** and always reports a check-run named `test`; the short-circuit is a step-level `if:`.
    - On a docs-only PR (`**/*.md`, `.ai/**`, `sprints/**`, `docs/**` only) the job skips `hatch run test` and reports green; on any PR touching `tests/`, `src/`, `pyproject.toml`, `.github/`, `loop_engine.mcp.json`, etc., pytest runs.
    - The changed-file detection uses the API and is SIGPIPE-safe (capture-then-match, no `gh --paginate | grep -q`); it fails safe (runs pytest) on error or empty result.
    - No change to the ruleset, the eight required check names, or `test_no_job_in_ci_workflow_can_report_skipped`.
    - Verified live on a throwaway docs-only PR and a throwaway code-touching PR (or documented equivalent) before this task's own PR merges.

- **Task 4: Measure `pytest-xdist -n auto` and decide**
  - **Description:** After Tasks 1-2 cut the spawn count, measure whether `pytest-xdist -n auto` still helps and stays clean. The slow tail is largely subprocess-bound (MCP servers, pytest subprocesses), which parallelizes well — but the spawning tests must be checked for shared-state collisions first (worktree/`run_id` isolation, `tmp_path`, the session-scoped providers from Task 2 which do **not** parallelize across workers trivially). Land the dependency + `-n auto` only if it produces a real, repeatable speedup with a green suite; otherwise record the measurement and the reason it was declined, and close the thread.
  - **Target Files:** `pyproject.toml` (+ `sbom.json` regen and `hatch run audit` if the dep lands — the dep-change discipline), or *(none — a recorded spike)* if declined.
  - **Acceptance Criteria:**
    - A recorded before/after wall-time measurement with `-n auto` on this host and/or CI.
    - If landed: full suite green under `-n auto` on a repeat run (no order/shared-state flake), `pyproject.toml` dep pinned, `sbom.json` regenerated, `hatch run audit` clean.
    - If declined: a one-paragraph rationale in the sprint's handoff/next-steps naming the blocker (e.g. session-scoped-provider contention), so it is a closed decision, not an open loose end.
