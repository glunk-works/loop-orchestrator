**Context from the review (locked decisions — do not re-open):**
- **The escalation ladder / persona-switch is the `ESCALATE` decision path, not `StageGateFailedError`.** In `execute_stage` (`core/engine.py`) a stage that can't ACCEPT ends three ways: (1) gate returns REVISE with findings *identical* to the prior attempt ⇒ engine converts to ESCALATE (the no-progress detector, routes to the resolver ladder); (2) gate returns REVISE with findings that keep *changing* until the revision budget is spent ⇒ `raise StageGateFailedError` (a hard `FAILED_STAGE`, **no** persona switch); (3) gate returns ESCALATE directly. Case (2) hard-fails the run — for PM that is a dead-end because PM's only resolver is the human.
- **Decision (Architect): case (2) should ESCALATE instead of hard-fail — for the PM stage ONLY.** Implemented as a per-stage flag (mirrors the existing per-stage `max_revisions`), default `False` so every other stage and the classic path keep hard-failing exactly as today. **Do not** make this a blanket change; the two existing `StageGateFailedError` tests must stay green unchanged.
- **Both engines share `execute_stage`** (`graph_engine.py` calls `core.engine.execute_stage`), so the engine fix lands once and both engines inherit it. No separate LangGraph edit.
- **`fold_answers` is stateless w.r.t. the persona instance** — it uses only module-level constants in `personas/pm/`. The classic sprint-20 plan intended it to stay a resume-time PM service usable by the declarative PM; the implementation wired `PMGenerator` at stage 0 without exposing it, which broke `resume --from-issue`. The fix restores the intended shape.

**Security Considerations:** No new attack surface. No new dependency, subprocess, credential, or file-write surface. The `utf-8` read fix (Task 4) only pins an encoding on already-permitted `read_text` reads of in-repo trusted prompt/config assets (still read-only, no `yaml.load`). `fold_answers` continues to run on the resume path inside the run's worktree exactly as today. Module-boundary tests (`core/` imports no concrete persona; only `state_io` writes files) must remain green — in particular Task 1 must not cause `core/` to import a concrete persona.

**Risks & Blockers:**
- **The PM-only escalation change is a semantics change on the escalation path — keep it flag-scoped and defaulted off.** Setting `escalate_on_exhaustion` on any other stage would flip the two existing `StageGateFailedError` tests and change classic behavior; the default `False` is load-bearing. Verify both existing raise-tests still pass **without edits**.
- **Do not fake `revision_history` or reconstruct per-field human questions** (already-locked sprint-20 decisions). The combined single escalation question carrying all unsatisfied findings is the intended shape.
- **`fold_answers` byte-parity on resume.** Extracting it to a module function must not change its behavior; assert the declarative resume path folds answers identically to the classic path (same `project_spec`, same impact classification).
- **No live end-to-end verification on this branch** (no LLM key). Verification is unit-level against the fake-LLM harness, consistent with sprint 20; append any deferred live check to `sprints/DEFERRED_VERIFICATION.md`.
- **Item #4 from the review (key_merge findings accumulation) is deliberately NOT in this sprint** — see "Deferred (needs Architect decision)" below. Do not implement it here.

- **Task 4: `utf-8` encoding on declarative prompt/config reads (review finding #5, PLAUSIBLE)**
  - **Description:** `personas/declarative/config.py` reads prompt files (`load_prompt` → `resolve_prompt_path(...).read_text()`) and config files (`load_generator_config` → `Path(path).read_text()`) with no `encoding`, so they decode with the platform-default codec. The referenced prompt files (`prompts/02_*`, `prompts/03_*`) and `configs/pm.yaml` contain U+2014 (em-dash); under a C/POSIX locale (`requires-python >=3.12`; PEP 686 UTF-8-mode default only from 3.15) `build_default_loop()` with the flag set raises `UnicodeDecodeError` before any run. Add `encoding="utf-8"` to both `read_text` calls. (Classic personas embed these as source-decoded string literals, so they were immune — this restores that immunity.)
  - **Target Files:** `src/loop_engine/personas/declarative/config.py`, `tests/personas/declarative/test_config.py`
  - **Acceptance Criteria:** `load_prompt` and `load_generator_config` decode as UTF-8 regardless of locale; a test constructs the three declarative generators (or loads a config/prompt containing a non-ASCII char) and asserts no decode error. Lint clean.
