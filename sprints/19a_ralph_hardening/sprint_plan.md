### FILEPATH: /sprints/19a_ralph_hardening/sprint_plan.md

**Sprint Goal:** Phase 4a hardening — close the four code-review findings recorded against the Ralph-loop Coder commit (`195f7b7`, roadmap follow-up #6 (a)–(d)) as **one focused, green, flag-scoped commit** *before* Phase 4 · part 2 (sprint 20) begins. Every fix lives entirely inside the Ralph surface (`personas/coder_iac/ralph.py`, `core/coder_gate.py`, `personas/agile_sprint_breakdown/manifest.py`) — all behind `LOOP_ENGINE_CODER=ralph`, so the default `classic` path is untouched and stays byte-identical. Load-bearing invariant preserved and *strengthened*: **the Ralph loop terminates deterministically and now also self-heals** — it advances while making progress, repairs a cross-task regression instead of dead-ending on it, escalates only when genuinely stuck (identical findings), and stays hard-bounded by the iteration cap and the per-run USD budget. The headline fix is (a): a green-but-red-suite state (all tasks checked off yet a regression turned the suite red) must trigger a **repair increment**, not an escalation dead-end — otherwise the "loop until green" promise breaks exactly on the regressions it exists to catch.

**Dependencies:** Phase 4 · part 1 (`195f7b7`) — this sprint hardens the code that commit shipped and touches nothing outside it. It is a **prerequisite for sprint 20**: the part-2 plan (`sprints/20_declarative_generators/sprint_plan.md`) is disjoint from every file here, and these fixes must NOT be folded into it (one concern per reviewable commit; a correctness hole in a just-shipped loop should not ride behind a larger unrelated refactor, and the deferred live-host verification (#5) is only meaningful against a loop that can actually self-heal). Reuses the existing `.agent/` semantic-state layer, the `_CoderToolBackend` tool loop, and `execute_stage`'s revise/identical-findings-escalate machinery **unchanged** — no engine change (keeps `execute_stage` generic, consistent with deferred follow-up #4).

**Design decisions baked in:**
- **(a) is a persona-side self-heal, not an `execute_stage` change.** The regression-repair path lives in `RalphCoderPersona.run` (triggered by a distinct, stable gate signal), so `execute_stage` stays generic. When the manifest is fully checked off but the suite is red, the gate emits a **dedicated regression finding** (a stable prefix constant, distinct from the incomplete-coverage status finding); the persona, seeing no selectable task *and* that regression finding, runs a **repair increment** (one fresh-context tool loop scoped to "diagnose and fix the regression; add no new features") instead of no-opping. Repairs are bounded exactly like task increments: each is one `execute_stage` revision, counted against `max_revisions`, debited against the budget; a repair that cannot converge escalates via the existing identical-findings guard (stable pytest output) or fails honestly at the cap.
- **The "no selectable task" state is now three-way, disambiguated by the gate's finding — not re-derived in the persona.** (1) all tasks done + green ⇒ gate ACCEPTs (persona never runs); (2) all tasks done + red ⇒ regression finding ⇒ **repair increment**; (3) outstanding-but-all-blocked tasks (an unmet-dep dead-end) ⇒ ordinary incomplete-coverage status finding ⇒ persona no-ops ⇒ existing escalation. The persona distinguishes (2) from (3) solely by the presence of the regression finding, so the blocked-work escalation behavior is unchanged.
- **(b) tightens dependency matching to sprint-qualified tokens; the order-safe fallback stays.** `manifest._dependency_sprint_paths` stops matching any bare `\d+` in the `Dependencies:` prose. It matches (i) a sprint **directory/name token** (`01_ci_cd_foundation` or `ci_cd_foundation`) appearing whole in the text, and (ii) a number **qualified** by a sprint context (`sprint 3`, `sprint #3`, `#3`, case-insensitive). Incidental digits (`OAuth2`, `RFC 6749`) no longer forge deps. The existing conservative fallback (non-empty, non-"None" `Dependencies:` with no recognized token ⇒ depend on the immediately preceding sprint) is retained because it only ever adds a plan-order dependency, which is always safe.
- **(c) report sections are upserted by task id, not blindly appended.** `implementation_reports` stays a `dict[sprint_path, str]` of prose (unchanged shape, still opaque to `CoderGate`/`RalphCoderGate` helpers), but a task's section is now written idempotently: re-running a task (after escalation/resolution) **replaces** its existing `### Task <id>:` block instead of appending a duplicate. Repair increments upsert a single `### Regression fix` section (latest repair supersedes) so repeated repairs stay bounded.
- **(d) the model sees carried resolution answers plus the latest gate status — not just `findings[-1]`.** The persona partitions its `findings` into resolution answers (a stable `RESOLUTION_FINDING_PREFIX`, shared with `engine._resolution_findings`) and gate status lines, and includes **all** resolution answers (bounded — one per answered question, carried only through the Coder stage) plus the **single most recent** gate-status/regression line. A resolution answer therefore survives across every post-re-entry iteration, while the prompt stays bounded and current.

**Security Considerations:**
- **No new subprocess / write / credential / isolation surface.** The repair increment reuses the same `run_tool_loop` + `_CoderToolBackend` as a task increment — the same, already-present untrusted-code surface, governed by the same budget and iteration cap. The gate still runs its in-process pytest **through the sprint-18 isolation guard** (`_raise_if_sandboxed`), which this sprint must not move or weaken: the new regression branch in the gate emits its finding **after** the guard fires under `container`/`sandbox`, exactly like the existing red-suite branch. Assert the guard still precedes any pytest execution.
- **All `.agent/`, report, and manifest writes stay on the `tools/state_io` path** (`write_artifact`/`write_scratchpad`/`append_memory`); the append-only `MEMORY.md` invariant is preserved (repair increments *append* a lesson, never rewrite). No module here imports `keyring` or opens files directly.
- **Tightening (b) cannot broaden the dependency graph.** The new matcher is strictly more conservative (fewer matches) with the same order-safe fallback, so it can only *remove* spurious edges, never introduce a cycle or a dangling dep the `ManifestArtifactGate` would have to reject.

**Risks & Blockers:**
- **A repair increment that changes files each pass alters pytest output, so the identical-findings guard may not fire** — such a run instead terminates at the `max_revisions` (iteration) cap via the existing `FAILED_STAGE` snapshot, and the budget remains the ultimate governor. This is acceptable v1 termination (bounded, honest, snapshot-persisted); "escalate-on-cap instead of fail" remains deferred follow-up #4. Do not add an unbounded repair loop.
- **The regression signal must be unambiguous.** (a) hinges on the persona reliably telling "all tasks done + red" (repair) from "outstanding blocked tasks" (escalate). Use a shared prefix constant emitted only by the gate's all-done-but-red branch — never string-sniff pytest output in the persona. A negative test must prove a blocked-but-incomplete state does **not** trigger a repair increment.
- **Still no live end-to-end verification on this branch** (no LLM key; no container runtime). Verification is unit-level against the fake-LLM harness: the regression repair path, the tightened dependency matcher, report-section idempotency, and the findings-partition prompt. Real convergence/cost of a self-healing Ralph run stays deferred to the live host run (`sprints/DEFERRED_VERIFICATION.md`).
- **Do not regress part 1's guarantees.** The coverage-aware ACCEPT (every task checked + green), the isolation guard placement, the flag-gating, and the no-parity-claim-vs-classic all stand unchanged; existing Ralph tests stay green verbatim except where a test encoded the old dead-end behavior (update it to assert the repair path, and say so in the commit).
