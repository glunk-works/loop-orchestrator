### FILEPATH: /sprints/28_gate_pytest_sandbox/sprint_plan.md

**Sprint Goal:** Finish Phase 3b's one deferred piece — route the **Coder gates' verification pytest through the same MCP container sandbox the Coder's `run_tests` *tool* already uses** — so a Coder-ACCEPT → `COMPLETED` becomes reachable under `LOOP_ENGINE_ISOLATION=container`. This closes finding **F-GATE-SANDBOX** and the sprint-18/§3 deferral (`sprints/DEFERRED_VERIFICATION.md`), and is the **prerequisite that unblocks sprint 27's V1(complete)/V2/V3** host runs (none of which can reach a green gate under container today). After this sprint, `core/coder_gate.py::_raise_if_sandboxed` is **gone** — the gate no longer refuses under sandbox modes; it verifies inside the sandbox.

**Why this is its own sprint (not folded into 27):** the flip block (27) is *subtractive* — it deletes proven-redundant scaffolding and adds no behavior. This sprint is *additive* — it adds the gate-sandbox dispatch path that Phase 3b left inert. Mixing an additive prerequisite into the subtractive block would violate 27's "removes scaffolding, adds no new behavior" contract. This sprint sequences **strictly before** sprint 27's V-runs: V1(complete)/V2/V3 all assume a green gate under container, which is false until this lands.

**Context (the exact gap being closed):**
- The Coder's `run_tests` **tool** was sandboxed in Phase 3b: on `LOOP_ENGINE_TOOLS=mcp` + `ISOLATION=container`, the model's tool loop dispatches `run_tests` through `build_coder_tool_provider(cwd=worktree)` → a hardened `--network none --cap-drop ALL --read-only` container (`tools/mcp/provider.py::container_server_params`).
- The Coder **gates** (`CoderGate` + `RalphCoderGate`, `core/coder_gate.py`) verify with their **own** pytest run via `_run_gate_pytest` → `tools/coder_tools/run_tests.py::run_pytest`, executed **in-process**. `_raise_if_sandboxed` refuses this under `sandbox_runtime_mode() is not None` — correctly, because running model-generated tests in the orchestrator process is the exact thing container mode exists to prevent. That refusal was always a placeholder for "route the gate's pytest through the sandbox too" (the sprint-18 deferral); this sprint does that routing and removes the placeholder.
- **Design decision (locked 2026-07-10, Opus/Architect planning pass; user-confirmed):** the isolation-aware dispatch lives in a new **`tools/mcp` helper** (`run_gate_pytest`), mirroring how `_CoderToolBackend.resolve` (`personas/coder_iac/shared.py:164`) already branches on `sandbox_runtime_mode()`. `core/coder_gate.py` calls it and gains a `core → tools/mcp` import edge (it already imports `tools/coder_tools` + `tools/isolation`; the gate is the evidence-executing part of `core`, not an abstract one). The rejected alternative was an injected `test_runner` seam threaded through `loops/default/loop.py` + both engine entry points — cleaner boundary but heavier plumbing, and unlike Sprint 26's `issue_filer` the sandbox path must activate **automatically** under container mode (driven by `sandbox_runtime_mode()`), not as an opt-in capability, so a runtime self-selecting helper is the better fit.

**Scope / what lands:**
- A single-source-of-truth **format/parse pair** for the `run_tests` result string in `tools/coder_tools/run_tests.py` (so the gate can recover a structured `(exit_code, output)` from the MCP tool's formatted output without a duplicated magic string).
- A new isolation-aware **`run_gate_pytest(path, cwd) -> (exit_code, output)`** in `tools/mcp` that selects sandboxed-provider vs in-process dispatch.
- `core/coder_gate.py` rewired to delegate to it; `_raise_if_sandboxed` and both its call sites deleted.
- Mocked-suite coverage proving the gate ACCEPT/REVISE decisions are identical whether the pytest ran in-process or through a (fake) sandbox provider, across exit codes `0` / `5` (no tests) / nonzero.
- `DEFERRED_VERIFICATION.md` §3 + finding F-GATE-SANDBOX updated to record the gap as **closed by sprint 28** (the *host* re-verification of a real container-sandboxed gate run remains part of sprint 27's V1/V2).

**Out of scope:**
- The `sandbox` (bwrap) substrate stays **inert** (no runtime on this branch) — the helper covers it by construction via `sandbox_runtime_mode()`, but only `container` is host-verified (in sprint 27). No bwrap launch is exercised here.
- **Per-task test selection** (the *other* half of the §3 deferral — the gate runs the whole `src/` suite, not the single task's tests). Explicitly deferred again; this sprint only closes the *sandboxing* gap, not the selection granularity. Note it stays open in `DEFERRED_VERIFICATION.md`.
- Any flag deletion / flip (that is sprint 27). No new flag, no new dependency, no `pyproject.toml`/`sbom.json` change.
- Reusing the persona's already-open provider for the gate — the gate opens its own short-lived provider per pytest run (one `--rm` container per gate verification). Pooling/reuse is a possible later optimization, not this sprint.

**Security Considerations:**
- **No new sanctioned subprocess surface.** The gate's sandboxed pytest is launched by the MCP `stdio_client` (the same mechanism, and the same `container_server_params`, that already launches the coder-tools server for the model's tool loop) — `run_gate_pytest` calls `build_coder_tool_provider`, it does not shell out. The four sanctioned surfaces (`coder_tools` pytest, `issue_io`+`repo_io` `gh`, `worktree` `git worktree`, `git_io` local `git`) are unchanged; `tests/tools/test_subprocess_surfaces.py` must stay green. The in-process `run_pytest` (the `coder_tools` pytest surface) remains the executor **only** on `none`/`worktree`, exactly as today.
- **No degradation to in-process under sandbox modes.** Under `container`/`sandbox`, `run_gate_pytest` dispatches through the provider or raises `IsolationUnavailableError` (from `build_coder_tool_provider` when no runtime/image is available) — it never silently runs the gate's pytest in-process. This preserves the load-bearing isolation invariant that motivated `_raise_if_sandboxed` in the first place; the guard is *replaced by real sandboxed execution*, not merely removed.
- **The single-bind-mount contract is preserved.** The gate's provider mounts only the worktree (`container_server_params`'s one `-v wt:wt` bind) — no second mount, no repo root, no `state/`, no keyring, no network. Same container hardening as the tool path.
- **No new credential path / no keyring import** in `tools/mcp` or `core/coder_gate.py`; `tools/llm/client.py` stays the only keyring importer.

**Risks & Blockers:**
- **Result-string parse must be exact.** The gate branches on `PYTEST_NO_TESTS_COLLECTED == 5`, `!= 0`, and `== 0`; a parse that misreads the exit code silently flips ACCEPT↔REVISE. Mitigation: factor **one** `format_run_tests_result`/`parse_run_tests_result` pair so producer and consumer share the format, and unit-test the round-trip (incl. the `5` and multi-line-output cases). The `run_tests` **tool** wrapper must be reimplemented in terms of the shared formatter so there is no second copy of the prefix string.
- **The "no tests produced" (exit 5) path must survive.** Today `_run_gate_pytest` converts a *missing* `test_path` tree into the `PYTEST_NO_TESTS_COLLECTED` sentinel via an orchestrator-side `Path(test_path).exists()` check **before** running pytest. Keep that existence check orchestrator-side in `run_gate_pytest` — it is valid under container mode because the worktree is a host-FS tree the orchestrator created and bind-mounts (same filesystem it can `Path.exists()`), so only the *pytest execution* moves into the container, not the existence probe. Confirm a missing tree still yields REVISE-"no tests were produced" under sandbox mode (fake provider not even reached).
- **cwd symmetry.** In-process `run_pytest` keys off `Path.cwd()`; the sandbox provider mounts/`-w`s the passed `cwd`. The gate must pass the same worktree cwd it runs under (`Path.cwd()` at gate time == the worktree, since the gate executes immediately after the Coder stage in the same run, exactly as `_CoderToolBackend` assumes). Pass `cwd=Path.cwd()` explicitly from the gate; do not let the two branches disagree on the target tree.
- **The `core → tools/mcp` edge is a *convention* change, not a test change.** The only static core-boundary test (`tests/core/test_boundaries.py`) pins solely the *persona* import rule (`core` imports no concrete persona module beyond `personas.base`) — it does **not** restrict `core → tools/*`, and `core/coder_gate.py` already imports `tools/coder_tools` + `tools/isolation`. So the new edge needs **no** boundary-test update; record it only in the CLAUDE.md module-boundary note for the Coder gate. Keep the edge scoped to `coder_gate.py`; do not import `tools/mcp` elsewhere in `core`.
- **Both gates fixed by the shared helper — zero classic-specific investment.** `CoderGate` (classic, slated for deletion in sprint 27 Task 4) and `RalphCoderGate` share `_run_gate_pytest`; rewiring that one helper fixes both for free. Do not add `CoderGate`-specific code — the user has confirmed the classic Coder is being retired. Fixing the shared path is not classic investment; it is the cheapest correct change.
- **This sprint is fully verifiable on the mocked suite** (fake provider), but the *real* container-sandboxed gate run is only observable on a daemon-bearing host — that observation is **sprint 27's V1(complete)/V2**, not a task here. The green gate for *this* sprint is the mocked suite + lint/format/audit/sbom; the host proof is recorded in 27.
