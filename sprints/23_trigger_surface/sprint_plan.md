### FILEPATH: /sprints/23_trigger_surface/sprint_plan.md

**Sprint Goal:** Build **Phase 5 piece 2 — the trigger surface**: a FastAPI webhook server that turns a GitHub event into a loop-engine run. On an issue **labeled `agent-action`** or a **slash-command comment** (`/agent-run`), the server HMAC-verifies the delivery, parses the issue's title+body into a run's requirements, and **dispatches a real graph run of the existing default loop** through an injectable `RunDispatcher` seam. This is a **capability slice** (mirroring 22b): it lands the new `src/loop_engine/trigger/` package (app + parser + dispatcher), a shared run-starter refactored out of the CLI, the FastAPI runtime dependency, hermetic `TestClient` tests, a boundary static test, and docs. It does **NOT** deploy the server (no uvicorn, no hosting), does **NOT** add a fourth subprocess surface (dispatch is in-process), and does **NOT** wire the maintenance/bootstrap factory flows (pieces 3–4, Sprints 24+). Live verification is deferred to a daemon-bearing host.

**Dependencies:** Sprint 22b (`7b46227`→`5bc3811`) landed the github MCP server + `tools/repo_io` factory verbs — piece 1, the foundation this trigger will eventually chain into the factory flows. This sprint does **not** call those verbs yet (that is the maintenance/bootstrap flows); it proves the webhook→run bridge against the **existing default loop**. Precedents to mirror exactly: `src/loop_engine/cli.py::run` (the canonical "build `State` from `human_input` → `worktree_run` → `_select_engine()(loop, state, client, start_index=0)`" invocation this sprint factors into a shared runner and reuses); `tests/tools/test_*` boundary tests (the no-keyring / no-direct-write assertion style, extended here to the new package). **Read the "Context (locked decisions)" section before starting** — the Architect (Opus) settled all six gating decisions in the 23 planning pass; do not re-open them.

**Context (locked decisions — do not re-open):**
- **Deliverable boundary = capability slice, real run against the existing default loop** *(locked 2026-07-09, Opus/Architect, 23 planning pass)*. The webhook dispatches a genuine `run_loop`/`run_graph_loop` of the **default** loop, with `human_input` = the issue's title+body. It ships app + parser + dispatcher + shared runner + FastAPI dep + hermetic tests + docs. It does **NOT** deploy (no uvicorn, no hosting decision), and does **NOT** wire the maintenance flow (clone→branch→green-gate→push→PR) or bootstrap flow — those are Sprints 24+ and remain the callers that will use `tools/repo_io`. Do not introduce any `tools/repo_io` call, any clone, or any factory-verb chaining in this sprint.
- **Dispatch = an injectable `RunDispatcher` seam with a single in-process implementation; NO fourth subprocess surface** *(locked 2026-07-09, Opus/Architect)*. The webhook must ACK GitHub within seconds, but a run takes minutes, so dispatch is asynchronous. The chosen shape: a `RunDispatcher` Protocol (one method), with 23 shipping **one** implementation — `InProcessDispatcher` — that runs the loop **in-process** on a worker thread by calling the shared runner directly (exactly as `cli.run` does). It does **NOT** spawn `loop-engine run` as a subprocess. The "**exactly three sanctioned subprocess surfaces**" invariant (coder_tools' pytest, `issue_io`'s & `repo_io`'s `gh`, `worktree`'s `git worktree`) is **unchanged** — the dispatched loop's own worktree/gh/pytest calls are those existing surfaces reached from a new caller, not a new surface. The seam exists so a durable-queue / process-isolated implementation can swap in behind it in a later sprint without touching the app. **Do NOT** add `subprocess`/`Popen`/`os.system`/`os.exec*` anywhere in `trigger/`.
- **Web dependency = FastAPI (pinned); uvicorn DEFERRED with hosting** *(locked 2026-07-09, Opus/Architect)*. Add FastAPI as a pinned **runtime** dependency; test the ASGI app hermetically via Starlette's `TestClient` (no bound port, no daemon). Do **NOT** pin uvicorn — the ASGI runner / port binding is part of the deferred hosting decision and lands with deployment in a later sprint. `TestClient` needs `httpx`, added as a **dev/test** dependency (hatch default env), not a runtime dependency. This is loop-engine's first web dependency; the pin must be audit-green and compatible with the existing `pydantic==2.13.4`.
- **Webhook auth = HMAC over the raw body, secret from an env var, fail-closed** *(locked 2026-07-09, Opus/Architect)*. Signature verification of GitHub's `X-Hub-Signature-256` (HMAC-SHA256 over the **raw request bytes**, constant-time compare) is **in-scope and mandatory** — a webhook receiver without it is an open run-trigger. The shared secret comes from an environment variable (`LOOP_ENGINE_WEBHOOK_SECRET`); if it is unset the app **refuses to construct / start** (fail-closed) — it never falls open to unsigned requests. This touches **no** invariant: the secret is not a credential the keyring owns, so `trigger/` imports no `keyring` and the "only `tools/llm/client.py` imports keyring" invariant is untouched. Infisical / non-interactive PAT provisioning stays the orthogonal, deferred decision.
- **Trigger grammar = two bare-verb triggers; requirements = issue title+body (unified)** *(locked 2026-07-09, Opus/Architect)*. Two trigger types, both bare "start it" verbs carrying no payload: (a) an `issues` event with action `labeled` whose label name is **`agent-action`**; (b) an `issue_comment` event with action `created` whose comment matches the slash command **`/agent-run`** (line-prefix match). For **both**, `human_input` = the issue's `title` + "\n\n" + `body`. The `issue_comment` body is only the trigger token, never the payload. Every other event/action/label/command — including GitHub's `ping` — is a **no-op** (return 2xx without dispatching, so redeliveries and unrelated hooks are silently and safely ignored). Budget = the existing `DEFAULT_BUDGET_USD` (5.00); loop = `"default"`.
- **Placement = new top-level `src/loop_engine/trigger/` package; orchestrator-level caller** *(locked 2026-07-09, Opus/Architect)*. The trigger surface is a new top-level package (sibling to `core/`, `tools/`, `personas/`, `mcp_servers/`). It is an **orchestrator-level caller** — like `cli.py` it may import `core` (engine), the concrete default loop, `State`, `LLMClient`, and `worktree`; it is **not** a `tool` (do not place it under `tools/`) and not an MCP server. Its enforced boundary posture, asserted by a new static test: **imports no `keyring`, performs no direct file writes** (all persistence goes through the engine / `state_io`), and **adds no subprocess surface**. **No CLI `serve` subcommand in 23** (it would need uvicorn) — the deliverable is the importable ASGI `app` + the dispatcher.

**Security Considerations:** The webhook is a **run-trigger** — a valid request causes the engine to execute model-driven code — so HMAC verification is the trust boundary and must be airtight: read the **raw body bytes before any JSON parsing** (FastAPI `await request.body()`), compute `hmac.new(secret, raw, sha256)`, and compare against the `X-Hub-Signature-256` header with `hmac.compare_digest` (never `==`). Missing header, malformed header, or mismatch → reject (401/403) and **do not dispatch**. Missing `LOOP_ENGINE_WEBHOOK_SECRET` at construction → fail-closed (raise; the app does not start). The secret is never logged and never appears in a source file, config, or test fixture — tests use an obviously-fake literal (e.g. `"test-webhook-secret-not-a-real-credential"`) so `gitleaks` stays clean. `RunRequest` is a Pydantic model with `extra="forbid"`; it has an invalid-input-rejection test (DoD). No credential enters `trigger/` (no `keyring` import); no direct file write (no `open`/`write_text`/`write_bytes` — the engine owns snapshotting via `state_io`); **no new subprocess surface** (dispatch is in-process; the sanctioned-surfaces count stays **three**). The dispatched run still executes under the sandboxed-devcontainer operating assumption. Redeliveries and the label+comment double-fire are de-duplicated in-memory (see Task 3) so one issue never launches two concurrent runs.

**Risks & Blockers:**
- **First web dependency — pin + SBOM + audit are load-bearing.** FastAPI pulls `starlette`, `anyio`, and (already-pinned) `pydantic` transitively. Pin FastAPI to a current stable release compatible with `pydantic==2.13.4`, verify `hatch run audit` is green (no high/critical CVE at merge), and **regenerate `sbom.json` via `hatch run sbom`** (the DoD gate for any `pyproject.toml` dep change). `httpx` is a **dev/test** dep only (TestClient) — it must not leak into runtime `dependencies`.
- **Raw-body ordering.** HMAC must be computed over the exact bytes GitHub signed. If any middleware or handler consumes/reparses the body first, the signature will not match. Verify against the raw `await request.body()` and only then parse JSON. Add a test with a known secret + precomputed signature proving a valid delivery passes and a tampered body fails.
- **The dispatched loop is synchronous and blocking.** `run_loop` blocks (subprocess + network). Running it directly on the event loop would stall the webhook handler. `InProcessDispatcher` must run it **off the event loop** (a worker thread — `asyncio.to_thread` / a `ThreadPoolExecutor` / `threading.Thread`), so the handler returns 202 immediately. App tests inject a **fake dispatcher** so no real loop ever runs in CI (hermetic); the `InProcessDispatcher` itself is tested with the shared runner patched.
- **No live end-to-end verification on this branch.** No host here can bind a port, receive a real GitHub delivery, or run a full loop with `gh` auth + network. Coverage is hermetic only (TestClient + injected/faked dispatcher + patched runner). Append the live check — real webhook delivery → real default-loop run — to `sprints/DEFERRED_VERIFICATION.md`, together with the deferred uvicorn/hosting bring-up.
- **Do not scope-creep into the factory flows.** No `tools/repo_io` call, no clone, no branch/PR, no `git push` surface. The maintenance flow (piece 3) is the next sprint and owns the local-git subprocess-surface decision deferred from 22b.

- **Task 2: `RunRequest` model + GitHub-event parser (`trigger/parse.py`)**

- **Task 3: `RunDispatcher` seam + `InProcessDispatcher` (`trigger/dispatch.py`)**
  - **Description:** Add `src/loop_engine/trigger/dispatch.py` with a `RunDispatcher` Protocol — `async def dispatch(self, request: RunRequest) -> None` (fire-and-forget: schedule the run and return, so the handler can 202 immediately) — and one implementation, `InProcessDispatcher`, that runs the loop **in-process, off the event loop**: it de-duplicates on `(repo_full_name, issue_number)` against an in-memory set of active runs (a redelivered or label+comment double event for an already-running issue is dropped, logged, and does not launch a second run), then schedules `runner.run_new(request.human_input, budget_usd=request.budget_usd, loop_name=request.loop_name)` on a worker thread (`asyncio.to_thread` or an executor) so the blocking loop never stalls the event loop, clearing the active-set entry in a `finally`. It logs run start/end (no secret, no PII beyond the issue reference). Real durable queuing / cross-restart rate-limiting is explicitly deferred — this is best-effort, in-memory, single-process. Do **not** add any subprocess call.
  - **Target Files:** `src/loop_engine/trigger/dispatch.py`, `tests/trigger/test_dispatch.py` (new)
  - **Acceptance Criteria:** `InProcessDispatcher.dispatch` invokes `runner.run_new` (patched to a fake) exactly once per distinct issue with the request's `human_input`/`budget_usd`/`loop_name`, off the event loop; a second `dispatch` for the same `(repo, issue)` while the first is still active does **not** invoke the runner again; after a run completes the issue can be dispatched again. `dispatch` returns without waiting for the (patched, blocking) run to finish. `RunDispatcher` is a `typing.Protocol`. `trigger/dispatch.py` contains no `subprocess`/`Popen`/`os.system`/`os.exec*` and imports no `keyring`. Lint/format clean.

- **Task 4: FastAPI app — HMAC verify → parse → dispatch (`trigger/app.py`)**
  - **Description:** Add `src/loop_engine/trigger/app.py` exposing an app **factory** `create_app(dispatcher: RunDispatcher | None = None) -> FastAPI` (the factory injects the dispatcher so tests pass a fake; default = `InProcessDispatcher()`), plus a module-level `app = create_app()` for eventual hosting. Read `LOOP_ENGINE_WEBHOOK_SECRET` **at construction**; if unset, raise (fail-closed — the app does not start). Single route `POST /webhook`: (1) `raw = await request.body()`; (2) verify `X-Hub-Signature-256` via `hmac.compare_digest(expected, "sha256=" + hmac_sha256(secret, raw))` — missing/malformed/mismatch → `401` (do not dispatch, do not parse further); (3) parse JSON from `raw`, read the `X-GitHub-Event` header, call `parse_event`; (4) `None` → `204` no-op; a `RunRequest` → `await dispatcher.dispatch(req)` then `202 Accepted`. Optionally a `GET /health` → `200` for liveness (harmless, aids later hosting). No secret is ever logged.
  - **Target Files:** `src/loop_engine/trigger/app.py`, `tests/trigger/test_app.py` (new; uses Starlette `TestClient` with an injected fake dispatcher + `LOOP_ENGINE_WEBHOOK_SECRET` monkeypatched to a fake literal)
  - **Acceptance Criteria (all via `TestClient`, hermetic):** a correctly-signed labeled-`agent-action` delivery → `202` and the fake dispatcher receives the expected `RunRequest`; a correctly-signed `/agent-run` comment → `202`; a correctly-signed `ping`/unrelated event → `204` and the dispatcher is **not** called; a delivery with a wrong/absent signature → `401` and the dispatcher is **not** called; constructing the app with `LOOP_ENGINE_WEBHOOK_SECRET` unset raises (fail-closed). The signature check uses `hmac.compare_digest` over the raw body. Lint/format clean.
