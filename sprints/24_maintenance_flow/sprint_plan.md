### FILEPATH: /sprints/24_maintenance_flow/sprint_plan.md

**Sprint Goal:** Build **Phase 5 piece 3 — the maintenance flow**: the **first caller to chain `tools/repo_io`'s factory verbs into an end-to-end run over a target repo**. `flows/maintenance` clones a target repo, cuts a local feature branch, runs the **existing default loop with its cwd inside the clone** (so the target's `CLAUDE.md` + `.agent/STATE.md` are absorbed by the existing readers), and — **only on a green gate** — commits, `git push`es the branch, and opens a PR against `develop`. This is a **capability slice** (mirroring 22b/23): it lands a new **`tools/git_io`** module (the flow-forced local-git write surface deferred from 22b), a new **`runner.run_in_tree`** entrypoint, a new top-level **`src/loop_engine/flows/maintenance/`** package behind an injectable run-step seam, hermetic tests, a package boundary static test, and docs. **Auto-merge stays prohibited** — there is no merge verb and `open_pr` is the terminal GitHub call. It does **NOT** wire a CLI subcommand, does **NOT** wire the trigger surface (23) to dispatch this flow, and does **NOT** build the bootstrap flow (piece 4). Live clone→push→PR verification is deferred to a daemon-bearing host.

**Dependencies:** Sprint 22b (`7b46227`→`5bc3811`) landed `tools/repo_io`'s four factory verbs (`create_repository`, `clone_repo`, `create_branch`, `open_pr`) — piece 1, the foundation this flow chains. Sprint 23 (`5ff8c02`→`212beeb`) landed `src/loop_engine/runner.py` (`run_new`) and the `src/loop_engine/trigger/` package — piece 2, the precedent for a new top-level orchestrator-level caller with an injectable seam + a boundary static test. Precedents to mirror exactly: `tools/worktree/manager.py::_git` (the fixed-argv/`shell=False`/`timeout` local-git subprocess posture `git_io` reuses); `tools/repo_io/github.py::_validate_clone_dest` (the clone-tree path validation `git_io` reuses); `src/loop_engine/runner.py::run_new` (the loop-build/State-init `run_in_tree` reuses, minus the `worktree_run` wrapper); `src/loop_engine/trigger/dispatch.py` (the `Protocol` seam shape) and `tests/trigger/test_boundaries.py` (the no-keyring / no-direct-write boundary-test style, extended here — with the deliberate difference that `flows/` *does* reach a new subprocess surface, `tools/git_io`, which the test asserts is the **only** one). **Read the "Context (locked decisions)" section before starting** — the Architect (Opus) settled all gating decisions in the 24 planning pass; do not re-open them.

**Context (locked decisions — do not re-open):**
- **Deliverable boundary = capability slice, real run against the existing default loop over a real clone** *(locked 2026-07-09, Opus/Architect, 24 planning pass)*. The flow clones a real target repo and dispatches a genuine `run_loop`/`run_graph_loop` of the **default** loop with cwd inside the clone. It ships `tools/git_io` + `runner.run_in_tree` + the `flows/maintenance` package + hermetic tests + a boundary test + docs. It does **NOT** add a CLI subcommand (`loop-engine maintain`), does **NOT** wire the trigger surface to dispatch this flow, and does **NOT** build the bootstrap flow (piece 4). Those are Sprints 25+. Do not introduce a CLI entrypoint or any `trigger/` change in this sprint.
- **Git-write surface = a NEW dedicated `tools/git_io` module; this is honestly a FOURTH sanctioned subprocess surface** *(locked 2026-07-09, Opus/Architect)*. Local git writes (`checkout -b`, `add`, `commit`, `push`) in the cloned tree are the flow-forced surface deferred from 22b. It lives in its **own** module — NOT bolted onto `tools/repo_io` (which shells `gh` = the remote GitHub API; local `git` on a working tree is a different mechanism and a different responsibility) and NOT onto `tools/worktree` (which is strictly the orchestrator's *own* per-run isolation keyed off `run_id`; the target clone is a foreign tree). `git_io` mirrors `worktree._git`'s posture exactly (fixed argv, `shell=False`, `timeout`, `check`-handling) and validates the tree path by reusing `repo_io._validate_clone_dest`. The "exactly three sanctioned subprocess surfaces" invariant becomes **four** (coder_tools' pytest, `issue_io`'s & `repo_io`'s `gh`, `worktree`'s `git worktree`, **and `git_io`'s local `git`**) — update `CLAUDE.md` and the subprocess-surface boundary test to say **four** truthfully. **Rationale (do not re-litigate):** the invariant exists to make new execution surfaces conspicuous and reviewed; a clean, named, tested `git_io` module serves that better than stretching "git has two consumers" to hide the count. `git push` in a `gh`-cloned tree rides `gh`'s credential helper — **no new credential path**, no `keyring` touch.
- **Execution model = an injectable run-step seam, wired in the concrete path to a new `runner.run_in_tree`; absorb = run in the clone's cwd** *(locked 2026-07-09, Opus/Architect)*. `flows/maintenance` takes an injectable "run step" (a `Protocol`/callable seam, like 23's `RunDispatcher`) so tests inject a fake and the Ralph-vs-default refinement can swap in later without touching the flow. The concrete run step is **`runner.run_in_tree`** — it runs the default loop with cwd pinned to the clone tree and does **NOT** open `worktree_run` (the clone is *itself* the isolation boundary; `run_new`'s orchestrator-worktree wrapper would chdir into the wrong tree when `LOOP_ENGINE_ISOLATION` is on). "**Absorb `CLAUDE.md` + `.agent/STATE.md`**" = run with cwd inside the clone so the *existing* readers pick them up (`agent_state.read_scratchpad()` reads `.agent/STATE.md` from cwd; `CLAUDE.md` is just a file in the tree) — **no new absorb code**.
- **Green gate = `coder_tools`' pytest on the clone; push + PR are green-ONLY; red ⇒ no push, no PR** *(locked 2026-07-09, Opus/Architect)*. After the inner run, the flow runs the target repo's test suite through the existing sanctioned `coder_tools.run_tests` pytest surface against the clone tree. **Green** → `git_io.commit_all` + `git_io.push_branch` + `repo_io.open_pr`. **Red** → return a failing status; **no commit, no push, no PR**. `open_pr`'s base defaults to **`develop`**, overridable. Auto-merge stays impossible — no merge verb exists and `open_pr` is the terminal GitHub call.
- **Placement = new top-level `src/loop_engine/flows/` package (sibling of `trigger/`, `cli.py`); orchestrator-level caller** *(locked 2026-07-09, Opus/Architect)*. `flows/maintenance` is an **orchestrator-level caller** — like `cli.py`/`trigger/` it may import `core`/the default loop/`State`/`LLMClient`/`runner`/`repo_io`/`git_io`; it is **not** a `tool` and not an MCP server. Its enforced boundary posture, asserted by a new static test: **imports no `keyring`, writes no files directly** (all local-git writes go through `tools/git_io`; all state persistence through the engine/`state_io`), and introduces **no subprocess surface of its own** — it *calls* `tools/git_io` (which owns the sole new surface). The `flows/` boundary test differs from `trigger/`'s in exactly this: `git_io` is the **one** permitted new surface, asserted present-and-only.

**Security Considerations:** The maintenance flow **executes model-driven code over a foreign repository and then pushes the result to GitHub**, so two boundaries are load-bearing. (1) **Clone-tree confinement:** `clone_repo`'s destination and every `git_io` tree argument are validated with `repo_io._validate_clone_dest` (relative, no `..`, no symlink escape out of the run tree) so a crafted target name/path can never write outside the run tree; `git_io`'s argv is fixed and `shell=False`, so a branch/remote/path string is never shell-interpreted. (2) **Push is green-gated and non-merging:** nothing is pushed unless `coder_tools`' pytest passes on the clone, and the terminal GitHub op is `open_pr` — there is no merge verb, so the flow can never auto-merge into `develop`. No credential enters `flows/` or `git_io` (no `keyring` import); `git push` rides `gh`'s clone-established credential helper. `MaintenanceRequest` is a Pydantic model with `extra="forbid"` and has an invalid-input-rejection test (DoD). The inner run still executes under the sandboxed-devcontainer operating assumption. `git_io` never logs a remote URL that could embed a token, and no secret appears in a source file, config, or test fixture (tests use obviously-fake slugs/paths and a local temp git repo, so `gitleaks` stays clean).

**Risks & Blockers:**
- **`run_in_tree` must NOT re-enter `worktree_run`.** `run_new` wraps the loop in `worktree_run(run_id)`, which chdirs into the *orchestrator's own* worktree when `LOOP_ENGINE_ISOLATION` is set — the wrong tree for maintenance. `run_in_tree` runs the loop with cwd pinned to the **clone** and opens **no** `worktree_run`. Add a test that, with isolation flagged on, the loop still runs in the clone tree (not `.worktrees/<run_id>`). Pin `state_io`'s state root deliberately if snapshots must land outside the clone (decide + document in Task 2; default: snapshots follow the engine's normal behavior relative to the clone cwd unless a reason to pin emerges).
- **Green gate must key off the clone, not the orchestrator tree.** `coder_tools.run_tests` keys off `Path.cwd()` / a validated path — ensure it runs against the **clone** tree, and that a **red** result short-circuits before any `git_io` write. Test both branches: green ⇒ commit/push/open_pr called with the right args and order; red ⇒ none called and a failing status returned.
- **The fourth subprocess surface is a real boundary change.** `CLAUDE.md`'s "exactly three sanctioned surfaces" clause and the subprocess-surface boundary test both currently assert **three**. Both must move to **four** in the same sprint, naming `git_io`'s local `git`. Do not leave the doc and the test disagreeing.
- **`git_io` tests are hermetic against a real temp git repo.** `git` is available in CI (it already backs `tools/worktree`). Test `git_io` end-to-end against a `tmp_path` `git init` repo (checkout/commit/push to a local bare remote), so no network/GitHub is needed. The flow-level tests inject a **fake run step** + patch/inject `repo_io`/`git_io` so no real clone, loop, or push runs in CI.
- **No live end-to-end verification on this branch.** No host here can `gh repo clone` a real repo, run a full loop with `gh` auth + network, and push a PR. Coverage is hermetic only. Append the live check — real clone → real default-loop run → green gate → real push → real PR against `develop` — to `sprints/DEFERRED_VERIFICATION.md`.
- **Do not scope-creep.** No CLI subcommand, no trigger wiring, no bootstrap flow, no new dependency (git_io adds none — **`sbom.json` unchanged**). The carried-forward `python` vs `sys.executable` `loop_engine.mcp.json` nit is orthogonal to this flow — leave it (carry-forward, below).
