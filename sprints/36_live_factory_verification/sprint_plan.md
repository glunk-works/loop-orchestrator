### FILEPATH: /sprints/36_live_factory_verification/sprint_plan.md

**Sprint Goal:** Prove that **the factory actually works** ‚Äî against real GitHub, with real side effects ‚Äî and fix the defect that would otherwise make that proof worthless. `sprints/DEFERRED_VERIFICATION.md` ¬ß5 (`github_server` live factory verbs), ¬ß8 (bootstrap: create ‚Üí clone ‚Üí scaffold ‚Üí push `main` ‚Üí create `develop`) and ¬ß7 (maintenance: clone ‚Üí run ‚Üí gate ‚Üí push ‚Üí PR) are the only checks in this repo with real side effects on GitHub, and together they are the only evidence that the product's central claim is true. After 25 sprints it is still unverified. But **BL-21 says `flows/bootstrap` ships repos with an unprotected `main`** ‚Äî a defect *in the very flow ¬ß8 exists to verify* ‚Äî so this sprint **fixes BL-21 first and verifies the fixed flow**, chaining one disposable scratch repo through all three sections: ¬ß8 births it, ¬ß7 maintains it.

**Out of scope:** ¬ß6 (live webhook) ‚Äî it is the one check that genuinely needs a routable address, and it is correctly owned by **BL-24**; do not fold it in on the strength of FD1. ¬ß1 (caching + USD smoke) ‚Äî owned by **BL-3**. Any change to loop-engine's *own* ruleset, the eight required checks, or the four workflows. Adding CI to the scaffold (see FD4 ‚Äî tempting, and explicitly deferred). Any Dependabot PR. Any new dependency (so **no `sbom` regeneration**).

**Context (locked decisions ‚Äî do not re-open):**

- **FD1 ‚Äî This devcontainer IS the host. ¬ß5/¬ß7/¬ß8 never needed a "daemon-bearing" one; that premise is stale and it is what deferred them for 25 sprints** *(locked 2026-07-14, Opus/Architect, sprint 36 planning pass; user-confirmed)*. `DEFERRED_VERIFICATION.md` ¬ß5 states "this devcontainer has no `gh` auth and no network" and all three sections demand "a daemon-bearing host". **Measured, not assumed:** `gh auth status` is green here (account `Seuss27`, fine-grained PAT), the network resolves, and the Anthropic key is present in the keyring (`EncryptedFileKeyring`, `loop-engine/anthropic_api_key`). What ¬ß5/¬ß7/¬ß8 actually require is *authenticated `gh` + network + a disposable scratch repo* ‚Äî **none of which is a daemon**. The word "daemon-bearing" was inherited from **¬ß6**, which is the only section that must *bind a port GitHub can reach*, and ¬ß6 is out of scope. **Correct the premise in `DEFERRED_VERIFICATION.md` as part of Task 7** ‚Äî leaving it there re-defers these checks the next time someone reads the file.
- **FD2 ‚Äî The one and only blocker is a single PAT permission, and the same grant unlocks both the BL-21 fix and the verification** *(locked 2026-07-14, Opus/Architect; measured)*. Probed live against the real API:
  ```
  POST orgs/glunk-works/repos                    -> 403   X-Accepted-Github-Permissions: administration=write
  POST repos/glunk-works/loop-engine/rulesets    -> 403   X-Accepted-Github-Permissions: administration=write
  GET  orgs/glunk-works/rulesets                 -> 403   Resource not accessible by personal access token
  ```
  The PAT can read/write Actions and read org secrets (sprint 34) but has **no `administration` permission at all**. `repo_io.create_repository` shells `gh repo create`, which hits the first endpoint ‚Äî so **¬ß5 and ¬ß8 cannot run today, and the BL-21 fix cannot run either, for the same reason.** This is Task 1, it is a **human action**, and it is a ~2-minute grant. **Nothing else in this sprint can proceed until it lands.**
- **FD3 ‚Äî BL-21's fix is impossible on a private repo under the org's Free plan, so `flows/bootstrap` must default to PUBLIC** *(locked 2026-07-14, Opus/Architect; user-confirmed)*. BL-21 correctly notes repo-level rulesets "*are* free **on public repos**" ‚Äî that qualifier is load-bearing and was not followed through. Under Free, rulesets and branch protection on **private** repos require GitHub Team, so a `create_ruleset` call from `run_bootstrap` would hit **the same 403 that killed the org-level fix**, on the default path, for every repo the factory makes. But `BootstrapRequest.private` defaults to **`True`** (`src/loop_engine/flows/bootstrap/flow.py`). The sketch in BL-21 therefore does not work as written. **Resolution: flip the default to `private=False`.** Protection becomes the invariant the factory guarantees; privacy becomes an **opt-in that knowingly forfeits it**. Corroborating evidence that this matches reality: all five repos in `glunk-works` are already **public**, and loop-engine's own ruleset ‚Äî the proof that repo-level rulesets are free at all ‚Äî exists *only because loop-engine is public*.
- **FD4 ‚Äî The shipped ruleset must require ZERO status checks. A generated repo has no CI whatsoever** *(locked 2026-07-14, Opus/Architect; verified by reading the scaffold)*. `tools/scaffold/templates/` contains `CLAUDE.md`, `pyproject.toml.tmpl`, `README.md.tmpl`, `gitignore.tmpl`, `src/__PKG_NAME__/__init__.py.tmpl`, `tests/test_smoke.py.tmpl` ‚Äî and **no `.github/workflows/` at all**. So a required status check in the shipped ruleset would be **permanently pending and the repo could never merge anything**: the exact deadlock BL-21 warns about, and the mirror-image of BL-11 (there, gates that looked required and blocked nothing; here, gates that block everything and can never pass). Ship `deletion` + `non_fast_forward` + `pull_request`, and **nothing else**. Adding a CI workflow to the scaffold so that a check *could* be required is a real and defensible idea ‚Äî it is **deliberately out of scope**, because it is a new feature wearing a bugfix's clothes and it would make this sprint's live verification depend on a workflow that has never run anywhere.
- **FD5 ‚Äî The ruleset targets BOTH `refs/heads/main` AND `refs/heads/develop`** *(locked 2026-07-14, Opus/Architect; user-confirmed)*. BL-21 did not ask *which branch*, and the obvious answer is wrong. Mirroring loop-engine's own ruleset (which targets exactly `refs/heads/main`) would protect the branch a generated repo **never writes to** and leave wide open the one the factory **actually uses**: `flows/maintenance` opens its PRs with `base=develop`, and the scaffolded `CLAUDE.md` instructs generated repos that "PRs target the integration branch (`develop`), never `main`/`master` directly". Protecting `main` alone would be a gate on the door nobody uses.
- **FD6 ‚Äî `create_ruleset` is orchestrator-only. The github MCP server stays at exactly FOUR verbs** *(locked 2026-07-14, Opus/Architect)*. BL-21 flags that exposing it would widen the pinned four-verb set to five, and that the "four verbs, pairwise disjoint" assertion in `tests/tools/test_mcp_provider.py` is load-bearing. It is, and there is already **precedent for exactly this shape**: `repo_io.resolve_repo_slug` is a `repo_io` verb that is deliberately *not* an MCP verb, for the same reason (orchestrator-side introspection, never a model tool). `create_ruleset` follows it. `mcp_servers/github_server` is **not touched**, and the pairwise-disjoint test must still assert **four**.
- **FD7 ‚Äî `create_ruleset` runs LAST in the bootstrap chain, after `create_branch(develop)`** *(locked 2026-07-14, Opus/Architect)*. Ordering is load-bearing in *both* directions here. It cannot run before `push_branch(main)` ‚Äî a `pull_request` rule on `refs/heads/main` would reject the scaffold's own initial direct push, and bootstrap would deadlock on its first commit. Running it before `create_branch(develop)` raises a live question about whether the API branch-creation is itself blocked by the rule. Installing it **last** sidesteps that question entirely and is deterministic. The chain becomes: `create_repository` ‚Üí `clone_repo` ‚Üí `checkout_branch(main)` ‚Üí `write_skeleton` ‚Üí `commit_all` ‚Üí `push_branch(main)` ‚Üí `create_branch(develop, base=main)` ‚Üí **`create_ruleset`**.
- **FD8 ‚Äî ¬ß8's output is ¬ß7's input; one scratch-repo lifecycle serves all three sections** *(locked 2026-07-14, Opus/Architect; user-confirmed)*. ¬ß7 needs a repo with a real, runnable test suite; ¬ß8 *produces* exactly that (the scaffold ships `tests/test_smoke.py` and a `pyproject.toml`). Chaining them is not merely an economy ‚Äî **it is the factory's actual claim**: that it can *birth* a protected repo and then *maintain* it through a gated PR. Run them in that order, against the same repo.
- **FD9 ‚Äî A ruleset that EXISTS is not a ruleset that BLOCKS. Prove the gate is functional, not merely present** *(locked 2026-07-14, Opus/Architect)*. This is the whole of **BL-16**, and of BL-11, BL-18 and BL-20: this repo's recurring defect is *a check that verified the wrong property while reporting success*. `gh api repos/{o}/{r}/rulesets` returning our ruleset proves it was **created**. It does not prove it **rejects anything**. Task 5's acceptance therefore requires an **attempted direct push to `main` on the scratch repo that is observed to fail** ‚Äî the negative test is the deliverable, not a bonus.

**Security Considerations:** This sprint has **real, irreversible side effects on real GitHub** ‚Äî it is the only sprint in this repo's history of which that is true, and every guardrail below exists because of it.

- **The scratch repo is PUBLIC** (FD3 ‚Äî it must be, or the ruleset cannot be installed). It is world-readable for its whole life. **Nothing sensitive may enter it**: no key, no token, no real requirements document. ¬ß7 runs a real default loop *inside* it and commits the result, so the loop's artifacts land in a public repo ‚Äî seed it with a trivial, throwaway task.
- **The PAT gains `administration=write` (FD2), which is a genuine privilege escalation**: it can create *and delete* repositories across `glunk-works`, and rewrite rulesets ‚Äî including **loop-engine's own**. Scope it to the org, and treat every `create_repository` / `create_ruleset` / repo-delete call in this sprint as pointed at a live weapon. **Never run any of this sprint's flows against `glunk-works/loop-engine`.** Hard-code the scratch repo's name and read it back before every destructive call.
- **Teardown is part of the sprint, not the cleanup after it.** The scratch repo, its branches and its PR are deleted at the end (Task 7). A `DEFERRED_VERIFICATION` section that leaves debris behind has not been discharged.
- **¬ß7 spends real money.** Budget the inner run at **$5.00** (the comparable V2 run cost $2.24). The engine's budget ceiling is the control; do not raise it mid-run.
- No new subprocess surface, no new credential path, and **no new keyring import** ‚Äî `create_ruleset` shells the already-authenticated `gh`, exactly as its four `repo_io` siblings do. The five sanctioned subprocess surfaces are unchanged. Still **no merge verb** exists, so auto-merge remains impossible for the factory's output as well as its input.

**Risks & Blockers:**
- **Task 1 (the PAT grant) blocks literally everything else** (FD2). Do not start Task 2 hoping the grant lands later ‚Äî the tests for `create_ruleset` are hermetic and *will* pass without it, which means a green suite proves nothing about whether the verb can actually run. That is the BL-16 trap, in advance.
- **Do not template loop-engine's eight required checks into the generated ruleset** (FD4). It is the single most likely mistake in this sprint and it produces a repo that can never merge anything ‚Äî a defect that would ship to every repo the factory ever makes.
- **`flows/bootstrap` and `tools/repo_io` are `src/`** ‚Äî this sprint's PR requires an `architect-review` from a **fresh session** (`/handoff` ‚Üí new session ‚Üí `/resume` ‚Üí `/code-review` ‚Üí post). `/model opus` mid-session is not a review session.
- **The live sections are not `pytest`.** ¬ß5/¬ß7/¬ß8 are executed protocols whose evidence is *observed output*, not a green suite. Record what was **observed**, not what was expected ‚Äî and if a section fails, that failure is a finding and a sprint deliverable, not a reason to retry until it passes.
- **A failed live check must not be "fixed" by weakening the check.** If ¬ß7's inner loop fails to converge, that is a ¬ß7 result.
- Bootstrap's `private=False` flip changes a default that a caller may be relying on. Grep for `BootstrapRequest(` across the repo and the trigger/flows callers before changing it.
