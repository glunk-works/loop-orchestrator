**Security Considerations:** The issue server **re-fronts human-escalation GitHub Issues** â€” three boundaries are load-bearing. (1) **No credential in-process:** `issue_io_server` imports **no** `keyring` and makes **no** `gh` call at import time (discovery/`list_tools` stays offline/hermetic); the actual `gh` call rides `gh`'s own authentication, exactly as `tools/issue_io` and `github_server` already do â€” no new credential path, the `tools/llm/client.py`-only-imports-keyring invariant is untouched. (2) **Orchestrator-invoked, never a model tool:** the issue verbs are reached **only** through `build_issue_provider()`, a consumer-scoped provider structurally separate from `build_coder_tool_provider` (which names only `coder_tools`), so the agentic Coder's tool loop can never name or reach `{create_issue, read_issue}` â€” assert the three server tool-sets (`coder_tools` `{read_file,list_files,grep,run_tests}` / `github` `{create_repository,clone_repo,create_branch,open_pr}` / `issue` `{create_issue,read_issue}`) are **pairwise disjoint**. (3) **Untrusted-input surface is unchanged and already-bounded:** answer parsing operates on **human-supplied issue comments** â€” this was already true; the pure `parse_issue_answers` is the same regex-bounded parse moved verbatim (no `eval`, no shell), and the issue body/title are rendered only from run state (`run_id`, question text) â€” no secret is embedded and no token-bearing URL is logged. `create_issue`'s `title`/`body`/`label` are first-party rendered strings, not attacker-controlled. No secret appears in any source file, config, or test fixture (tests use obviously-fake issue numbers/URLs and stubbed `gh` output, so `gitleaks` stays clean). The two `gh`-touching verbs carry **no merge/close-write** capability beyond what `issue_io` already had.
