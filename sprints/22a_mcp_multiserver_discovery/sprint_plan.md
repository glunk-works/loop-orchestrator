### FILEPATH: /sprints/22a_mcp_multiserver_discovery/sprint_plan.md

**Sprint Goal:** Pay down cross-cutting follow-up **#3** (`loop_engine.mcp.json`-driven multi-server MCP discovery) â€” the first sub-sprint of **Phase 5**. Convert the hard-coded single-server MCP wiring (`build_coder_tool_provider`, which today constructs one `StdioServerParameters` for the coder-tools server) into a **config-file-driven, N-server, consumer-scoped registry**. This unblocks Sprint 22b (the native `github_server`, the system's *second* MCP server) without that sprint having to invent discovery. It is a **pure client-side refactor**: it adds **no** MCP server, **no** subprocess surface, **no** credential, and **no** file-write surface. The load-bearing gate is **coder-tools parity** â€” the agentic Coder's tool path must behave byte-identically to today.

**Dependencies:** Phase 2 MCP tooling (`7368411`) â€” this sprint refactors its client (`tools/mcp/provider.py`) and assumes it is in place. The multi-session **routing** substrate already exists: `MCPToolProvider.__init__` already accepts `list[StdioServerParameters]`, `_serve()` already aggregates every server's `list_tools()` into `self._tools` and maps each tool name to its owning session (`self._sessions[tool.name] = session`), and `execute()` already routes by tool name. The gap this sprint fills is purely **discovery** (a `loop_engine.mcp.json` loader that produces that server list) plus **consumer-scoping** (each consumer gets only the servers it asked for). No new third-party dependency â€” reuse stdlib `json` + the already-pinned `pydantic` for the config model. **Read the "Context (locked decisions)" section before starting** â€” the Architect (Opus) settled the design in the Phase 5 planning pass; do not re-open it.

**Context (locked decisions â€” do not re-open):**
- **First Phase 5 slice = the github MCP server, built native re-front, foundation-first.** 22a is its prerequisite: config-driven discovery so a *second* server can exist. (github server itself is 22b, gated separately.)
- **Consumer split is a hard invariant (cross-cutting #2).** The MCP provider is consumed **only** by the **model's** coder tool loop (`personas/coder_iac/shared.py:177`). The Phase-5 github verbs are **orchestrator-invoked, not model tools** â€” they must **never** appear in the model's tool list. Therefore discovery is **consumer-scoped**: a consumer builds a provider for the server(s) *it* names, and `build_coder_tool_provider` must continue to yield **exactly** `{read_file, list_files, grep, run_tests}` even when `loop_engine.mcp.json` also declares other (e.g. `github`) servers. This is encoded as a test **now**, before github exists (Task 4).
- **Heterogeneous launch profiles are expected and intentional.** coder-tools launches **sandboxed / no-network** with **runtime-computed** params (worktree cwd + `sandbox_runtime_mode()` â†’ container/bwrap) because it runs untrusted **model** code. Static network/trusted servers (github in 22b, state-io later) launch from a **static `loop_engine.mcp.json` spec**. So `loop_engine.mcp.json` declares static launch specs; the coder-tools server keeps its existing isolation-aware runtime launch as a **recognized special case** â€” its config entry (if present) is only its *local* (non-sandbox) profile, and `build_coder_tool_provider`'s container/sandbox override still applies on top. **Do not** try to push the container/sandbox argv into `loop_engine.mcp.json`.
- **Config filename is `loop_engine.mcp.json` at repo root â€” NOT `.mcp.json`** *(revised 2026-07-09, Opus/Architect, mid-implementation)*. Repo-root `.mcp.json` is **already** Claude Code's own project MCP config (`{"mcpServers": {...}}`, the devcontainer's hosted-github wiring, committed `65ed47c`) â€” a different schema/purpose and a filename Claude Code reserves. loop-engine uses a distinct, product-namespaced file for its own stdio launch specs; the two never mix. The loader's `_CONFIG_FILENAME` is `loop_engine.mcp.json`. (This unblocks the `blocked` state: the original plan named `.mcp.json`, whose presence made `load_mcp_config()` crash on Claude Code's `mcpServers` key under `extra="forbid"`.)
- **`loop_engine.mcp.json` is optional, with a built-in default.** When the file is absent, behavior is **exactly today's**: a built-in `coder_tools` server definition (equivalent to the current `coder_tools_server_params`). Absence must never break an existing run. The file, when present, augments/overrides by logical server name.
- **Config is validated `extra="forbid"`** (repo convention, mirrors `State`) so a typo'd server key fails loudly rather than being silently ignored.
- **This sprint sits on the existing `LOOP_ENGINE_TOOLS=mcp` path â€” no new flag.** The classic (non-mcp) coder path is untouched. When `LOOP_ENGINE_TOOLS` is unset, none of this code runs.

**Security Considerations:** No new attack surface. **No new subprocess surface** â€” servers are still spawned exactly as today, by the MCP `stdio_client` inside `MCPToolProvider._serve()` (the sanctioned-surfaces list in `CLAUDE.md` is unchanged: the three fixed-argv surfaces plus the stdio-client server launch). **No credential** enters this module (still true of `tools/mcp`). **No file-write** â€” the `loop_engine.mcp.json` loader only *reads* a trusted, in-repo config file with `encoding="utf-8"` and `json.loads` (never `eval`/`yaml.load`); it does not write. The one net-new security-relevant property is a **positive control**: the consumer-scope guard (Task 4) provably prevents orchestrator-only servers' tools from leaking into the model's tool loop â€” assert it now so 22b's github tools cannot reach the model even by misconfiguration. Module-boundary tests must stay green: `tools/mcp` still imports no `keyring` and writes no files; `core/` unaffected.

**Risks & Blockers:**
- **Coder-tools parity is load-bearing.** After the refactor, `build_coder_tool_provider(cwd=...)` must produce a provider whose discovered tool set and launch behavior are identical to today under all three isolation modes (local / container / sandbox). The container/sandbox override (`sandbox_runtime_mode()` â†’ `container_server_params`/`sandbox_server_params`) must still fire and still mount only the worktree. Assert the existing `tests/tools/test_mcp_provider.py` expectations hold unchanged, and that the local path's `StdioServerParameters` (command/args/cwd) equal the pre-refactor values.
- **`loop_engine.mcp.json`-absent fallback must be byte-identical to today.** The built-in default `coder_tools` definition must equal `coder_tools_server_params(cwd)`. A test must cover the no-file case explicitly.
- **Do not aggregate across consumers.** It must be impossible for `build_coder_tool_provider` to return github (or any non-coder) tools even if `loop_engine.mcp.json` lists them. The scoping is by explicit server-name selection, not "launch everything."
- **Config-file discovery is trusted-input only.** `loop_engine.mcp.json` is a first-party repo file, not attacker-controlled; but still validate shape (`extra="forbid"`, required `command`/`args`) so a malformed entry fails at load, not at an obscure runtime point. Do **not** support arbitrary shell strings â€” `command` + `args: list[str]` only (no `shell=True`, consistent with every existing launch).
- **No live end-to-end verification on this branch** (no LLM key, no second real server yet). Verification is unit-level: the two-server discovery/routing test (Task 3) uses trivial in-repo stdio servers (or the coder-tools server declared twice under two logical names), consistent with the existing provider tests. Append any deferred live check to `sprints/DEFERRED_VERIFICATION.md`.

**Tasks:**

- **Task 1: `loop_engine.mcp.json` schema + loader (`load_mcp_config`)**
  - **Description:** Add a config loader to `tools/mcp` (e.g. `tools/mcp/config.py`) that reads an optional `loop_engine.mcp.json` and returns a validated mapping of **logical server name â†’ launch spec**. Define a Pydantic model (`extra="forbid"`) for the file: `{"servers": {"<name>": {"command": str, "args": list[str], "cwd": str | None}}}`. Resolve the file path (repo root `loop_engine.mcp.json`; document the location). **Absent file â‡’ return the built-in default only** â€” a single `coder_tools` entry equivalent to today's `coder_tools_server_params()` (do not hard-require the file). Present file â‡’ merge over the built-in default by logical name (an explicit `coder_tools` entry overrides the built-in local profile; other names are added). Read with `encoding="utf-8"`, parse with `json.loads`. No file writes.
  - **Target Files:** `src/loop_engine/tools/mcp/config.py` (new), `src/loop_engine/tools/mcp/__init__.py` (export), `tests/tools/test_mcp_config.py` (new)
  - **Acceptance Criteria:** With no `loop_engine.mcp.json`, `load_mcp_config()` returns exactly one server (`coder_tools`) whose spec equals the current `coder_tools_server_params` (command/args). With a `loop_engine.mcp.json` declaring a second server (`github`), both appear; an entry with an unknown key raises a validation error (`extra="forbid"`); a missing `command`/`args` raises. Loader performs no writes and imports no keyring. Lint clean.

- **Task 2: `build_provider_for(names, *, cwd)` + refactor `build_coder_tool_provider` to use it**
  - **Description:** Add a consumer-scoped builder to `tools/mcp/provider.py`: `build_provider_for(names: list[str], *, cwd=None) -> MCPToolProvider` that loads the config (Task 1), selects **only** the named servers, materializes each into `StdioServerParameters`, and returns `MCPToolProvider([...])`. For the `coder_tools` server, preserve the existing **isolation-aware** launch: when `sandbox_runtime_mode()` is `container`/`sandbox`, use `container_server_params`/`sandbox_server_params` (mount only the worktree) exactly as today; otherwise use the config's local spec. Then **refactor** `build_coder_tool_provider(cwd)` to be a thin wrapper: `build_provider_for(["coder_tools"], cwd=cwd)`. Behavior and signature of `build_coder_tool_provider` are unchanged for its one caller (`personas/coder_iac/shared.py`).
  - **Target Files:** `src/loop_engine/tools/mcp/provider.py`, `src/loop_engine/tools/mcp/__init__.py` (export `build_provider_for`), `tests/tools/test_mcp_provider.py`
  - **Acceptance Criteria:** `build_coder_tool_provider(cwd)` yields a provider whose sole server's `StdioServerParameters` equal today's under local mode, and still routes through `container_server_params`/`sandbox_server_params` under `container`/`sandbox` mode (existing isolation assertions pass unchanged). `build_provider_for(["coder_tools", "github"], ...)` (with a test `loop_engine.mcp.json`) constructs a two-server provider. Selecting a name absent from config raises a clear error. Lint clean.

- **Task 3: Multi-server discovery + routing test (proves cross-cutting #3 is paid down)**
  - **Description:** Add a test that launches **two** stdio servers from a test `loop_engine.mcp.json` and asserts end-to-end discovery + routing through the real `MCPToolProvider` (context-manager enter/exit). Use two trivial first-party stdio servers under `tests/` fixtures (e.g. a tiny `echo`/`ping` FastMCP server), **or** declare the existing coder-tools server twice under two logical names with distinct tool prefixes â€” whichever keeps the test hermetic and offline. Assert: both servers' tools appear in `provider.tools`; a call to a tool from server A routes to A's session and returns A's output; a call to a B tool routes to B; an unknown tool raises `MCPToolError`.
  - **Target Files:** `tests/tools/test_mcp_provider.py` (or new `tests/tools/test_mcp_multiserver.py`), `tests/tools/fixtures/` (tiny stdio server if that route chosen)
  - **Acceptance Criteria:** A single `MCPToolProvider` built from a two-server config discovers both tool sets and dispatches each call to the correct server; unknown-tool dispatch raises `MCPToolError`; the provider tears down both sessions cleanly on exit (no hung threads/loops). Test is fully offline and deterministic. Lint clean.
